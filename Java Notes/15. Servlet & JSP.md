## **Module 15 — Java Web Development (Servlets & JSP)**

### **15.1 Servlet Basics**

- Overview
    
    # **Servlet Basics**
    
    ---
    
    ## **1. Definition and Explanation**
    
    A **Servlet** is a Java class that runs on a web server to handle client requests (typically HTTP requests) and generate dynamic responses.
    
    - Servlets form the core of **Java Web Application Architecture**.
    - They are managed by a **Servlet Container** (Tomcat, Jetty, GlassFish), which controls lifecycle, threading, mapping, and request/response handling.
    
    In simple terms, a Servlet is a server-side component used to create **dynamic web pages** and handle **backend processing** in Java web applications.
    
    ---
    
    ---
    
    ## **2. Purpose of Servlets**
    
    Servlets are used for:
    
    - Processing HTTP requests (GET, POST, PUT, DELETE)
    - Generating dynamic HTML or JSON responses
    - Handling form submissions
    - Managing sessions and authentication
    - Serving REST APIs (in older Servlet-style architectures)
    
    ---
    
    ---
    
    ## **3. Servlet Architecture Overview**
    
    Client → Web Browser
    
    Browser sends HTTP request → Servlet Container → Servlet processes request → Container sends HTTP response back to browser.
    
    Servlet container manages:
    
    - Request/Response lifecycle
    - Multi-threading
    - Object pooling
    - Mapping URLs to servlets
    
    ---
    
    ---
    
    ## **4. Key Servlet Interfaces/Classes**
    
    | Component | Purpose |
    | --- | --- |
    | `Servlet` | Core interface |
    | `HttpServlet` | Abstract class simplifying HTTP-based communication |
    | `ServletRequest` | Represents incoming request |
    | `ServletResponse` | Represents outgoing response |
    | `HttpServletRequest` | Extends request for HTTP features |
    | `HttpServletResponse` | Extends response for HTTP features |
    
    Most applications extend **HttpServlet** rather than implement Servlet directly.
    
    ---
    
    ---
    
    ## **5. Basic Servlet Class Definition**
    
    ### Servlet Example
    
    ```java
    import java.io.*;
    import jakarta.servlet.*;
    import jakarta.servlet.http.*;
    
    public class HelloServlet extends HttpServlet {
        @Override
        protected void doGet(HttpServletRequest req, HttpServletResponse resp)
                throws ServletException, IOException {
    
            resp.setContentType("text/html");
            PrintWriter out = resp.getWriter();
            out.println("<h1>Hello from Servlet!</h1>");
        }
    }
    
    ```
    
    ---
    
    ---
    
    ## **6. Servlet Lifecycle**
    
    Servlet lifecycle is managed by the container:
    
    1. **Instantiation**
        
        Object created by container
        
    2. **Initialization (`init()`)**
        
        Called once when servlet loaded
        
    3. **Request Handling (`service()` → `doGet()` / `doPost()`)**
        
        Called for every request
        
    4. **Destruction (`destroy()`)**
        
        Called when container unloads servlet
        
    
    ### Lifecycle Summary
    
    ```
    init() → service() → destroy()
    
    ```
    
    ---
    
    ---
    
    ## **7. URL Mapping / Deployment**
    
    ### Option 1: `web.xml`
    
    ```xml
    <servlet>
        <servlet-name>HelloServlet</servlet-name>
        <servlet-class>com.example.HelloServlet</servlet-class>
    </servlet>
    
    <servlet-mapping>
        <servlet-name>HelloServlet</servlet-name>
        <url-pattern>/hello</url-pattern>
    </servlet-mapping>
    
    ```
    
    ### Option 2: Annotation Mapping
    
    ```java
    @WebServlet("/hello")
    public class HelloServlet extends HttpServlet { ... }
    
    ```
    
    ---
    
    ---
    
    ## **8. Handling HTTP Methods**
    
    Servlet class overrides:
    
    | Method | HTTP Verb |
    | --- | --- |
    | `doGet()` | GET |
    | `doPost()` | POST |
    | `doPut()` | PUT |
    | `doDelete()` | DELETE |
    | `doHead()` | HEAD |
    | `doOptions()` | OPTIONS |
    
    Servlet container automatically calls correct method.
    
    ---
    
    ---
    
    ## **9. Request Processing Flow**
    
    Processing steps:
    
    1. Browser sends request `/login?user=abc`
    2. Container creates `HttpServletRequest` and `HttpServletResponse`
    3. Servlet reads parameters:
        
        ```java
        String user = req.getParameter("user");
        
        ```
        
    4. Servlet writes response using `PrintWriter`
    5. Container returns HTTP response to browser
    
    ---
    
    ---
    
    ## **10. Request Parameter Access**
    
    ```java
    String name = request.getParameter("name");
    String[] roles = request.getParameterValues("role");
    
    ```
    
    ---
    
    ---
    
    ## **11. Forwarding/Redirecting**
    
    ### Forward (Server-side)
    
    ```java
    req.getRequestDispatcher("success.jsp").forward(req, resp);
    
    ```
    
    ### Redirect (Client-side)
    
    ```java
    resp.sendRedirect("login.jsp");
    
    ```
    
    ---
    
    ---
    
    ## **12. Sessions & Cookies**
    
    Servlets provide built-in session management:
    
    ```java

    session.setAttribute("user", "John");
    
    ```
    
    Cookie handling:
    
    ```java
    Cookie cookie = new Cookie("token", "abc123");
    resp.addCookie(cookie);
    
    ```

    ## **13. Advantages of Servlets**
    
    - Java-based, platform independent
    - Efficient (multi-threaded execution)
    - Extensible and secure
    - Works well with JSP, MVC, frameworks like Spring MVC

    The servlet container is **the office**, managing multiple receptionists and their workflow.

- HTTP request/response lifecycle
    
    # **HTTP Request/Response Lifecycle (Servlet-Based Web Applications)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    The **HTTP Request/Response Lifecycle** describes the full journey of a client request from the browser to the server and back.
    
    In Java web applications, this lifecycle runs through the **Servlet Container** (Tomcat, Jetty, GlassFish) which:
    
    - Receives HTTP requests
    - Dispatches them to servlets
    - Sends formatted responses back to the client
    
    Understanding this lifecycle is essential for debugging, performance optimization, security, and system design.
    
    ---
    
    ---
    
    # **2. High-Level Lifecycle Flow**
    
    ### Steps
    
    1. **Client triggers request** (browser, REST call, form submit)
    2. **Server receives request**
    3. **Container routes to servlet based on URL mapping**
    4. **Servlet reads input/request parameters**
    5. **Servlet processes logic**
    6. **Servlet writes response output**
    7. **Response returned to client**
    
    ---
    
    ---
    
    # **3. Detailed Step-by-Step Lifecycle**
    
    ---
    
    ## **Step 1: Client Sends HTTP Request**
    
    - A browser submits a request such as:
        
        ```
        GET /login?user=abc
        
        ```
        
    - Request also contains headers, cookies, method (GET/POST/etc.).
    
    ---
    
    ## **Step 2: Web Server Receives Request**
    
    - Server like Apache/Nginx forwards request to application server (Tomcat).
    
    ---
    
    ## **Step 3: Request Enters Servlet Container**
    
    The servlet container:
    
    - Parses request
    - Creates **HttpServletRequest** and **HttpServletResponse** objects
    
    ---
    
    ## **Step 4: URL Pattern Matching & Mapping**
    
    Container matches the request URL against mappings:
    
    | URL | Servlet |
    | --- | --- |
    | `/login` | LoginServlet |
    | `/dashboard` | DashboardServlet |
    
    Example annotation mapping:
    
    ```java
    @WebServlet("/login")
    
    ```
    
    ---
    
    ## **Step 5: Thread Creation**
    
    Servlet container usually allocates **one thread per request**.
    
    This enables concurrent processing.
    
    ---
    
    ## **Step 6: Servlet Instantiation (if needed)**
    
    If servlet not created yet:
    
    - Container loads servlet class
    - Creates instance
    - Calls `init()` once
    
    ---
    
    ## **Step 7: Service Method Invocation**
    
    Container calls `service()` on servlet:
    
    ```java
    service(HttpServletRequest req, HttpServletResponse resp)
    
    ```
    
    This dispatches to respective methods:
    
    | HTTP Method | Servlet Method |
    | --- | --- |
    | GET | doGet() |
    | POST | doPost() |
    | PUT | doPut() |
    | DELETE | doDelete() |
    
    ---
    
    ## **Step 8: Request Processing**
    
    Servlet logic executes:
    
    ### Reads input:
    
    ```java
    String name = req.getParameter("name");
    Cookie[] cookies = req.getCookies();
    HttpSession session = req.getSession();
    
    ```
    
    ### Executes business logic:
    
    - Connect database
    - Validate credentials
    - Call services
    
    ---
    
    ## **Step 9: Response Creation**
    
    Servlet builds output using `HttpServletResponse`:
    
    ```java
    resp.setContentType("text/html");
    PrintWriter out = resp.getWriter();
    out.println("<h1>Welcome</h1>");
    
    ```
    
    Setting headers and cookies:
    
    ```java
    resp.addCookie(new Cookie("token","123"));
    
    ```
    
    Forwarding to JSP or redirecting also part of response building:
    
    ```java
    req.getRequestDispatcher("dashboard.jsp").forward(req, resp);
    
    ```
    
    ---
    
    ## **Step 10: Container Sends Response**
    
    - Writes HTTP status code (200, 302, 404, etc.)
    - Writes headers
    - Sends response content back to browser
    
    Browser renders UI or JSON output based on content type.
    
    ---
    
    ## **Step 11: Connection Cleanup**
    
    - Thread returned to pool
    - Input/output streams closed
    - If servlet shutting down, container calls `destroy()`
    
    ---
    
    ---
    
    # **4. Full Lifecycle Diagram**
    
    ```
    Browser → HTTP Request → Web Server → Application Server
      → Servlet Container → Servlet → Response Generation
      → Browser ← HTTP Response
    
    ```
    
    ---
    
    ---
    
    # **5. Additional Lifecycle Concepts**
    
    ---
    
    ### **Session Management**
    
    During steps 8–9, container manages:
    
    - `HttpSession` creation
    - Tracking via JSESSIONID cookies or URL rewriting
    
    ---
    
    ### **Filters**
    
    Filters intercept requests:
    
    ```
    Client → Filter → Servlet → Response → Filter → Client
    
    ```
    
    Used for:
    
    - Authentication
    - Logging
    - Compression
    
    ---
    
    ### **Listeners**
    
    Lifecycle callbacks to track:
    
    - app start/stop
    - session creation/destruction
    - request handling
    
    ---
    
    ---
    
    # **6. Real-world Analogy**
    
    Imagine visiting a restaurant:
    
    1. Guest (client) arrives (request).
    2. Host receives request (web server).
    3. Waiter assigned (thread).
    4. Order taken (request parsing).
    5. Kitchen prepares meal (business logic).
    6. Waiter serves food (response).
    
    Servlet container plays the role of *manager*, coordinating all steps.

    - Debugging unexpected state issues
    - Preventing thread-safety bugs
    - Controlling performance: pooling, caching, filtering
    - Correct application architecture design

    # **8. Summary**
    
    The HTTP Request/Response lifecycle defines how a client request reaches servlet code, is processed, and returned as a response.
    
    Servlet container manages object lifecycle, routing, threading, session handling, and cleanup, enabling scalable and dynamic Java web applications.
    
- init(), service(), destroy()
    
    # **Servlet Lifecycle Methods: init(), service(), destroy()**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Every servlet follows a well-defined lifecycle managed by the **Servlet Container** (e.g., Tomcat, Jetty).
    
    The three core lifecycle methods are:
    
    1. **init()** — initialization
    2. **service()** — request handling
    3. **destroy()** — cleanup before removal
    
    These methods ensure servlet instances are:
    
    - Created once
    - Used to serve many requests
    - Cleaned properly when not needed
    
    Understanding lifecycle behavior helps in writing correct and efficient server-side code.
    
    ---
    
    ---
    
    # **2. init() Method**
    
    ---
    
    ## **2.1 Purpose**
    
    `init()` is called **once** when servlet is first loaded.
    
    Tasks performed:
    
    - Initialization
    - DB connection setup
    - Configuration loading
    - Resource allocation
    
    ---
    
    ## **2.2 Invocation Behavior**
    
    - Executed once per servlet instance.
    - Called before handling first request.
    - Container guarantees thread safety — no concurrent calls to `init()`.
    
    ---
    
    ## **2.3 Method Signature**
    
    ```java
    public void init() throws ServletException {}
    
    ```
    
    ---
    
    ## **2.4 Example Usage**
    
    ```java
    @Override
    public void init() {
        System.out.println("Servlet initialized");
    }
    
    ```
    
    ---
    
    ---
    
    # **3. service() Method**
    
    ---
    
    ## **3.1 Purpose**
    
    `service()` is invoked **per request**.
    
    It determines the HTTP method and forwards call to appropriate handler:
    
    | HTTP | Servlet Method |
    | --- | --- |
    | GET | doGet() |
    | POST | doPost() |
    | PUT | doPut() |
    | DELETE | doDelete() |
    
    ---
    
    ## **3.2 Invocation Behavior**
    
    - Called multiple times.
    - One call per user request.
    - Container typically allocates one thread per request.
    
    ---
    
    ## **3.3 Method Signature**
    
    ```java
    public void service(HttpServletRequest req, HttpServletResponse resp)
            throws ServletException, IOException {}
    
    ```
    
    ---
    
    ## **3.4 Example Internal Behavior**
    
    Typical workflow inside service():
    
    ```
    Incoming request → service()
    → calls doGet() or doPost()
    
    ```
    
    You rarely override it; instead override doGet()/doPost().
    
    ---
    
    ---
    
    # **4. destroy() Method**
    
    ---
    
    ## **4.1 Purpose**
    
    `destroy()` runs **once**, when servlet is being removed/unloaded.
    
    Used to:
    
    - Close DB connections
    - Release resources
    - Cleanup memory
    - Save state/logging
    
    ---
    
    ## **4.2 Invocation Behavior**
    
    - Run once in servlet’s lifetime.
    - Called before container shutdown or servlet redeployment.
    
    ---
    
    ## **4.3 Method Signature**
    
    ```java
    public void destroy() {}
    
    ```
    
    ---
    
    ## **4.4 Example Usage**
    
    ```java
    @Override
    public void destroy() {
        System.out.println("Servlet destroyed");
    }
    
    ```
    
    ---
    
    ---
    
    # **5. Servlet Lifecycle Diagram**
    
    ```
    Container Loads Servlet
         ↓ init()
    Servlet Ready
         ↓ service() → doGet(), doPost(), etc. (per request)
    Container Unloads Servlet
         ↓ destroy()
    Servlet Instance Removed
    
    ```
    
    ---
    
    ---
    
    # **6. Real-world Analogy**
    
    Think of servlet lifecycle as an **employee working a shift**:
    
    - `init()` = employee gets assigned workspace and resources
    - `service()` = employee handles multiple customer requests during shift
    - `destroy()` = employee wraps up work, hands over reports, and logs out
    
    ---
    
    ---
    
    # **7. Best Practices**
    
    - Keep `init()` lightweight; avoid long waits
    - Avoid overriding `service()` unless custom behavior needed
    - Clean resources reliably in `destroy()`
    - Ensure thread safety in `service()` handlers
    
    ---
    
    ---
    
    # **8. Summary**
    
    The servlet lifecycle is orchestrated by the container via:
    
    | Method | When Called | Count |
    | --- | --- | --- |
    | `init()` | When servlet first loads | Once |
    | `service()` | For every client request | Many times |
    | `destroy()` | When servlet is removed | Once |
    
    Knowing these methods helps build efficient servlets and manage resources correctly.
    
- HttpServletRequest & HttpServletResponse
    
    # **HttpServletRequest & HttpServletResponse**
    
    ---
    
    ## **1. Definition and Explanation**
    
    The **HttpServletRequest** and **HttpServletResponse** interfaces represent the **request–response model** in the Java Servlet API.
    
    - **HttpServletRequest** encapsulates details of the **incoming HTTP request** (URL, parameters, headers, body, cookies, session, method).
    - **HttpServletResponse** represents the **outgoing HTTP response** sent back to the client (status, content type, output body, redirection, cookies).
    
    Together, they enable servlets to read user input and generate server output.
    
    ---
    
    ---
    
    # **2. HttpServletRequest**
    
    ---
    
    ## **2.1 Purpose**
    
    Used to **read everything that comes from the client**, such as:
    
    - URL path and query string
    - Form data (POST body)
    - HTTP method (GET/POST/etc.)
    - Cookies and session
    - Request headers
    
    ---
    
    ## **2.2 Common Operations**
    
    ### Extracting Parameters
    
    ```java
    String name = request.getParameter("name");
    String[] roles = request.getParameterValues("role");
    
    ```
    
    ---
    
    ### Reading Path / URL Details
    
    ```java
    String uri = request.getRequestURI();
    String method = request.getMethod();
    
    ```
    
    ---
    
    ### Accessing Request Headers
    
    ```java
    String agent = request.getHeader("User-Agent");
    
    ```
    
    ---
    
    ### Working with Session
    
    ```java
    HttpSession session = request.getSession();
    session.setAttribute("username", "John");
    
    ```
    
    ---
    
    ### Reading Cookies
    
    ```java
    Cookie[] cookies = request.getCookies();
    
    ```
    
    ---
    
    ### Handling File Uploads (Multipart)
    
    Servlet 3+ supports multipart uploads using `@MultipartConfig`.
    
    ---
    
    ---
    
    ## **2.3 Real-time Use Cases**
    
    - Login form submission
    - Search filtering parameters
    - Shopping cart session retrieval
    - File upload handling
    
    ---
    
    ---
    
    # **3. HttpServletResponse**
    
    ---
    
    ## **3.1 Purpose**
    
    Used to **construct and send response** back to client including:
    
    - HTML output
    - JSON/XML payloads
    - Redirects
    - Cookies
    - HTTP status codes
    
    ---
    
    ## **3.2 Common Operations**
    
    ### Sending Output
    
    ```java
    response.setContentType("text/html");
    PrintWriter out = response.getWriter();
    out.println("<h1>Hello Client</h1>");
    
    ```
    
    ---
    
    ### Setting Status Codes
    
    ```java
    response.setStatus(HttpServletResponse.SC_OK);
    
    ```
    
    ---
    
    ### Redirection
    
    ```java
    response.sendRedirect("login.jsp");
    
    ```
    
    ---
    
    ### Sending Cookies
    
    ```java
    Cookie cookie = new Cookie("token", "123");
    response.addCookie(cookie);
    
    ```
    
    ---
    
    ### Download Responses
    
    You can set headers for file download:
    
    ```java
    response.setHeader("Content-Disposition", "attachment; filename=report.pdf");
    
    ```
    
    ---
    
    ---
    
    # **4. Interaction Between Request and Response**
    
    When a client interacts via browser:
    
    1. Container creates `HttpServletRequest` and fills it with request data.
    2. Container creates `HttpServletResponse`.
    3. Servlet processes request → writes result into response.
    4. Container sends response to browser.
    
    ---
    
    ---
    
    ## **5. Key Differences Table**
    
    | Feature | HttpServletRequest | HttpServletResponse |
    | --- | --- | --- |
    | Role | Read client input | Write server output |
    | Data Source | Browser / client | Server / servlet |
    | Contains | URL, headers, params, cookies | Headers, status, body, cookies |
    | Usage | Processing request data | Generating final output |
    
    ---
    
    ---
    
    # **6. Example Putting It Together**
    
    ```java
    @WebServlet("/hello")
    public class HelloServlet extends HttpServlet {
    
        @Override
        protected void doGet(HttpServletRequest request, HttpServletResponse response)
                throws ServletException, IOException {
    
            String name = request.getParameter("name");
    
            response.setContentType("text/html");
            PrintWriter out = response.getWriter();
    
            out.println("<h2>Hello " + name + "</h2>");
        }
    }
    
    ```
    
    ### Flow:
    
    1. User hits `/hello?name=John`
    2. Servlet reads parameter via `HttpServletRequest`
    3. Servlet writes HTML response via `HttpServletResponse`
    
    ---
    
    ---
    
    # **7. Real-world Analogy**
    
    - **HttpServletRequest** is like a form submitted by a customer — it contains all their details.
    - **HttpServletResponse** is like a printed receipt or reply generated by a service counter.
    
    ---
    
    ---
    
    # **8. Summary**
    
    - `HttpServletRequest` gives servlet access to everything sent by client.
    - `HttpServletResponse` enables servlet to send back status, headers, and content.
    
    Understanding these two APIs is foundational for web development because nearly all servlet-based communication works through them.
    

### **15.2 Servlet Concepts**

- Request attributes
    
    # **Request Attributes in Servlets**
    
    ---
    
    ## **1. Definition and Explanation**
    
    A **request attribute** is a temporary data object stored inside the `HttpServletRequest` object for **sharing data between components** during a single request lifecycle.
    
    Attributes:
    
    - Exist only **within current request**
    - Used to pass data from a servlet to JSP or between multiple servlets
    - Represent server-side data, unlike request parameters which come from client
    
    They are key to implementing MVC patterns where controllers send model objects to views.
    
    ---
    
    ---
    
    ## **2. Purpose of Request Attributes**
    
    Request attributes are used:
    
    - To transfer dynamic data between servlets
    - To expose data from controller to JSP
    - To maintain intermediate state without storing in session
    
    Examples:
    
    - Passing form validation results
    - Forwarding query results to JSP
    - Sending error or success messages
    
    ---
    
    ---
    
    ## **3. Methods Used**
    
    ### **Setting an Attribute**
    
    ```java
    request.setAttribute("user", userObject);
    
    ```
    
    ### **Retrieving an Attribute**
    
    ```java
    Object obj = request.getAttribute("user");
    
    ```
    
    ### **Removing an Attribute**
    
    ```java
    request.removeAttribute("user");
    
    ```
    
    Attributes are stored as key-value pairs.
    
    ---
    
    ---
    
    ## **4. Attribute Lifecycle**
    
    - Attributes live **only for duration of the request**
    - They are cleared when response is sent
    - They are **not valid after redirect**, because redirect creates a new request
    
    They persist only during:
    
    - Servlet → JSP forwards
    - Servlet → Servlet forwards
    
    ---
    
    ---
    
    ## **5. Example: Passing Data from Servlet to JSP**
    
    ### Servlet
    
    ```java
    User user = new User("John", "Admin");
    request.setAttribute("user", user);
    request.getRequestDispatcher("dashboard.jsp").forward(request, response);
    
    ```
    
    ### JSP
    
    ```
    Welcome, ${user.name}
    
    ```
    
    Forward keeps request attributes available to JSP.
    
    ---
    
    ---
    
    ## **6. Example: Servlet-to-Servlet Forwarding**
    
    ```java
    request.setAttribute("status", "success");
    RequestDispatcher rd = request.getRequestDispatcher("nextServlet");
    rd.forward(request, response);
    
    ```
    
    The `nextServlet` can retrieve:
    
    ```java
    String status = (String) request.getAttribute("status");
    
    ```
    
    ---
    
    ---
    
    ## **7. Difference Between Request Attribute & Request Parameter**
    
    | Aspect | Request Parameter | Request Attribute |
    | --- | --- | --- |
    | Source | Client sends (URL/form) | Server sets |
    | Lifespan | Until processed | Current request only |
    | Type | Always String | Any object |
    | Usage | User input | Passing calculated data |
    
    ---
    
    ---
    
    ## **8. Real-world Analogy**
    
    Request attributes act like **documents passed between internal departments** within a single customer visit.
    
    They are not saved permanently — only stored while the request moves internally.
    
    ---
    
    ---
    
    ## **9. Best Practices**
    
    - Use for internal temporary data — avoid storing large or persistent objects
    - Avoid heavy objects when passing between layers
    - Do not rely on them across redirects — session required for that
    
    ---
    
    ---
    
    ## **10. Summary**
    
    Request attributes are server-managed key-value data stored in `HttpServletRequest` that enable information sharing during a single request flow, typically between servlets and JSPs in MVC applications. They are transient, flexible, and fundamental for passing model data to views.
    
- Forward vs include
    
    # **Forward vs Include in Servlets**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Servlets often delegate processing to other components, typically JSPs or other servlets.
    
    Two mechanisms provided by `RequestDispatcher` are:
    
    1. **forward()** — completely transfers control to another resource.
    2. **include()** — temporarily includes output of another resource into current response.
    
    Understanding their differences is crucial for request handling and MVC design.
    
    ---
    
    ---
    
    # **2. forward()**
    
    ---
    
    ## **2.1 Definition**
    
    `forward()` forwards the **same request and response** to another servlet or JSP, **ending the current servlet’s output**.
    
    Only the target component generates the final response.
    
    ---
    
    ### **Syntax**
    
    ```java
    request.getRequestDispatcher("next.jsp").forward(request, response);
    
    ```
    
    ---
    
    ### **Behavior**
    
    - Client does **not** know redirection occurred.
    - URL in browser remains unchanged.
    - Request attributes and data persist.
    - Initial servlet stops writing output after the forward call.
    
    ---
    
    ---
    
    # **3. include()**
    
    ---
    
    ## **3.1 Definition**
    
    `include()` includes the **output of another resource** into the current response — both servlets contribute to the response.
    
    ---
    
    ### **Syntax**
    
    ```java
    request.getRequestDispatcher("header.jsp").include(request, response);
    
    ```
    
    ---
    
    ### **Behavior**
    
    - Both current servlet and included resource can output content.
    - Request attributes are shared.
    - Response remains open after include — caller continues writing.
    
    Common use cases:
    
    - Including reusable views (header/footer/sidebar)
    - Including fragments like menus or widgets
    
    ---
    
    ---
    
    # **4. Flow Comparison Diagram**
    
    ### Forward Flow
    
    ```
    Servlet A → forward() → Servlet B
               Servlet B generates full response
    
    ```
    
    ### Include Flow
    
    ```
    Servlet A writes response
      → include() → Servlet B writes part of response
    Servlet A continues writing
    
    ```
    
    ---
    
    ---
    
    # **5. When They Execute**
    
    | Aspect | forward() | include() |
    | --- | --- | --- |
    | Ends current servlet output? | Yes | No |
    | Continues after call? | No | Yes |
    | Called once per request | Yes | May be multiple |
    | Typical use | Page navigation / MVC | Page composition/reusability |
    
    ---
    
    ---
    
    # **6. Browser View**
    
    ### With forward()
    
    - Browser URL stays same (`/login`)
    - Only final output shown
    - No clue internal forwarding happened
    
    ### With include()
    
    - Browser sees **combined output** from both components
    
    ---
    
    ---
    
    # **7. Error Handling**
    
    - If calling servlet writes output before forward(), may cause `IllegalStateException` once response is committed.
    - Include() safely works anytime before commit.
    
    ---
    
    ---
    
    # **8. Real-world Analogy**
    
    - **forward()** is like transferring a caller from one department to another, where the new department fully handles the request.
    - **include()** is like conferencing in another specialist — both contribute to final response.
    
    ---
    
    ---
    
    # **9. Code Example Demonstration**
    
    ### Using forward()
    
    ```java
    request.setAttribute("user", userObj);
    request.getRequestDispatcher("profile.jsp").forward(request, response);
    
    ```
    
    ### Using include()
    
    ```java
    request.getRequestDispatcher("header.jsp").include(request, response);
    out.println("<h1>Main content</h1>");
    request.getRequestDispatcher("footer.jsp").include(request, response);
    
    ```
    
    ---
    
    ---
    
    # **10. Summary Table**
    
    | Feature | forward() | include() |
    | --- | --- | --- |
    | Response Completion | Target replaces original | Target contributes partial output |
    | Control Returns? | No | Yes |
    | URL changes? | No | No |
    | Use in MVC | Moving to view | Reusing layouts |
    | Output Order | Target only | Combined |
    
    ---
    
    ---
    
    ## **11. Summary**
    
    - `forward()` hands off processing completely, replacing original servlet’s output.
    - `include()` merges target’s response into caller’s.
    
    Together, they enable clean request routing and reusable UI composition in servlet-based applications.
    
- URL rewriting & cookies
    
    # **URL Rewriting & Cookies in Servlets**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Both **URL rewriting** and **Cookies** are mechanisms used for **session tracking** and passing data between client and server.
    
    - **Cookies** store data on the client browser and send it on every request.
    - **URL Rewriting** embeds data inside the URL when cookies cannot be used.
    
    Servlets support both approaches to maintain user state across multiple HTTP requests (since HTTP is stateless by default).
    
    ---
    
    ---
    
    # **2. Cookies**
    
    ---
    
    ## **2.1 Definition**
    
    A cookie is a small text-based key-value pair stored in the user’s browser and sent automatically with every request to the server.
    
    Cookies allow data persistence in client context — useful for:
    
    - Authentication
    - Preferences
    - Session tracking
    
    ---
    
    ## **2.2 Creating and Sending Cookies**
    
    ```java
    Cookie c = new Cookie("user", "John");
    c.setMaxAge(3600);  // expiry time
    response.addCookie(c);
    
    ```
    
    This sends the cookie to browser for storage.
    
    ---
    
    ## **2.3 Reading Cookies**
    
    ```java
    Cookie[] cookies = request.getCookies();
    for (Cookie c : cookies) {
        if (c.getName().equals("user")) {
            String name = c.getValue();
        }
    }
    
    ```
    
    ---
    
    ## **2.4 Issues with Cookies**
    
    - Users can disable cookies
    - Not secure without encryption
    - Limited storage size
    
    ---
    
    ---
    
    # **3. URL Rewriting**
    
    ---
    
    ## **3.1 Definition**
    
    URL rewriting means **embedding session identifiers or data within the URL path or query string**.
    
    Used when:
    
    - Browser disables cookies
    - Need to transfer temporary state
    
    Example:
    
    ```
    http://example.com/home?user=John&token=999
    
    ```
    
    ---
    
    ## **3.2 Implementing URL Rewriting in Servlets**
    
    The container provides session-safe URL rewriting:
    
    ```java
    String url = response.encodeURL("home.jsp");
    out.println("<a href='" + url + "'>Continue</a>");
    
    ```
    
    `encodeURL()` appends session ID if needed:
    
    ```
    home.jsp;jsessionid=24F123A9
    
    ```
    
    ---
    
    ---
    
    ## **3.3 Manual URL Rewriting**
    
    You may append parameters:
    
    ```java
    String nextPage = "profile.jsp?name=" + URLEncoder.encode(name, "UTF-8");
    response.sendRedirect(nextPage);
    
    ```
    
    ---
    
    ---
    
    ## **3.4 Drawbacks**
    
    - Exposed in address bar
    - Limited data size
    - Harder to maintain / prone to tampering
    
    ---
    
    ---
    
    # **4. URL Rewriting vs Cookies**
    
    | Feature | Cookies | URL Rewriting |
    | --- | --- | --- |
    | Storage Location | Client browser | URL |
    | Automatic sending | Yes | No (must append manually) |
    | Dependency | Browser must allow cookies | Works even if cookies disabled |
    | Visibility | Hidden (unless inspected) | Visible to user |
    | Capacity | Few KB | Limited by URL length |
    | Security | Can be secured over HTTPS | More exposed |
    | Maintenance | Automatic | Manual encoding required |
    
    ---
    
    ---
    
    # **5. When is URL Rewriting Used?**
    
    - Cookies disabled or blocked
    - Tracking session IDs without cookies
    - Passing temporary session credentials
    - Simple applications without cookies
    
    ---
    
    ---
    
    # **6. Real-world Analogy**
    
    - **Cookies** are like loyalty cards stored in a customer’s wallet — always presented when shopping.
    - **URL Rewriting** is like writing passcode on a note handed to store clerk — visible but functional when the card isn’t available.
    
    ---
    
    ---
    
    # **7. Best Practices**
    
    - Prefer Cookies for session tracking
    - Use HTTPS to secure cookies
    - Avoid sending sensitive data in URL rewriting
    - Use `encodeURL()` to support fallback session tracking
    
    ---
    
    ---
    
    # **8. Summary**
    
    - Cookies store state on the browser and provide automatic data transfer.
    - URL rewriting embeds state in URLs and works even when cookies are disabled.
    - Both mechanisms allow servlets to implement **session persistence** in an otherwise stateless HTTP model.
- HttpSession management
    
    # **HttpSession Management in Servlets**
    
    ---
    
    ## **1. Definition and Explanation**
    
    `HttpSession` represents a **server-side storage mechanism used to maintain user data across multiple HTTP requests**.
    
    Since HTTP is stateless, the session provides continuity by storing values associated with each user — shopping cart items, login details, preferences, etc.
    
    Each session has:
    
    - Unique **session ID**
    - Attributes stored server-side
    - Defined **lifecycle** controlled by timeout policies
    
    ---
    
    ---
    
    ## **2. How HttpSession Works Internally**
    
    1. User sends initial request
    2. Container creates a new session object
    3. Sends a **session ID** (usually as cookie `JSESSIONID`)
    4. Browser returns session ID on subsequent requests
    5. Container retrieves session object and returns stored values
    
    If cookies disabled, container falls back to **URL rewriting**.
    
    ---
    
    ---
    
    # **3. Creating / Retrieving a Session**
    
    ### Code
    
    ```java
    HttpSession session = request.getSession();       // creates if absent
    
    ```
    
    or
    
    ```java
    HttpSession session = request.getSession(false);  // returns null if absent
    
    ```
    
    ---
    
    ---
    
    # **4. Storing Data in Session**
    
    ### Add Attribute
    
    ```java
    session.setAttribute("user", "John");
    
    ```
    
    ### Retrieve Attribute
    
    ```java
    String user = (String) session.getAttribute("user");
    
    ```
    
    ### Remove Attribute
    
    ```java
    session.removeAttribute("user");
    
    ```
    
    ---
    
    ---
    
    # **5. Session Identification**
    
    Browsers maintain session mappings mostly via **cookies**:
    
    ```
    JSESSIONID=AB12345 =>
    sent on every request
    
    ```
    
    Fallback mechanism: **URL rewriting**
    
    ```
    example.com/dashboard;jsessionid=AB12345
    
    ```
    
    ---
    
    ---
    
    # **6. Session Timeout and Expiry**
    
    ### Timeout Configuration
    
    - Default is 30 minutes
    - Configurable in `web.xml`
    
    ```xml
    <session-config>
        <session-timeout>20</session-timeout>
    </session-config>
    
    ```
    
    ### Programmatic Timeout
    
    ```java
    session.setMaxInactiveInterval(600); // 10 minutes
    
    ```
    
    Session destroyed when idle beyond timeout or explicitly invalidated.
    
    ---
    
    ---
    
    # **7. Ending/Invalidating Session**
    
    To log out user or clear server context:
    
    ```java
    session.invalidate();
    
    ```
    
    This removes all attributes and kills session ID.
    
    ---
    
    ---
    
    # **8. Session Lifecycle**
    
    | Phase | Action |
    | --- | --- |
    | Creation | User first access or login |
    | Persistence | Attributes stored and maintained |
    | Invalid | Timeout or explicit invalidation |
    | Disposal | Container cleans resources |
    
    ---
    
    ---
    
    # **9. Session Attributes vs Request Attributes**
    
    | Feature | Request Attribute | Session Attribute |
    | --- | --- | --- |
    | Lifetime | One request | Multiple requests |
    | Visibility | Forward/include only | Persistent user session |
    | Storage | Request object | HttpSession object |
    | Usage | Passing from servlet to JSP | Tracking user identity/data |
    
    ---
    
    ---
    
    # **10. Security Considerations**
    
    - Use HTTPS to prevent session ID interception
    - Implement session regeneration on login
    - Use timeout policies for inactive sessions
    - Avoid storing sensitive raw values
    
    ---
    
    ---
    
    # **11. Real-world Analogy**
    
    HttpSession is like a **visitor pass** at a theme park:
    
    - Assigned once
    - Lets you access rides repeatedly
    - Collect and store personalized info
    - Invalidated when the day ends
    
    ---
    
    ---
    
    # **12. Best Practices**
    
    - Store minimal necessary data (avoid heavy objects)
    - Properly invalidate sessions on logout
    - Use secure cookies (`HttpOnly`, `Secure`, SameSite)
    - Avoid session creation on public resources
    
    ---
    
    ---
    
    # **13. Summary**
    
    HttpSession provides server-side mechanisms to persist user identity and data across requests.
    
    It maintains continuity via session ID (JSESSIONID), supports attribute storage, timeout handling, and explicit invalidation — making it central to authentication, shopping carts, dashboards, and any multi-step user workflow in servlet-based applications.
    

### **15.3 Filters & Listeners**

- Filter chain
    
    # **Filter Chain in Servlets**
    
    ---
    
    ## **1. Definition and Explanation**
    
    A **Filter Chain** represents the **sequential execution of multiple filters** applied to a request before reaching a servlet or JSP, and again during the response phase.
    
    It is managed by the servlet container to:
    
    - Preprocess incoming requests
    - Postprocess outgoing responses
    - Apply cross-cutting concerns (security, logging, compression, encoding)
    
    Filters implement shared logic without modifying individual servlets.
    
    ---
    
    ---
    
    # **2. Filter Concept Basics**
    
    A **filter** is a reusable component that intercepts request/response processing.
    
    Key tasks include:
    
    - Authentication / Authorization
    - Logging / Auditing
    - Request data transformations
    - Compression / Caching
    - Cross-origin control (CORS)
    - Character encoding setup
    
    Filters execute in sequence → **Filter Chain**
    
    ---
    
    ---
    
    # **3. Filter Lifecycle**
    
    Similar to servlets:
    
    - Initialization (`init()`)
    - Filtering execution (`doFilter()`)
    - Destruction (`destroy()`)
    
    ---
    
    ---
    
    # **4. doFilter() Method**
    
    This method receives:
    
    - `ServletRequest`
    - `ServletResponse`
    - `FilterChain`
    
    Filters can:
    
    1. Process request
    2. Forward request to next filter using:
    
    ```java
    chain.doFilter(request, response);
    
    ```
    
    1. Process response after next filter/servlet returns
    
    ---
    
    ---
    
    # **5. How Filter Chain Works**
    
    Consider multiple filters:
    
    ```
    F1 → F2 → F3 → servlet → F3 → F2 → F1 → client
    
    ```
    
    Execution order:
    
    - Request passes through each filter in order.
    - Servlet executes.
    - Filters process output in reverse order returning to client.
    
    ---
    
    ---
    
    # **6. Filter Chain Flow Example**
    
    ### Code in Filter F1
    
    ```java
    System.out.println("Before Servlet");
    chain.doFilter(request, response);
    System.out.println("After Servlet");
    
    ```
    
    Output sequence:
    
    ```
    F1 Before
    F2 Before
    Servlet Executes
    F2 After
    F1 After
    
    ```
    
    ---
    
    ---
    
    # **7. Filter Registration**
    
    ### Option 1: `web.xml` Configuration
    
    ```xml
    <filter>
        <filter-name>AuthFilter</filter-name>
        <filter-class>com.example.AuthFilter</filter-class>
    </filter>
    
    <filter-mapping>
        <filter-name>AuthFilter</filter-name>
        <url-pattern>/*</url-pattern>
    </filter-mapping>
    
    ```
    
    ---
    
    ### Option 2: Annotation Based
    
    ```java
    @WebFilter("/*")
    public class AuthFilter implements Filter { ... }
    
    ```
    
    ---
    
    ---
    
    # **8. Common Use Cases**
    
    | Filter Function | Example |
    | --- | --- |
    | Authentication | Reject unauthenticated requests |
    | Logging | Track request timing |
    | Compression | Apply gzip/deflate |
    | Input filtering | XSS prevention |
    | Encoding | `request.setCharacterEncoding("UTF-8")` |
    | CORS | Adding headers |
    
    ---
    
    ---
    
    # **9. Difference Between Filter and Servlet**
    
    | Aspect | Filter | Servlet |
    | --- | --- | --- |
    | Purpose | Pre/post processing | Main processing logic |
    | Called | Before and after servlet | When request mapped to it |
    | Output ability | Modifies request/response | Generates full response |
    
    ---
    
    ---
    
    # **10. Real-world Analogy**
    
    Filter chain is like **airport security checkpoints**:
    
    - Multiple gate checks
    - Each applies a rule (scan ticket, luggage check, passport scan)
    - Passenger (request) passes through all
    - Same checkpoints applied on exit path (response processing)
    
    ---
    
    ---
    
    # **11. Best Practices**
    
    - Keep filters reusable and focused on cross-cutting concerns
    - Maintain execution order properly
    - Always call `chain.doFilter()` unless blocking request
    - Avoid heavy business logic in filters
    
    ---
    
    ---
    
    # **12. Summary**
    
    Filter chain allows multiple filters to sequentially intercept and process request–response flows.
    
    It enables centralized concerns like security, logging, compression, and response transformation, ensuring clean separation of responsibilities in servlet-based applications.
    
- ServletContextListener
    
    # **ServletContextListener**
    
    ---
    
    ## **1. Definition and Explanation**
    
    `ServletContextListener` is an event-driven listener interface used to **monitor application-level lifecycle events** in Java web applications.
    
    It receives callbacks when:
    
    - The application starts (context created)
    - The application shuts down (context destroyed)
    
    This allows global setup/cleanup tasks to be performed.
    
    ---
    
    ---
    
    ## **2. Why ServletContextListener?**
    
    Useful for:
    
    - Initializing global resources
    - Loading configuration files
    - Setting up database connection pools
    - Starting scheduled tasks
    - Logging system startup/shutdown activity
    - Cleaning shared resources on shutdown
    
    It provides hooks into **web application lifecycle**, beyond per-request or per-session scope.
    
    ---
    
    ---
    
    ## **3. Key Methods**
    
    ### `contextInitialized(ServletContextEvent event)`
    
    Triggered when:
    
    - Application is deployed/started
    - ServletContext is created
    
    Used to load resources:
    
    ```java
    @Override
    public void contextInitialized(ServletContextEvent sce) {
        System.out.println("Application started");
    }
    
    ```
    
    ---
    
    ### `contextDestroyed(ServletContextEvent event)`
    
    Triggered when:
    
    - Application stops
    - Container unloads application
    
    Used for cleanup:
    
    ```java
    @Override
    public void contextDestroyed(ServletContextEvent sce) {
        System.out.println("Application shutting down");
    }
    
    ```
    
    ---
    
    ---
    
    ## **4. Registering ServletContextListener**
    
    ### Option 1: Via `web.xml`
    
    ```xml
    <listener>
        <listener-class>com.example.AppStartupListener</listener-class>
    </listener>
    
    ```
    
    ---
    
    ### Option 2: Annotation-Based
    
    ```java
    @WebListener
    public class AppStartupListener implements ServletContextListener {
    }
    
    ```
    
    ---
    
    ---
    
    ## **5. Accessing ServletContext**
    
    Inside listener methods:
    
    ```java
    ServletContext context = sce.getServletContext();
    context.setAttribute("globalConfig", configObj);
    
    ```
    
    Servlets and JSPs can later access it:
    
    ```java
    Config cfg = (Config) request.getServletContext().getAttribute("globalConfig");
    
    ```
    
    ---
    
    ---
    
    ## **6. Use Case Examples**
    
    ### **A. Load Data on Startup**
    
    - Read DB tables
    - Prepare cache
    - Preload settings or properties
    
    ### **B. Cleanup on Shutdown**
    
    - Close DB connections
    - Stop threads
    - Remove cached objects
    
    ---
    
    ---
    
    ## **7. Real-world Analogy**
    
    Think of `ServletContextListener` as **facility management** of a building:
    
    - Opens building and prepares systems when day begins (`contextInitialized`)
    - Closes sessions and shuts resources before leaving (`contextDestroyed`)
    
    ---
    
    ---
    
    ## **8. Best Practices**
    
    - Avoid long blocking tasks in startup callbacks
    - Avoid storing too much memory via ServletContext attributes
    - Always release and close resources in `contextDestroyed`
    
    ---
    
    ---
    
    ## **9. Summary**
    
    `ServletContextListener` enables application-level initialization and destruction logic to execute automatically during lifecycle events.
    
    It is essential for configuration loading, resource pooling, and controlled shutdown behavior in enterprise Java web applications.
    
- Session listener
    
    # **HttpSessionListener**
    
    ---
    
    ## **1. Definition and Explanation**
    
    `HttpSessionListener` is a Servlet API listener that monitors **creation and destruction events of HttpSession** objects.
    
    When a session is:
    
    - **Created** → listener receives a callback
    - **Destroyed** → listener receives another callback
    
    This allows session-based tracking, cleanup, analytics, and auditing logic.
    
    ---
    
    ---
    
    # **2. Why HttpSessionListener?**
    
    Useful for managing:
    
    - Logged-in user count
    - Expired session cleanup
    - Resource deallocation linked to a session
    - Audit logging
    - Notifications when session ends
    
    It enables monitoring at **session lifecycle level**, not per request.
    
    ---
    
    ---
    
    # **3. Key Methods**
    
    ### **A. sessionCreated(HttpSessionEvent event)**
    
    Triggered when a new session is created:
    
    - Often triggered on first call to `request.getSession()`
    - Called only once per session
    
    Example usage:
    
    ```java
    @Override
    public void sessionCreated(HttpSessionEvent se) {
        System.out.println("New session created");
    }
    
    ```
    
    ---
    
    ### **B. sessionDestroyed(HttpSessionEvent event)**
    
    Triggered when session:
    
    - Times out due to inactivity
    - Is explicitly invalidated via `session.invalidate()`
    
    Example usage:
    
    ```java
    @Override
    public void sessionDestroyed(HttpSessionEvent se) {
        System.out.println("Session destroyed");
    }
    
    ```
    
    ---
    
    ---
    
    # **4. Registering Session Listener**
    
    ### Option 1 — `web.xml`
    
    ```xml
    <listener>
        <listener-class>com.example.SessionListener</listener-class>
    </listener>
    
    ```
    
    ---
    
    ### Option 2 — Annotation
    
    ```java
    @WebListener
    public class SessionListener implements HttpSessionListener { }
    
    ```
    
    ---
    
    ---
    
    # **5. Accessing Session Object**
    
    Inside listener methods:
    
    ```java
    HttpSession session = se.getSession();
    String id = session.getId();
    
    ```
    
    Use this to log or track session metadata.
    
    ---
    
    ---
    
    # **6. Practical Use Cases**
    
    ## **A. Track Online Users**
    
    ```java
    static int activeUsers = 0;
    
    public void sessionCreated(HttpSessionEvent se) {
        activeUsers++;
    }
    
    public void sessionDestroyed(HttpSessionEvent se) {
        activeUsers--;
    }
    
    ```
    
    ---
    
    ## **B. Clean Up Session Scoped Resources**
    
    ```java
    Object obj = session.getAttribute("cart");
    if(obj != null){
        // cleanup
    }
    
    ```
    
    ---
    
    ## **C. Security Notifications**
    
    Alert when certain roles log in or session expires.
    
    ---
    
    ---
    
    # **7. Session Lifecycle Recap**
    
    | Stage | Listener Event |
    | --- | --- |
    | Session creation | `sessionCreated()` |
    | Session invalidation/timeout | `sessionDestroyed()` |
    
    ---
    
    ---
    
    # **8. Differences: ServletContextListener vs HttpSessionListener**
    
    | Area | ServletContextListener | HttpSessionListener |
    | --- | --- | --- |
    | Scope | Entire application lifecycle | Per-session lifecycle |
    | Trigger | App start/stop | Session create/destroy |
    | Use cases | DB pool/init config | Online users, session cleanup |
    
    ---
    
    ---
    
    # **9. Real-world Analogy**
    
    - `HttpSessionListener` works like **attendance log system**:
        - When a user enters → record logged
        - When user leaves → record removed
    
    ---
    
    ---
    
    # **10. Best Practices**
    
    - Avoid heavy processing inside listener callbacks
    - Keep count logic thread-safe if tracking active users
    - Never block listener methods — they affect lifecycle performance
    
    ---
    
    ---
    
    # **11. Summary**
    
    `HttpSessionListener` enables tracking of session creation and destruction, empowering developers to monitor user presence, perform cleanup, manage counters, and integrate lifecycle management into web applications.
    

### **15.4 JSP Fundamentals**

- Overview
    
    # **JSP Fundamentals**
    
    ---
    
    ## **1. Definition and Explanation**
    
    JavaServer Pages (JSP) is a **server-side view technology** used to create dynamic web pages in Java web applications.
    
    Unlike servlets — which generate HTML using Java code — JSP lets developers write **HTML with embedded Java constructs**, resulting in cleaner and more maintainable UI layers.
    
    JSPs are processed by containers (Tomcat, Jetty) which convert JSP pages into **servlet classes**, compile them, execute them, and return responses to clients.
    
    ---
    
    ---
    
    # **2. Purpose of JSP**
    
    - Simplify HTML generation inside Java applications
    - Separate UI from business logic (MVC pattern)
    - Support dynamic data display
    - Allow reuse through includes, tag libraries, expression language (EL)
    
    ---
    
    ---
    
    # **3. JSP Lifecycle**
    
    JSP lifecycle mirrors servlet lifecycle with preprocessing:
    
    1. Client requests `.jsp`
    2. Container **translates JSP to servlet**
    3. Compiles servlet to `.class`
    4. Loads class and calls:
        - `init()`
        - `service()`
        - `destroy()`
    5. Revised JSPs retrigger translation cycle
    
    ---

- JSP lifecycle
    
    # **JSP Lifecycle**
    
    ---
    
    ## **1. Definition and Explanation**
    
    The **JSP Lifecycle** refers to the sequence of steps followed by a container (Tomcat, Jetty, GlassFish) to process a JavaServer Page (.jsp) from first request to destruction.
    
    Unlike static HTML, JSPs are **compiled and executed as servlets**, meaning their lifecycle mirrors servlet lifecycle but with additional preprocessing steps.
    
    ---
    
    ---
    
    # **2. JSP to Servlet Conversion Flow**
    
    When a browser requests a JSP:
    
    1. JSP file is **translated** into a Servlet source file.
    2. Servlet file is **compiled** into a `.class`.
    3. Servlet instance is **loaded** into memory.
    4. Servlet lifecycle (`init() → service() → destroy()`) begins.
    
    These steps occur **automatically** and often transparently to developers.
    
    ---
    
    ---
    
    # **3. Detailed Lifecycle Stages**
    
    ---
    
    ## **Stage 1: Translation Phase**
    
    The container parses JSP markup and converts it to:
    
    ```java
    public class index_jsp extends HttpJspBase {
        public void _jspService(...) {
            // compiled JSP logic
        }
    }
    
    ```
    
    - Scriptlets, EL expressions, JSTL tags become Java code.
    - HTML becomes response writer statements.
    
    Translation occurs only when JSP changes or first accessed.
    
    ---
    
    ---
    
    ## **Stage 2: Compilation Phase**
    
    The generated Servlet Java source file is compiled to bytecode:
    
    ```
    index_jsp.java → index_jsp.class
    
    ```
    
    Servlet errors (syntax, missing tags) show here.
    
    ---
    
    ---
    
    ## **Stage 3: Class Loading**
    
    - The compiled JSP servlet is loaded into JVM.
    - Container assigns class loader instance.
    
    ---
    
    ---
    
    ## **Stage 4: Instantiation**
    
    - Container creates an instance of the generated servlet class.
    
    ---
    
    ---
    
    ## **Stage 5: Initialization Phase (`jspInit()`)**
    
    Called **once per JSP lifecycle**.
    
    Used for:
    
    - Initial setup
    - Loading resources
    
    Signature:
    
    ```java
    public void jspInit() {
        // initialization logic
    }
    
    ```
    
    ---
    
    ---
    
    ## **Stage 6: Request Processing (`_jspService()`)**
    
    Equivalent of servlet `service()` method.
    
    Executed **for every request**.
    
    Automatically generated by container — we do not override this method manually.
    
    Handles:
    
    - Request logic
    - Output generation
    
    Signature conceptually:
    
    ```java
    public void _jspService(HttpServletRequest req, HttpServletResponse resp)
    
    ```
    
    ---
    
    ---
    
    ## **Stage 7: Destroy Phase (`jspDestroy()`)**
    
    Called once when:
    
    - Application shuts down
    - Container unloads JSP servlet
    
    Use for:
    
    - Cleanup
    - Logging
    - Resource release
    
    Signature:
    
    ```java
    public void jspDestroy() {
        // cleanup logic
    }
    
    ```
    
    ---
    
    ---
    
    # **4. Lifecycle Summary Diagram**
    
    ```
    JSP File
     ↓
    Translation to Servlet
     ↓
    Compilation
     ↓
    Class Loading
     ↓
    Instantiation
     ↓
    jspInit()
     ↓
    _jspService() (for each request)
     ↓
    jspDestroy()
    
    ```
    
    ---
    
    ---
    
    # **5. Comparison with Servlet Lifecycle**
    
    | Servlet | JSP Equivalent |
    | --- | --- |
    | `init()` | `jspInit()` |
    | `service()` | `_jspService()` |
    | `destroy()` | `jspDestroy()` |
    
    ---
    
    ---
    
    # **6. Real-world Analogy**
    
    Think of JSP lifecycle like **training and deploying an employee**:
    
    - First training (translation + compilation)
    - Take position (instantiation + init)
    - Handle customer requests repeatedly (service)
    - Retire at shutdown time (destroy)
    
    ---
    
    ---
    
    # **7. Best Practices**
    
    - Avoid heavy logic in JSP — use controllers/services
    - Use EL/JSTL instead of scriptlets
    - Cache expensive data loaded in jspInit()
    - Ensure jspDestroy() cleans shared resources
    
    ---
    
    ---
    
    # **8. Summary**
    
    JSP lifecycle enhances servlet lifecycle by automatically translating and compiling a JSP page into a servlet before execution.
    
    Understanding conversion, servlet-equivalent methods, and processing flow is essential for optimizing performance and structuring clean MVC-driven web applications.
    
- EL (Expression Language)
    
    # **Expression Language (EL) in JSP**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Expression Language (EL) in JSP is a powerful feature that enables **easy access to data stored in Java beans, request/session/application attributes, and implicit objects** — without using Java scriptlets.
    
    EL expressions are written using the `${...}` syntax and evaluate to dynamic values that get rendered in the JSP output.
    
    Example:
    
    ```
    ${user.name}
    
    ```
    
    EL greatly simplifies MVC-based JSP development by keeping Java logic out of the view.
    
    ---
    
    ---
    
    # **2. Purpose of EL**
    
    EL is used to:
    
    - Display dynamic values with simple syntax
    - Access request/session attributes
    - Evaluate arithmetic, logical, and relational expressions
    - Iterate collections (with JSTL)
    - Access Java bean properties
    
    It enhances readability and separation of concerns in JSP.
    
    ---
    
    ---
    
    # **3. EL Syntax Basics**
    
    ### Expression Format
    
    ```
    ${expression}
    
    ```
    
    ### Example Outputs
    
    ```
    Welcome ${user.name}
    
    ```
    
    ---
    
    ---
    
    # **4. EL Scope Resolution**
    
    EL automatically searches attributes in following order:
    
    1. `pageScope`
    2. `requestScope`
    3. `sessionScope`
    4. `applicationScope`
    
    Example:
    
    ```
    ${user}
    
    ```
    
    First check pageScope; if not present, search next scope.
    
    ---
    
    ---
    
    # **5. Scope Access Using Prefixes**
    
    Explicit scope access:
    
    ```
    ${requestScope.user}
    ${sessionScope.cart}
    ${applicationScope.config}
    
    ```
    
    ---
    
    ---
    
    # **6. Accessing Bean Properties**
    
    If a bean has `getName()` and `getAge()`:
    
    ```
    ${user.name}
    ${user.age}
    
    ```
    
    EL uses JavaBean getter conventions automatically.
    
    ---
    
    ---
    
    # **7. EL Operators**
    
    ### **Logical Operators**
    
    ```
    &&   ||   !
    
    ```
    
    ### **Relational**
    
    ```
    == != < > <= >=
    
    ```
    
    ### **Arithmetic**
    
    ```
    + - * / %
    
    ```
    
    ### **Conditional (Ternary)**
    
    ```
    ${age >= 18 ? 'Adult' : 'Minor'}
    
    ```
    
    ---
    
    ---
    
    # **8. Implicit Objects in EL**
    
    EL exposes several implicit objects:
    
    | Object | Usage |
    | --- | --- |
    | `param` | Request parameters |
    | `paramValues` | Multi-value params |
    | `header` | HTTP headers |
    | `cookie` | Cookie values |
    | `initParam` | context-param values |
    | `pageScope/requestScope/sessionScope/applicationScope` | Attribute access |
    
    Example:
    
    ```
    ${param.username}
    ${cookie.userToken.value}
    ${header["User-Agent"]}
    
    ```
    
    ---
    
    ---
    
    # **9. Collection Access**
    
    Access list element:
    
    ```
    ${products[0].name}
    
    ```
    
    Access map value:
    
    ```
    ${employee['id']}
    
    ```
    
    ---
    
    ---
    
    # **10. EL with JSTL Example**
    
    JSTL + EL example:
    
    ```
    <c:forEach items="${products}" var="p">
        ${p.name} - ${p.price}<br>
    </c:forEach>
    
    ```
    
    ---
    
    ---
    
    # **11. Handling Nulls and Empty Values**
    
    EL supports:
    
    ```
    ${empty list}
    ${not empty sessionScope.user}
    
    ```
    
    ---
    
    ---
    
    # **12. EL vs Scriptlets**
    
    | Feature | Scriptlet | EL |
    | --- | --- | --- |
    | Syntax | Java inside JSP | Cleaner expressions |
    | Separation | Logic mixed | View-focused |
    | Maintainability | Harder | Easier |
    | MVC compliance | Weak | Strong |
    
    Modern JSP best practices discourage scriptlets and prefer EL + JSTL.
    
    ---
    
    ---
    
    # **13. Real-world Analogy**
    
    Using EL is like **reading data labels** directly instead of digging through code — it offers a clean and declarative way to show dynamic information.
    
    ---
    
    ---
    
    # **14. Best Practices**
    
    - Avoid scriptlets — favor EL+JSTL
    - Keep UI logic minimal — business logic in controllers
    - Use explicit scope identifiers for clarity when needed
    - Always validate values in servlet before sending to JSP
    
    ---
    
    ---
    
    # **15. Summary**
    
    Expression Language replaces Java code in JSP UI with a simple, readable syntax for accessing application data.
    
    It improves maintainability, readability, and MVC separation, especially when combined with JSTL and servlet controllers in modern JSP applications.
    
- JSTL basics
    
    # **JSTL Basics (JSP Standard Tag Library)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    JSTL (JavaServer Pages Standard Tag Library) is a **collection of reusable tags** that simplify adding logic (iteration, conditionals, formatting, XML handling, SQL operations) inside JSP pages.
    
    It reduces or eliminates scriptlets and aligns JSP UI development with MVC practices.
    
    ---
    
    ---
    
    # **2. Purpose of JSTL**
    
    JSTL helps:
    
    - Reduce Java code in JSP
    - Improve readability and maintainability
    - Standardize presentation logic
    - Work seamlessly with Expression Language (EL)
    
    ---
    
    ---
    
    # **3. JSTL Tag Libraries**
    
    JSTL is made of multiple tag groups:
    
    | Tag Library | URI | Purpose |
    | --- | --- | --- |
    | Core | `http://java.sun.com/jsp/jstl/core` | Flow control, iteration, variables |
    | Format | `http://java.sun.com/jsp/jstl/fmt` | Data formatting, i18n |
    | SQL | `http://java.sun.com/jsp/jstl/sql` | Database access (rarely used now) |
    | Functions | `http://java.sun.com/jsp/jstl/functions` | String operations |
    | XML | `http://java.sun.com/jsp/jstl/xml` | XML parsing and display |
    
    ---
    
    ---
    
    # **4. Adding JSTL to JSP**
    
    ### Tag Library Declaration
    
    ```
    <%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>
    
    ```
    
    ---
    
    ---
    
    # **5. Common JSTL Core Tags**
    
    ---
    
    ## **5.1 `<c:out>` — Displaying Values Safely**
    
    ```
    <c:out value="${user.name}" />
    
    ```
    
    ---
    
    ## **5.2 `<c:set>` — Creating/Updating Variables**
    
    ```
    <c:set var="name" value="John" />
    
    ```
    
    ---
    
    ## **5.3 `<c:if>` — Conditional Rendering**
    
    ```
    <c:if test="${age > 18}">
        You are adult.
    </c:if>
    
    ```
    
    ---
    
    ## **5.4 `<c:choose>` — Multi-Condition Logic**
    
    ```
    <c:choose>
        <c:when test="${age > 18}">Adult</c:when>
        <c:otherwise>Minor</c:otherwise>
    </c:choose>
    
    ```
    
    ---
    
    ## **5.5 `<c:forEach>` — Iteration**
    
    ```
    <c:forEach var="item" items="${products}">
        ${item.name} - ${item.price}<br/>
    </c:forEach>
    
    ```
    
    ---
    
    ## **5.6 `<c:forTokens>` — Splitting Strings**
    
    ```
    <c:forTokens items="A,B,C" delims="," var="x">
        ${x}<br/>
    </c:forTokens>
    
    ```
    
    ---
    
    ## **5.7 `<c:remove>` — Removing JSP Variables**
    
    ```
    <c:remove var="username"/>
    
    ```
    
    ---
    
    ---
    
    # **6. Formatting Tags (fmt)**
    
    Used for internationalization,i18n, formatting dates and numbers.
    
    ### Declaration
    
    ```
    <%@ taglib uri="http://java.sun.com/jsp/jstl/fmt" prefix="fmt" %>
    
    ```
    
    Example:
    
    ```
    <fmt:formatNumber value="${amount}" type="currency" />
    
    ```
    
    ---
    
    ---
    
    # **7. JSTL Functions Library**
    
    Declaration:
    
    ```
    <%@ taglib uri="http://java.sun.com/jsp/jstl/functions" prefix="fn" %>
    
    ```
    
    Common example:
    
    ```
    ${fn:length(list)}
    ${fn:contains(name, 'Admin')}
    
    ```
    
    ---
    
    ---
    
    # **8. JSTL SQL Tags**
    
    These tags can run SQL queries:
    
    ```
    <sql:setDataSource.../>
    <sql:query.../>
    
    ```
    
    They are **not recommended** for enterprise use — data access should be in DAO/service layers.
    
    ---
    
    ---
    
    # **9. JSTL Advantages**
    
    - Cleaner, script-free JSP
    - Strong MVC alignment
    - Readable logic blocks
    - Enhanced maintainability
    - Works naturally with EL
    
    ---
    
    ---
    
    # **10. Real-world Analogy**
    
    JSTL is like **prebuilt building blocks** for constructing UI features — loops, conditions, formatting — instead of manually coding logic.
    
    ---
    
    ---
    
    # **11. Best Practices**
    
    - Prefer JSTL + EL instead of scriptlets
    - Keep logic simple — complex logic belongs in controllers/services
    - Avoid JSTL SQL tags in production
    
    ---
    
    ---
    
    # **12. Summary**
    
    JSTL provides standardized tag libraries for iteration, conditions, formatting, and EL support, enabling clean UI logic without scriptlets in JSPs.
    
    It plays a vital role in MVC applications through reusable, readable, and maintainable JSP design.
