### **5.1 Introduction to Design Patterns**

# **Introduction to Design Patterns**

---

## **1. Definition and Explanation**

### **What Are Design Patterns?**

Design patterns are **proven, reusable solutions** to common software design problems that appear repeatedly in real-world development.

They provide **standardized templates** and **best practices** for structuring classes and objects in a flexible, maintainable, and extensible way.

### **Purpose**

- Reduce development time by reusing established solutions
- Improve code readability through common vocabulary
- Enhance maintainability and scalability
- Encourage loose coupling, high cohesion, and clean architecture
- Serve as guidelines, not fixed code implementations

---

## **2. Categories of Design Patterns**

Design patterns are traditionally divided into **three major categories** defined in the “Gang of Four” (GoF) book.

---

## **A. Creational Patterns**

### **Definition**

Creational patterns deal with **object creation mechanisms**, ensuring flexibility and control over how objects are instantiated.

### **Examples**

- **Singleton** — One instance globally
- **Factory Method** — Delegates object creation to subclasses
- **Abstract Factory** — Creates families of related objects
- **Builder** — Step-by-step complex object construction
- **Prototype** — Object cloning to avoid costly new operations

### **Purpose**

- Control instantiation
- Improve flexibility of object creation
- Hide construction logic

---

## **B. Structural Patterns**

### **Definition**

Structural patterns define how classes and objects form **larger structures** while maintaining **low coupling**.

### **Examples**

- **Adapter** — Convert one interface to another
- **Decorator** — Add functionality without modifying class
- **Facade** — Provide simplified interface to complex subsystems
- **Composite** — Tree-like object structures
- **Proxy** — Access control, lazy loading, or remote access

### **Purpose**

- Simplify complex structures
- Enable object composition
- Promote flexible and scalable architectures

---

## **C. Behavioral Patterns**

### **Definition**

Behavioral patterns focus on **object interactions**, communication, and the distribution of responsibilities.

### **Examples**

- **Strategy** — Replaceable algorithms
- **Observer** — Event-based updates to multiple dependents
- **Command** — Encapsulates actions as objects
- **Template Method** — Base algorithm with overridable steps
- **Iterator** — Sequential access to collection elements
- **State** — Alter behavior based on internal state

### **Purpose**

- Improve communication between objects
- Encapsulate logic
- Promote loose coupling

---

### **5.2 Creational Patterns**

- Singleton
    
    # **Creational Patterns — Singleton**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Singleton Pattern** ensures that a class has **only one instance** throughout the entire application and provides a **global point of access** to it.
    
    ### **Common Use Cases**
    
    - Configuration managers
    - Caches
    - Logging frameworks
    - Connection pools
    - Thread pools
    - Registry or metadata managers
    
    The goal is to prevent multiple instances and manage a single shared resource safely.
    
    ---
    
    ## **2. Static Instance Singleton (Traditional Implementation)**
    
    ### **A. Eager Initialization**
    
    ### **Syntax**
    
    ```java
    public class Singleton {
        private static final Singleton INSTANCE = new Singleton();
    
        private Singleton() { }
    
        public static Singleton getInstance() {
            return INSTANCE;
        }
    }
    
    ```
    
    ### **Characteristics**
    
    - Instance created at class loading time
    - Simple and thread-safe
    - Might create instance even if unused (wastes memory)
    
    ---
    
    ### **B. Lazy Initialization (Non-thread-safe)**
    
    ```java
    public class Singleton {
        private static Singleton instance;
    
        private Singleton() { }
    
        public static Singleton getInstance() {
            if (instance == null) {
                instance = new Singleton();   // Not thread-safe
            }
            return instance;
        }
    }
    
    ```
    
    ### **Problem**
    
    - In multithreaded environments, two threads may create two instances.
    
    ---
    
    ### **C. Thread-safe Lazy Initialization (Synchronized Method)**
    
    ```java
    public class Singleton {
        private static Singleton instance;
    
        private Singleton() { }
    
        public static synchronized Singleton getInstance() {
            if (instance == null) {
                instance = new Singleton();
            }
            return instance;
        }
    }
    
    ```
    
    ### **Drawback**
    
    - Synchronized method can be slow for frequently accessed Singleton.
    
    ---
    
    ### **D. Double-Checked Locking (Efficient, Lazy, Thread-safe)**
    
    ```java
    public class Singleton {
        private static volatile Singleton instance;
    
        private Singleton() { }
    
        public static Singleton getInstance() {
            if (instance == null) {
                synchronized (Singleton.class) {
                    if (instance == null) {
                        instance = new Singleton();
                    }
                }
            }
            return instance;
        }
    }
    
    ```
    
    ### **Why `volatile`?**
    
    Prevents instruction reordering to ensure safe publication.
    
    ---
    
    ### **E. Holder Class (Best Practice)**
    
    ```java
    public class Singleton {
        private Singleton() { }
    
        private static class Holder {
            static final Singleton INSTANCE = new Singleton();
        }
    
        public static Singleton getInstance() {
            return Holder.INSTANCE;
        }
    }
    
    ```
    
    ### **Benefits**
    
    - Lazy
    - Thread-safe
    - No synchronization cost
    - Recommended approach before enum singletons
    
    ---
    
    ## **3. Enum-based Singleton (Effective Java Recommended)**
    
    ### **Definition**
    
    Using an enum with a single element to create a guaranteed singleton.
    
    ### **Syntax**
    
    ```java
    public enum Singleton {
        INSTANCE;
    
        public void show() {
            System.out.println("Singleton instance");
        }
    }
    
    ```
    
    ### **Why Enum Singleton Is Best**
    
    - Guaranteed single instance by JVM
    - Thread-safe by design
    - Prevents reflection-based instance creation
    - Prevents serialization breaking (automatic)
    - Simplest and most secure singleton in Java
    
    ### **Limitation**
    
    - Cannot extend a class
    - Suitable for stateless or lightweight singletons
    
    ---
    
    ## **4. Thread Safety Considerations**
    
    ### **A. Dangers in Multithreaded Environments**
    
    - Race conditions causing multiple instance creation
    - Unsafe lazy initialization
    - Lack of visibility without `volatile`
    
    ---
    
    ### **B. Ways to Ensure Thread Safety**
    
    | Approach | Thread Safety | Lazy? | Performance |
    | --- | --- | --- | --- |
    | Eager initialization | Yes | No | Fast |
    | Synchronized getter | Yes | Yes | Slower |
    | Double-checked locking | Yes | Yes | Fast |
    | Holder class | Yes | Yes | Very fast |
    | Enum singleton | Yes | Yes | Fastest, safest |
    
    ---
    
    ### **C. Reflection Attack Problem**
    
    Reflection can break private constructors:
    
    ```java
    Constructor<Singleton> c = Singleton.class.getDeclaredConstructor();
    c.setAccessible(true);
    Singleton s2 = c.newInstance(); // Creates new instance
    
    ```
    
    ### **Solution**
    
    - Enum singleton is immune
    - Otherwise throw exception inside constructor:
    
    ```java
    private Singleton() {
        if (instance != null) {
            throw new RuntimeException("Use getInstance()!");
        }
    }
    
    ```
    
    ---
    
    ### **D. Serialization Attack Problem**
    
    Serialization can create a new instance unless:
    
    ```java
    protected Object readResolve() {
        return getInstance();
    }
    
    ```
    
    ### **Enum singletons** do not require this.
    
    ---
    
    ## **5. Real-world Analogy**
    
    | Singleton Concept | Analogy |
    | --- | --- |
    | Only one instance | Only one Prime Minister in a country |
    | Global access | Everyone refers to the same leader |
    | Thread safety | Prevent multiple people simultaneously taking over |
    | Enum singleton | Constitution enforced single leader |
    
    ---
    
    ## **6. JVM / Compiler Behavior Notes**
    
    - Class loading ensures thread safety for static initializers.
    - Enums have implicit private constructors and final instances.
    - Double-checked locking requires `volatile` due to JVM instruction reordering rules.
    
    ---
    
    ## **7. Important Points**
    
    - Singleton ensures exactly one instance across application.
    - Use enum singletons for most scenarios.
    - Beware of reflection and serialization breaking traditional singletons.
    - Holder class pattern is efficient and lazy.
    - Double-checked locking must use `volatile` for correctness.
    
    ---
    
    ## **8. Summary**
    
    | Technique | Lazy | Thread-safe | Simple | Recommended |
    | --- | --- | --- | --- | --- |
    | Eager static instance | No | Yes | Yes | For small, light objects |
    | Lazy (non-thread-safe) | Yes | No | Yes | Avoid |
    | Synchronized method | Yes | Yes | Medium | Rarely used now |
    | Double-checked locking | Yes | Yes | Medium | Good option |
    | Holder class | Yes | Yes | Yes | Excellent |
    | Enum singleton | Yes | Yes | Yes | **Best option** |
    
    **In summary:**
    
    The Singleton pattern guarantees a single instance, but implementing it correctly requires attention to thread safety, reflection, serialization, and performance. Enum-based singletons are the modern, recommended solution in Java.
    
- Factory Method
    
    # **Factory Method Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Factory Method Pattern** is a creational design pattern that **defines an interface for creating objects**, but lets subclasses decide **which class to instantiate**.
    
    It allows a class to defer object creation to subclasses.
    
    ### **Purpose**
    
    - Remove direct object creation (`new`) from client code
    - Enable loose coupling between creator and product
    - Support extensibility by allowing new product types without modifying existing code
    - Encapsulate object creation logic
    
    ---
    
    ## **2. Key Components**
    
    ### **A. Product (Interface or Abstract Class)**
    
    Defines the type of objects the factory will create.
    
    ### **B. Concrete Products**
    
    Implementations of the Product interface.
    
    ### **C. Creator (Abstract Class or Interface)**
    
    Declares the factory method.
    
    ### **D. Concrete Creators**
    
    Implement the factory method and instantiate specific products.
    
    ---
    
    ## **3. Basic Structure**
    
    ### **Product**
    
    ```java
    interface Transport {
        void deliver();
    }
    
    ```
    
    ### **Concrete Products**
    
    ```java
    class Truck implements Transport {
        public void deliver() {
            System.out.println("Delivering by Truck");
        }
    }
    
    class Ship implements Transport {
        public void deliver() {
            System.out.println("Delivering by Ship");
        }
    }
    
    ```
    
    ### **Creator**
    
    ```java
    abstract class Logistics {
        abstract Transport createTransport();
    
        public void planDelivery() {
            Transport t = createTransport();
            t.deliver();
        }
    }
    
    ```
    
    ### **Concrete Creators**
    
    ```java
    class RoadLogistics extends Logistics {
        Transport createTransport() {
            return new Truck();
        }
    }
    
    class SeaLogistics extends Logistics {
        Transport createTransport() {
            return new Ship();
        }
    }
    
    ```
    
    ### **Usage**
    
    ```java
    Logistics logistics = new RoadLogistics();
    logistics.planDelivery();
    
    ```
    
    ---
    
    ## **4. Factory Method Characteristics**
    
    - Uses **inheritance**: subclass decides the object type
    - Promotes **loose coupling**
    - Encapsulates object creation
    - Supports extension without modifying existing classes (Open-Closed Principle)
    
    ---
    
    ## **5. When to Use the Factory Method**
    
    - When object creation should be **decoupled** from usage
    - When the exact type of object may vary at runtime
    - When framework libraries need to allow optional customization
    - When multiple related product types need to be created
    
    ---
    
    ## **6. Factory Method vs Simple Factory**
    
    | Aspect | Factory Method | Simple Factory |
    | --- | --- | --- |
    | Based on | Inheritance | A single factory class |
    | Extensibility | High (add subclasses) | Low (modify factory) |
    | Design pattern | GoF pattern | Not a GoF pattern |
    | Customization | Subclasses override factory method | Hard to customize |
    
    ---
    
    ## **7. Real-world Examples**
    
    ### **A. Java Examples**
    
    - `java.util.Calendar.getInstance()`
    - `java.sql.DriverManager.getConnection()`
    - `javax.xml.parsers.DocumentBuilderFactory.newInstance()`
    
    These methods choose the concrete implementation at runtime.
    
    ---
    
    ### **B. Spring Framework Examples**
    
    - **BeanFactory** and **ApplicationContext**
        - Uses factory method internally to produce beans
    - **FactoryBean**
        - Allows custom bean instantiation via a factory method pattern
    
    ---
    
    ## **8. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Factory Method | Chef decides which dish to cook based on the order |
    | Creator | Restaurant kitchen |
    | Product | Dish prepared |
    | Concrete Creator | Chef specializing in a specific cuisine |
    
    ---
    
    ## **9. JVM / Compiler Notes**
    
    - Factory Method isolates constructors, improving compatibility with reflection.
    - JVM optimizes direct constructor calls; factory calls may add indirection.
    - Spring leverages factory methods for proxy creation and lifecycle management.
    
    ---
    
    ## **10. Important Points**
    
    - Encapsulates creation logic in subclasses.
    - Reduces direct dependency on concrete classes.
    - Easy to integrate with dependency injection and frameworks.
    - Supports Open–Closed Principle.
    - Avoids `switch` or `if-else` blocks in client code.
    
    ---
    
    ## **11. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Defer object creation to subclasses |
    | **Key Idea** | Subclasses choose which product to return |
    | **Strength** | Extensible creation without modifying code |
    | **Used In** | Java APIs, Spring DI, factory-based libraries |
    
    **In summary:**
    
    The Factory Method pattern provides a flexible and extensible way to create objects by delegating instantiation to subclasses, promoting clean design, better maintainability, and improved extensibility.
    
- Abstract Factory
    
    # **Abstract Factory Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Abstract Factory Pattern** is a creational design pattern that provides an interface for creating **families of related or dependent objects** **without specifying their concrete classes**.
    
    It allows the client to create objects that belong to the same group (family) while isolating the creation logic from concrete implementations.
    
    ### **Purpose**
    
    - Ensure that products created together are **compatible**
    - Remove direct instantiation (`new`) from client code
    - Support switching between product families easily
    - Promote loose coupling and adherence to the **Open–Closed Principle**
    
    ---
    
    ## **2. Why Abstract Factory?**
    
    Consider a UI toolkit that needs:
    
    - Windows-style buttons, checkboxes
    - Mac-style buttons, checkboxes
    
    Choosing compatible products is crucial:
    
    - Windows Button + Windows Checkbox
    - Mac Button + Mac Checkbox
    
    Abstract Factory ensures **consistent product families**.
    
    ---
    
    ## **3. Structure and Components**
    
    ### **A. Abstract Product Interfaces**
    
    Define product types within a family.
    
    ### **B. Concrete Products**
    
    Different variants of the products (Windows, Mac).
    
    ### **C. Abstract Factory**
    
    Declares factory methods for all product types.
    
    ### **D. Concrete Factories**
    
    Create specific product families.
    
    ### **E. Client**
    
    Uses the factory to create product objects without knowing their exact classes.
    
    ---
    
    ## **4. Basic Example**
    
    ### **Step 1 — Abstract Products**
    
    ```java
    interface Button {
        void click();
    }
    
    interface Checkbox {
        void toggle();
    }
    
    ```
    
    ### **Step 2 — Concrete Products**
    
    ```java
    class WindowsButton implements Button {
        public void click() {
            System.out.println("Windows Button clicked");
        }
    }
    
    class MacButton implements Button {
        public void click() {
            System.out.println("Mac Button clicked");
        }
    }
    
    class WindowsCheckbox implements Checkbox {
        public void toggle() {
            System.out.println("Windows Checkbox toggled");
        }
    }
    
    class MacCheckbox implements Checkbox {
        public void toggle() {
            System.out.println("Mac Checkbox toggled");
        }
    }
    
    ```
    
    ### **Step 3 — Abstract Factory**
    
    ```java
    interface GUIFactory {
        Button createButton();
        Checkbox createCheckbox();
    }
    
    ```
    
    ### **Step 4 — Concrete Factories**
    
    ```java
    class WindowsFactory implements GUIFactory {
        public Button createButton() { return new WindowsButton(); }
        public Checkbox createCheckbox() { return new WindowsCheckbox(); }
    }
    
    class MacFactory implements GUIFactory {
        public Button createButton() { return new MacButton(); }
        public Checkbox createCheckbox() { return new MacCheckbox(); }
    }
    
    ```
    
    ### **Step 5 — Client Code**
    
    ```java
    class Application {
        private Button button;
        private Checkbox checkbox;
    
        public Application(GUIFactory factory) {
            button = factory.createButton();
            checkbox = factory.createCheckbox();
        }
    
        public void render() {
            button.click();
            checkbox.toggle();
        }
    }
    
    ```
    
    ### **Usage**
    
    ```java
    GUIFactory factory = new WindowsFactory();
    Application app = new Application(factory);
    app.render();
    
    ```
    
    ---
    
    ## **5. Characteristics of Abstract Factory**
    
    - Produces **families** of related products
    - Enforces **consistency** across product variants
    - Client never deals with `new` directly
    - Factories encapsulate object creation logic
    
    ---
    
    ## **6. When to Use Abstract Factory**
    
    - When your system needs to support **multiple product families**
    - When you want to switch between families at runtime
    - When enforcing compatibility among related objects
    - When object creation must be **fully abstracted**
    
    ---
    
    ## **7. Abstract Factory vs Factory Method**
    
    | Aspect | Abstract Factory | Factory Method |
    | --- | --- | --- |
    | Purpose | Create families of related objects | Allow subclasses to choose which object to create |
    | Object Types | Multiple products | One product |
    | Uses | Composition | Inheritance |
    | Extensibility | Add new factories | Add new subclasses |
    | Example | GUI themes | Customizing one product creation |
    
    ---
    
    ## **8. Examples in Java**
    
    ### **A. Java’s Abstract Factory Implementations**
    
    - `DocumentBuilderFactory` → XML parsers
    - `DatatypeFactory` → XML datatype processors
    - `javax.xml.stream.XMLOutputFactory`
    - `javax.sound.sampled.AudioSystem`
    
    These APIs produce **product families**.
    
    ### **B. Swing Look and Feel**
    
    Swing’s pluggable LAF (Metal, Nimbus, Windows) uses Abstract Factory to generate UI components.
    
    ---
    
    ## **9. Abstract Factory in Spring**
    
    Spring’s bean creation system heavily uses factory patterns.
    
    Examples:
    
    - **ApplicationContext / BeanFactory**
    - **FactoryBean** (abstract factory for bean creation)
    - Environment-specific beans (dev/test/prod profiles)
    - Auto-configuration families in Spring Boot (Starters)
    
    ---
    
    ## **10. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Abstract Factory | Furniture catalog that produces entire sets (modern, Victorian, minimalist) |
    | Factory | Workshop making products of one style |
    | Product Family | Table + Chair + Sofa in matching style |
    | Client | Customer ordering a complete matching set |
    
    ---
    
    ## **11. JVM / Compiler Notes**
    
    - Factories encapsulate instantiation; JVM optimizes repeated calls well.
    - Helpful for frameworks using reflection to translate configuration into objects.
    - Encourages clean architecture without tight coupling to concrete classes.
    
    ---
    
    ## **12. Important Points**
    
    - Abstract Factory groups product creation logically.
    - Ensures consistency among family members.
    - Client never touches concrete classes.
    - Easily switch entire product families by changing factory implementation.
    - Follows **Open–Closed**, **Dependency Inversion**, **Liskov Substitution** principles.
    
    ---
    
    ## **13. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Provide interface to create related objects without specifying concrete classes |
    | **Key Benefit** | Easily switch product families |
    | **Usage** | UI toolkits, database drivers, messaging clients |
    | **Relation** | A super-pattern built on multiple Factory Methods |
    
    **In summary:**
    
    The Abstract Factory pattern provides a flexible and scalable approach to creating **related object families**, promoting consistency, modularity, and clean architecture—especially important in UI systems, frameworks, and large application platforms.
    
- Builder Pattern
    
    # **Builder Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Builder Pattern** is a creational design pattern that separates the **construction of a complex object** from its **representation**, allowing the same construction process to create different representations.
    
    It solves problems where:
    
    - A class has **too many constructor parameters**
    - Parameters have combinations that create complexity
    - Object creation should be flexible and readable
    
    ### **Purpose**
    
    - Create complex objects step-by-step
    - Avoid telescoping constructors
    - Support immutability
    - Improve code readability using fluent APIs
    
    ---
    
    ## **2. Problem Before Builder Pattern (Telescoping Constructors)**
    
    Example:
    
    ```java
    public class User {
        private String name;
        private int age;
        private String address;
        private boolean isActive;
    
        public User(String name) {}
        public User(String name, int age) {}
        public User(String name, int age, String address) {}
        public User(String name, int age, String address, boolean isActive) {}
    }
    
    ```
    
    Issues:
    
    - Hard to maintain
    - Easy to pass wrong values
    - Unreadable constructor calls
    
    ---
    
    ## **3. Builder Pattern Solution**
    
    ### **Basic Structure**
    
    1. Define a static inner `Builder` class
    2. Add fields in builder same as outer class
    3. Provide chained setter methods
    4. Provide `build()` method to create the object
    
    ---
    
    ## **4. Builder Pattern Example**
    
    ### **Step 1 — Class with Private Constructor**
    
    ```java
    public class User {
        private final String name;
        private final int age;
        private final String address;
        private final boolean active;
    
        private User(Builder builder) {
            this.name = builder.name;
            this.age = builder.age;
            this.address = builder.address;
            this.active = builder.active;
        }
    
    ```
    
    ### **Step 2 — Builder Inner Class**
    
    ```java
        public static class Builder {
            private String name;
            private int age;
            private String address;
            private boolean active;
    
            public Builder name(String name) {
                this.name = name;
                return this;
            }
    
            public Builder age(int age) {
                this.age = age;
                return this;
            }
    
            public Builder address(String address) {
                this.address = address;
                return this;
            }
    
            public Builder active(boolean active) {
                this.active = active;
                return this;
            }
    
            public User build() {
                return new User(this);
            }
        }
    }
    
    ```
    
    ### **Usage**
    
    ```java
    User user = new User.Builder()
                    .name("Ayush")
                    .age(22)
                    .address("Delhi")
                    .active(true)
                    .build();
    
    ```
    
    Clean, readable, and maintainable.
    
    ---
    
    ## **5. Director Variation (Optional)**
    
    Some implementations define a **Director** class that controls the construction process externally.
    
    ```java
    class Director {
        public void constructSportsCar(CarBuilder builder) {
            builder.setEngine("V8").setSeats(2).setSunroof(true);
        }
    }
    
    ```
    
    Used in more formal GoF implementations.
    
    ---
    
    ## **6. Builder Pattern Characteristics**
    
    - Client using builder doesn’t need to know object structure
    - Builder supports constructing different “representations”
    - The object built is often **immutable**
    - Ideal for objects requiring many optional parameters
    
    ---
    
    ## **7. When to Use Builder Pattern**
    
    - When a class has more than 3–4 optional parameters
    - When object construction involves multiple steps
    - When you want readable, maintainable object creation
    - When immutability is desired
    
    ---
    
    ## **8. Builder Pattern in Java**
    
    ### **A. Java APIs Using Builder Pattern**
    
    - `StringBuilder`, `StringBuffer`
    - `java.lang.ProcessBuilder`
    - `java.time` classes like `LocalDateTime`
    - `Stream.Builder`
    - `HttpClient` (Java 11)
    - `URIBuilder` / `RequestBuilder` in external libs
    
    ---
    
    ### **B. Lombok @Builder**
    
    Lombok simplifies builder generation:
    
    ```java
    @Builder
    class User {
        private String name;
        private int age;
    }
    
    ```
    
    Automatically generates builder class with fluent API.
    
    ---
    
    ### **C. Spring Framework Example**
    
    Spring uses builders for:
    
    - `BeanDefinitionBuilder`
    - `MockMvcRequestBuilders`
    - `UriComponentsBuilder`
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Builder | Custom order form at a restaurant |
    | Product | Final dish prepared |
    | Director | Waiter suggesting preset combos |
    | Step-by-step building | Adding toppings/ingredients |
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Builder objects are optimized by JVM escape analysis; unused builders may not cause allocations.
    - Immutable objects constructed via builders are highly optimization-friendly.
    - Method chaining produces clean bytecode with clear receiver references.
    
    ---
    
    ## **11. Important Points**
    
    - Avoids telescoping constructors.
    - Supports optional fields naturally.
    - Improves readability using fluent design.
    - Often used for immutable object creation.
    - Reduces risk of incorrect parameter ordering.
    - Works well with method chaining and fluent APIs.
    
    ---
    
    ## **12. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Build complex objects step-by-step |
    | **Key Idea** | Use a builder class with chained methods |
    | **Strength** | Readable, flexible object creation |
    | **Common Use** | Immutable objects, many optional parameters |
    
    **In summary:**
    
    The Builder Pattern provides a clean, expressive, and flexible way to construct complex objects, making it a preferred choice in modern Java applications and frameworks.
    
- Prototype (shallow vs deep copy)
    
    # **Prototype Pattern (Shallow vs Deep Copy)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Prototype Pattern** is a creational design pattern used to create new objects by **copying existing objects (prototypes)** instead of instantiating them from scratch.
    
    It relies on **cloning** to efficiently create duplicates with the same internal state.
    
    ### **Purpose**
    
    - Improve performance when object creation is expensive
    - Simplify instances with many configuration steps
    - Preserve state while producing new instances
    - Support dynamic creation without knowing concrete classes
    
    ---
    
    ## **2. Key Concepts**
    
    ### **Prototype**
    
    An object that provides a method to clone itself.
    
    ### **Client**
    
    Requests a clone instead of constructing a new object.
    
    ### **Copy Types**
    
    - **Shallow Copy**
    - **Deep Copy**
    
    ---
    
    ## **3. Shallow Copy**
    
    ### **Definition**
    
    Shallow copy **duplicates only the top-level structure**, not nested objects.
    
    References inside the object are **copied**, not cloned.
    
    ### **Example**
    
    ```java
    class Address {
        String city;
    }
    
    class Person implements Cloneable {
        String name;
        Address address;
    
        public Person clone() throws CloneNotSupportedException {
            return (Person) super.clone();   // shallow copy
        }
    }
    
    ```
    
    ### **Behavior**
    
    ```java
    Person p1 = new Person("Ayush", new Address("Delhi"));
    Person p2 = p1.clone();
    
    p2.address.city = "Mumbai";
    
    System.out.println(p1.address.city);  // Mumbai (modified!)
    
    ```
    
    Both clones share the same nested object.
    
    ---
    
    ## **4. Deep Copy**
    
    ### **Definition**
    
    Deep copy duplicates the entire object graph, including nested objects.
    
    Every referenced object is independently cloned.
    
    ### **Example**
    
    ```java
    class Person implements Cloneable {
        String name;
        Address address;
    
        public Person clone() throws CloneNotSupportedException {
            Person copy = (Person) super.clone();
            copy.address = new Address(this.address.city); // deep copy
            return copy;
        }
    }
    
    ```
    
    ### **Behavior**
    
    ```java
    p2.address.city = "Mumbai";
    System.out.println(p1.address.city);  // Delhi (unchanged)
    
    ```
    
    Deep copy ensures full separation.
    
    ---
    
    ## **5. Prototype Implementation in Java**
    
    ### **Using Cloneable Interface**
    
    Java provides `Cloneable`, but it's somewhat problematic:
    
    - `clone()` is protected in `Object`
    - Requires handling of `CloneNotSupportedException`
    - Does not enforce deep copying
    - Many consider it a “broken” design
    
    ### **Basic Implementation**
    
    ```java
    class Prototype implements Cloneable {
        @Override
        protected Object clone() throws CloneNotSupportedException {
            return super.clone();
        }
    }
    
    ```
    
    ---
    
    ### **Custom Clone Without Cloneable**
    
    A preferred modern approach uses:
    
    - Copy constructors
    - Factory methods
    - Serialization for deep copy
    - Manual copying
    
    ### **Copy Constructor Example**
    
    ```java
    class Person {
        Person(Person other) {
            this.name = other.name;
            this.address = new Address(other.address);
        }
    }
    
    ```
    
    ---
    
    ## **6. Prototype Pattern Example**
    
    ### **Abstract Prototype**
    
    ```java
    interface Shape extends Cloneable {
        Shape clone();
        void draw();
    }
    
    ```
    
    ### **Concrete Prototype**
    
    ```java
    class Circle implements Shape {
        int radius;
    
        public Circle(int radius) {
            this.radius = radius;
        }
    
        public Circle(Circle other) {  // deep copy via copy constructor
            this.radius = other.radius;
        }
    
        public Shape clone() {
            return new Circle(this);
        }
    
        public void draw() {
            System.out.println("Drawing circle: " + radius);
        }
    }
    
    ```
    
    ### **Usage**
    
    ```java
    Shape original = new Circle(10);
    Shape copy = original.clone();
    copy.draw();
    
    ```
    
    ---
    
    ## **7. When to Use Prototype Pattern**
    
    - Object creation is expensive (e.g., database calls, large setups)
    - Many similar objects are needed
    - System must be decoupled from classes of objects
    - Want to avoid subclasses solely for instantiation
    
    ---
    
    ## **8. Prototype vs Builder vs Factory**
    
    | Pattern | When to Use |
    | --- | --- |
    | **Prototype** | Need clones of existing configured objects |
    | **Builder** | Objects require many optional parameters |
    | **Factory Method** | Let subclasses decide which object to create |
    | **Abstract Factory** | Create related families of objects |
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Shallow copy | Photocopy with attached sticky notes copied as position only |
    | Deep copy | Rewriting entire content with independent sticky notes |
    | Prototype | Using an existing template to produce new versions |
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - `clone()` uses a native method for fast memory copying.
    - Primitive fields copied directly; references duplicated.
    - Deep copy requires manual cloning of referenced objects.
    - Serialization-based deep copy is slow but guarantees correctness.
    
    ---
    
    ## **11. Important Points**
    
    - Shallow copy shares nested objects → dangerous for mutability.
    - Deep copy duplicates everything → safe but more expensive.
    - `Cloneable` is widely considered flawed; use copy constructors when possible.
    - Prototype simplifies creation of pre-configured objects.
    
    ---
    
    ## **12. Summary**
    
    | Concept | Shallow Copy | Deep Copy |
    | --- | --- | --- |
    | Copies nested objects? | ❌ No | ✔️ Yes |
    | Safe for mutable fields? | ❌ No | ✔️ Yes |
    | Performance | Fast | Slower |
    | Typical use | Lightweight prototypes | Full independence required |
    
    **In summary:**
    
    The Prototype Pattern provides a mechanism for cloning objects efficiently. Understanding shallow vs deep copy is critical, as deep copying ensures independent clone behavior, whereas shallow copying can cause shared, unintended state between objects.
    

### **5.3 Structural Patterns**

- Adapter Pattern
    
    # **Adapter Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Adapter Pattern** is a structural design pattern that allows **incompatible interfaces** to work together.
    
    It acts as a **bridge** between two separate interfaces by converting one interface into another that the client expects.
    
    ### **Purpose**
    
    - Enable the reuse of existing classes with incompatible interfaces
    - Integrate legacy systems with new code
    - Promote flexibility and loose coupling
    - Avoid modifying existing or external classes
    
    ---
    
    ## **2. Why Adapter Pattern?**
    
    Suppose you have:
    
    - A client expecting interface **A**
    - A service providing interface **B**
    
    Direct communication is impossible due to incompatible methods.
    
    The adapter solves this by translating calls.
    
    ---
    
    ## **3. Types of Adapters**
    
    ### **A. Class Adapter (Using Inheritance)**
    
    - Uses multiple inheritance (extends adaptee, implements target)
    - Not possible directly in Java (single inheritance)
    - Achievable only when adaptee is a class and not final
    
    ### **B. Object Adapter (Using Composition) — Common in Java**
    
    - Holds an instance of the adaptee
    - Delegates calls by converting the interface
    - Most commonly used approach in Java and Spring
    
    ---
    
    ## **4. Structure**
    
    ### **Target Interface**
    
    What the client expects.
    
    ```java
    interface MediaPlayer {
        void play(String file);
    }
    
    ```
    
    ### **Adaptee (Incompatible Interface)**
    
    ```java
    class AdvancedPlayer {
        void playMp4(String file) {
            System.out.println("Playing MP4: " + file);
        }
    }
    
    ```
    
    ### **Adapter**
    
    Converts Target → Adaptee.
    
    ```java
    class MediaAdapter implements MediaPlayer {
        private AdvancedPlayer advancedPlayer = new AdvancedPlayer();
    
        @Override
        public void play(String file) {
            advancedPlayer.playMp4(file); // conversion logic
        }
    }
    
    ```
    
    ### **Client**
    
    ```java
    class AudioPlayer implements MediaPlayer {
        @Override
        public void play(String file) {
            new MediaAdapter().play(file);
        }
    }
    
    ```
    
    ---
    
    ## **5. Realistic Example — Java I/O Streams**
    
    Java's I/O library heavily uses adapters:
    
    - `InputStreamReader` adapts a **byte stream** (`InputStream`) into a **character stream** (`Reader`).
    - `BufferedReader` adapts a simple `Reader` to provide buffering.
    
    Example:
    
    ```java
    Reader r = new InputStreamReader(System.in);
    
    ```
    
    ---
    
    ## **6. Adapter Pattern in Spring Framework**
    
    ### **Spring MVC — HandlerAdapter**
    
    Converts various controller styles (annotated, simple, HttpRequestHandler) into a unified interface.
    
    ### **Other Examples**
    
    - `WebMvcConfigurerAdapter` (legacy)
    - `PropertySourcesAdapter`
    - `EnvironmentCapable` adapters
    
    Spring uses adapters to integrate different components seamlessly.
    
    ---
    
    ## **7. When to Use Adapter Pattern**
    
    - When using existing classes whose interfaces do not match required APIs
    - When integrating third-party libraries
    - When migrating legacy systems
    - When working with multiple incompatible APIs
    
    ---
    
    ## **8. Adapter vs Decorator vs Facade**
    
    | Pattern | Purpose |
    | --- | --- |
    | **Adapter** | Convert one interface to another |
    | **Decorator** | Add new behavior without changing interface |
    | **Facade** | Simplify complex subsystem (does not change interface) |
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Adapter | A plug adapter between different socket types |
    | Target interface | The socket required by your device |
    | Adaptee | Foreign plug that doesn't fit |
    | Client | Your device |
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Object adapters allow composition → flexible and memory-efficient.
    - No JVM overhead besides simple delegation.
    - Class adapters limited by Java’s lack of multiple inheritance.
    
    ---
    
    ## **11. Important Points**
    
    - Adapter converts **interface**, not behavior.
    - Object adapter (composition) is recommended in Java.
    - Useful for third-party and legacy integration.
    - Promotes loose coupling and clean architecture.
    - Avoid modifying adaptee whenever possible.
    
    ---
    
    ## **12. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Allow incompatible interfaces to work together |
    | **Key Idea** | Create a bridge (adapter) converting calls |
    | **Used In** | Java I/O, Spring MVC, third-party integrations |
    | **Benefit** | Reuse code without modification |
    
    **In summary:**
    
    The Adapter Pattern enables seamless interaction between incompatible interfaces using either inheritance or composition, making it a central tool for system integration and legacy modernization.
    
- Facade Pattern
    
    # **Facade Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Facade Pattern** is a structural design pattern that provides a **simple, unified interface** to a **complex subsystem**.
    
    It hides internal complexities and exposes only the necessary functionality to the client.
    
    ### **Purpose**
    
    - Simplify interactions with complex or multiple subsystems
    - Provide a high-level interface for easy usage
    - Reduce dependencies between client and subsystem
    - Improve readability and maintainability
    
    ---
    
    ## **2. Why Facade Pattern?**
    
    Imagine a system with multiple steps and components:
    
    - Database operations
    - File processing
    - Network communication
    - Logging and validation
    
    The client would need to interact with multiple classes, making the code complex and tightly coupled.
    
    **Facade** acts as a single entry point, coordinating internal operations.
    
    ---
    
    ## **3. Structure**
    
    ### **Subsystem Classes**
    
    These contain complex logic and internal operations.
    
    ```java
    class AudioMixer {
        public void mix() { System.out.println("Mixing audio..."); }
    }
    
    class VideoEncoder {
        public void encode() { System.out.println("Encoding video..."); }
    }
    
    class FileExporter {
        public void export() { System.out.println("Exporting file..."); }
    }
    
    ```
    
    ### **Facade**
    
    Provides a simple interface hiding subsystem complexity.
    
    ```java
    class MediaProcessorFacade {
        private AudioMixer audio = new AudioMixer();
        private VideoEncoder video = new VideoEncoder();
        private FileExporter exporter = new FileExporter();
    
        public void processMedia() {
            audio.mix();
            video.encode();
            exporter.export();
        }
    }
    
    ```
    
    ### **Client**
    
    Interacts only with the Facade.
    
    ```java
    public class Client {
        public static void main(String[] args) {
            MediaProcessorFacade processor = new MediaProcessorFacade();
            processor.processMedia();
        }
    }
    
    ```
    
    ---
    
    ## **4. Benefits of Facade Pattern**
    
    - **Simplifies** complex subsystems
    - **Reduces coupling** (client depends on one class instead of many)
    - **Improves maintainability**
    - Provides **clear and structured workflows**
    - **Hides unnecessary details** from client
    - Good for **layered architecture**
    
    ---
    
    ## **5. When to Use the Facade Pattern**
    
    - When you want to provide a simplified API for a complex subsystem
    - When multiple modules need to interact but should remain decoupled
    - When you want to shield client code from internal changes
    - When integrating third-party libraries with messy APIs
    - When designing layered architectures (controller → service → DAO)
    
    ---
    
    ## **6. Facade Pattern in Java**
    
    ### **A. Java Standard Library**
    
    - `javax.faces.context.FacesContext` simplifies JSF lifecycle
    - `java.net` hides OS-level networking details
    - JDBC’s `DriverManager` acts as a facade for different drivers
    
    ### **B. File I/O Example**
    
    `Files` class in `java.nio.file.Files` offers high-level file operations hiding internal complexities:
    
    ```java
    Files.copy(src, dest);
    Files.move(src, dest);
    
    ```
    
    ---
    
    ## **7. Facade Pattern in Spring Framework**
    
    Spring uses Facades widely to simplify complex processes.
    
    ### **Examples**
    
    | Facade | What It Simplifies |
    | --- | --- |
    | `JdbcTemplate` | Database operations (statement, connection, result set handling) |
    | `RestTemplate` / `WebClient` | HTTP operations |
    | `HibernateTemplate` (legacy) | ORM operations |
    | `ApplicationContext` | Bean creation and dependency injection |
    | `MockMvc` | Testing Spring MVC without full server |
    
    Spring hides the complexities of:
    
    - Connection pooling
    - Transactions
    - Serialization
    - Error handling
    - Resource cleanup
    
    ---
    
    ## **8. Facade vs Adapter vs Mediator**
    
    | Pattern | Purpose |
    | --- | --- |
    | **Facade** | Provide a simple interface to a complex subsystem |
    | **Adapter** | Convert one interface into another |
    | **Mediator** | Coordinates communication between objects |
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Component | Analogy |
    | --- | --- |
    | Subsystem | Different departments in a company |
    | Facade | Customer service representative |
    | Client | Customer who interacts with one representative |
    
    The facade handles all internal routing so the client doesn't have to.
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Facade introduces no performance overhead; it's only a wrapper.
    - JIT optimizes through method inlining when facade methods are small.
    - Helps maintain modular and clean bytecode in layered architecture.
    
    ---
    
    ## **11. Important Points**
    
    - Facade provides **only a simplified interface**, not new functionality.
    - Subsystems remain accessible if needed.
    - Encourages clean, layered design.
    - Multiple facades can exist for the same subsystem.
    - Ideal for legacy system integration.
    
    ---
    
    ## **12. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Simplify subsystem with a unified interface |
    | **Key Idea** | Wrap complex logic behind a single entry point |
    | **Usage** | Java I/O, JDBC, Spring templates |
    | **Benefit** | Loose coupling and improved readability |
    
    **In summary:**
    
    The Facade Pattern provides a clean, unified interface that hides complex subsystem interactions, greatly improving usability, maintainability, and modularity in Java and enterprise-level applications.
    
- Proxy Pattern
    
    # **Proxy Pattern (with Spring AOP Integration)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Proxy Pattern** is a structural design pattern where a **substitute or placeholder object** controls access to another object (called the *real subject*).
    
    The proxy adds a layer between the client and the actual object, enabling additional behavior such as:
    
    - Access control
    - Logging
    - Caching
    - Remote access
    - Lazy initialization
    - Security checks
    
    ### **Purpose**
    
    - Intercept and manage requests to the underlying object
    - Add functionality **without modifying the real object's code**
    - Support AOP-like behavior in object interaction
    
    ---
    
    ## **2. Types of Proxies**
    
    ### **A. Virtual Proxy**
    
    Creates expensive objects lazily.
    
    Example: Lazy-loading images in GUI.
    
    ### **B. Protection Proxy**
    
    Controls access (authorization, role-based access).
    
    ### **C. Remote Proxy**
    
    Represents an object located remotely (like RMI stubs).
    
    ### **D. Smart Proxy**
    
    Adds extra actions before/after method calls (logging, tracking).
    
    ### **E. Cache Proxy**
    
    Caches method results and serves repeated requests faster.
    
    ---
    
    ## **3. Basic Structure**
    
    ### **Subject Interface**
    
    ```java
    interface Service {
        void request();
    }
    
    ```
    
    ### **Real Subject**
    
    ```java
    class RealService implements Service {
        public void request() {
            System.out.println("RealService processing request");
        }
    }
    
    ```
    
    ### **Proxy**
    
    ```java
    class ServiceProxy implements Service {
        private RealService realService;
    
        public void request() {
            if (realService == null) {
                realService = new RealService(); // lazy initialization
            }
            System.out.println("Logging: before request");
            realService.request();
            System.out.println("Logging: after request");
        }
    }
    
    ```
    
    ### **Client**
    
    ```java
    Service service = new ServiceProxy();
    service.request();
    
    ```
    
    ---
    
    ## **4. Advantages of Proxy Pattern**
    
    - Adds behavior without changing actual class code
    - Supports lazy loading
    - Facilitates access control
    - Enables distributed systems (RPC, gRPC, RMI)
    - Helps with performance monitoring, caching, etc.
    
    ---
    
    ## **5. When to Use Proxy Pattern**
    
    - When you need additional logic around method calls
    - When object creation is expensive
    - When working with remote services
    - When centralizing access control or logging
    - When implementing AOP-like behavior manually
    
    ---
    
    # **6. Proxy Pattern in Spring AOP**
    
    Spring AOP is one of the most practical and widely used applications of the Proxy Pattern.
    
    ---
    
    ## **A. How Spring AOP Uses Proxy Pattern**
    
    Spring AOP wraps target beans with **proxy objects**.
    
    When the client calls a method, the proxy intercepts the call, applies advice (before, after, around), and then forwards the call to the actual target object.
    
    ---
    
    ## **B. Types of Proxies Used by Spring**
    
    ### **1. JDK Dynamic Proxy (Interface-based Proxy)**
    
    - Used when the target bean implements at least one interface.
    - Proxy class implements the same interface.
    - Uses `java.lang.reflect.Proxy` and `InvocationHandler`.
    
    ### **2. CGLIB Proxy (Class-based Proxy)**
    
    - Used when the target class has no interfaces.
    - Creates a subclass of the target at runtime.
    - Enabled by:
        - Spring Boot (default if needed)
        - `@EnableAspectJAutoProxy(proxyTargetClass = true)`
    
    ---
    
    ## **C. Spring AOP Interceptor Chain**
    
    Spring applies proxies for:
    
    - `@Transactional`
    - `@Cacheable`
    - `@Async`
    - `@Secured`
    - Custom AOP advice (`@Aspect` classes)
    
    Each call flows like:
    
    ```
    Client → Proxy → Advice → Target Bean
    
    ```
    
    ---
    
    ## **D. Example of Spring AOP Proxy**
    
    ### **Target Bean**
    
    ```java
    @Service
    public class PaymentService {
        public void process() {
            System.out.println("Payment processed");
        }
    }
    
    ```
    
    ### **Aspect**
    
    ```java
    @Aspect
    @Component
    public class LoggingAspect {
        @Before("execution(* com.app.PaymentService.*(..))")
        public void log() {
            System.out.println("Logging before payment");
        }
    }
    
    ```
    
    ### **Runtime Behavior**
    
    When calling:
    
    ```java
    paymentService.process();
    
    ```
    
    Actual execution is:
    
    ```
    Proxy → log() → Real process()
    
    ```
    
    ---
    
    ## **E. Why Spring Prefers Proxy Pattern for AOP?**
    
    1. **Lightweight** — uses runtime bytecode generation
    2. **Non-invasive** — does not modify target logic
    3. **Fast** — avoids modifying actual classes
    4. **Flexible** — supports multiple advices per method
    5. **Safe** — respects Java access and type system
    
    ---
    
    ## **F. Proxy Limitations in Spring AOP**
    
    - Cannot apply advice to **private** or **final** methods
    - Self-invocation (calling method inside same class) bypasses proxy
    - CGLIB cannot subclass final classes
    - JDK proxy requires interfaces
    
    These are important architectural constraints.
    
    ---
    
    # **7. Proxy vs Decorator vs Facade**
    
    | Pattern | Purpose |
    | --- | --- |
    | **Proxy** | Controls access; adds cross-cutting features |
    | **Decorator** | Adds behavior dynamically without controlling access |
    | **Facade** | Simplifies complex subsystem behind one interface |
    
    ---
    
    ## **8. Real-world Analogy**
    
    | Proxy Type | Analogy |
    | --- | --- |
    | Virtual Proxy | Netflix shows a thumbnail and loads video later |
    | Protection Proxy | Security guard checking identity before entry |
    | Smart Proxy | Courier tracking updates before delivery |
    | Remote Proxy | ATM talking to remote bank server |
    
    ---
    
    ## **9. JVM / Compiler Notes**
    
    - JDK dynamic proxy uses reflection (slower invocation).
    - CGLIB uses bytecode generation (faster after warm-up).
    - Spring chooses proxy mechanism automatically based on class/interface availability.
    - JIT inlines repeated calls for faster execution.
    
    ---
    
    ## **10. Important Points**
    
    - Proxy controls access to target objects.
    - Spring AOP heavily relies on proxy pattern.
    - Two proxy mechanisms: JDK Dynamic Proxy & CGLIB.
    - Proxies enable modular cross-cutting concerns (logging, security, transactions).
    - Avoids code duplication and improves maintainability.
    
    ---
    
    ## **11. Summary**
    
    | Feature | Explanation |
    | --- | --- |
    | **Intent** | Provide a surrogate/placeholder to control access |
    | **Key Benefit** | Intercept method calls without modifying target |
    | **Used In** | Spring AOP, RMI, security checks, caching |
    | **Spring Relation** | Spring AOP implements advice using proxies |
    
    **In summary:**
    
    The Proxy Pattern is central to Spring’s AOP system, enabling cross-cutting features like logging, transactions, and security without modifying business logic. It acts as an interceptor layer that wraps and enhances the behavior of targeted objects.
    
- Decorator Pattern
    
    # **Decorator Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Decorator Pattern** is a structural design pattern that allows behavior to be **dynamically added** to an object **without modifying its original class**.
    
    It wraps an existing object inside another object, extending or enhancing its functionality.
    
    ### **Purpose**
    
    - Add responsibilities at runtime
    - Keep the original class simple
    - Avoid subclass explosion
    - Promote composition over inheritance
    - Increase flexibility in functionality extension
    
    ---
    
    ## **2. When to Use Decorator Pattern**
    
    - When you want to add responsibilities to objects *dynamically*
    - When subclassing leads to large class hierarchies
    - When you want to follow the Open–Closed Principle (extend behavior without modifying classes)
    - When multiple combinations of behaviors are needed
    
    ---
    
    ## **3. Structure**
    
    ### **Component (Interface or Abstract Class)**
    
    Defines the base type.
    
    ```java
    interface Coffee {
        String getDescription();
        double getCost();
    }
    
    ```
    
    ### **Concrete Component**
    
    Actual object to be decorated.
    
    ```java
    class SimpleCoffee implements Coffee {
        public String getDescription() { return "Simple Coffee"; }
        public double getCost() { return 50; }
    }
    
    ```
    
    ### **Decorator (Base Wrapper Class)**
    
    Wraps a component and delegates calls.
    
    ```java
    abstract class CoffeeDecorator implements Coffee {
        protected Coffee coffee;
    
        public CoffeeDecorator(Coffee coffee) {
            this.coffee = coffee;
        }
    }
    
    ```
    
    ### **Concrete Decorators**
    
    Add new behavior.
    
    ```java
    class MilkDecorator extends CoffeeDecorator {
        public MilkDecorator(Coffee c) { super(c); }
    
        public String getDescription() {
            return coffee.getDescription() + ", Milk";
        }
    
        public double getCost() {
            return coffee.getCost() + 10;
        }
    }
    
    class SugarDecorator extends CoffeeDecorator {
        public SugarDecorator(Coffee c) { super(c); }
    
        public String getDescription() {
            return coffee.getDescription() + ", Sugar";
        }
    
        public double getCost() {
            return coffee.getCost() + 5;
        }
    }
    
    ```
    
    ### **Client Usage**
    
    ```java
    Coffee coffee = new SimpleCoffee();
    coffee = new MilkDecorator(coffee);
    coffee = new SugarDecorator(coffee);
    
    System.out.println(coffee.getDescription());
    System.out.println(coffee.getCost());
    
    ```
    
    ---
    
    ## **4. Behavior**
    
    - Decorators wrap around components
    - Additional functionality is added without changing original component
    - Can apply multiple decorators at runtime
    - Result is a chain of decorators
    
    ---
    
    ## **5. Advantages of Decorator Pattern**
    
    - Adds behavior without modifying existing classes
    - No need for large inheritance hierarchies
    - Highly flexible; decorators can be composed in any order
    - Supports runtime extension of behavior
    - Conforms to the **Open–Closed Principle**
    
    ---
    
    ## **6. Disadvantages**
    
    - More objects created → complexity increases
    - Debugging can be harder (nested wrappers)
    - Order of decorators matters
    
    ---
    
    ## **7. Real-world Examples in Java**
    
    ### **A. Java I/O Library**
    
    The most famous example:
    
    ```java
    InputStream is =
        new BufferedInputStream(
            new DataInputStream(
                new FileInputStream("file.txt")));
    
    ```
    
    Here:
    
    - `FileInputStream` = concrete component
    - `DataInputStream`, `BufferedInputStream` = decorators
    
    ### **B. Collections**
    
    ```java
    List<String> syncList = Collections.synchronizedList(list);
    List<String> unmodifiableList = Collections.unmodifiableList(list);
    
    ```
    
    Both return **decorated versions** of the original list.
    
    ---
    
    ## **8. Decorator Pattern vs Proxy vs Adapter**
    
    | Pattern | Purpose |
    | --- | --- |
    | **Decorator** | Add new behavior dynamically without modifying class |
    | **Proxy** | Control access (security, lazy loading, logging) |
    | **Adapter** | Convert interface into another |
    
    Decorator → “Enhance behavior”
    
    Proxy → “Control access”
    
    Adapter → “Translate interface”
    
    ---
    
    ## **9. Decorator Pattern in Spring**
    
    Spring uses decorators often:
    
    ### **A. Spring Security**
    
    `OncePerRequestFilter` and other filters wrap requests → filter chain.
    
    ### **B. Spring Web MVC**
    
    Wrapping request/response with customized behavior.
    
    ### **C. AOP Advice**
    
    Decorators wrap method execution to perform extra behavior.
    
    ### **D. BeanPostProcessor**
    
    Wraps beans to add additional behavior such as proxies → decorator-like behavior.
    
    ---
    
    ## **10. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Component | Pizza base |
    | Decorators | Toppings (cheese, olives, extra sauce) |
    | Client | You assembling pizza with custom toppings |
    
    Each topping is added **without changing** the base pizza.
    
    ---
    
    ## **11. JVM / Compiler Notes**
    
    - Decorators use delegation; JVM efficiently optimizes calls via inlining.
    - Depth of decorator chain affects stack trace readability.
    - Minimal memory overhead because decorators generally store only wrapped object.
    
    ---
    
    ## **12. Important Points**
    
    - Wraps an object to add or modify behavior.
    - Uses composition, not inheritance.
    - Supports runtime behavior modification.
    - Java I/O is a canonical example.
    - Best for flexible feature combinations.
    
    ---
    
    ## **13. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Dynamically add behavior to objects by wrapping |
    | **Key Idea** | Composition of objects (wrapping layers) |
    | **Benefits** | Flexible extension, avoids subclass explosion |
    | **Used In** | Java I/O, Collections, Spring filters, AOP |
    
    **In summary:**
    
    The Decorator Pattern provides a flexible, runtime-friendly mechanism to add functionality to objects without modifying existing code, making it one of the most powerful structural patterns in Java.
    
- Composite Pattern
    
    # **Composite Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Composite Pattern** is a structural design pattern that allows you to treat **individual objects (leaves)** and **collections of objects (composites)** uniformly.
    
    It represents part–whole hierarchies using a **tree structure**, where both simple and complex elements share a **common interface**.
    
    ### **Purpose**
    
    - Model hierarchical structures (trees)
    - Treat individual and grouped objects the same
    - Simplify client code by allowing uniform access
    - Promote composition over inheritance
    
    ---
    
    ## **2. When to Use Composite Pattern**
    
    - When you need to represent a **tree-like** structure: menus, folders, UI components
    - When clients should not differentiate between single items and groups
    - When you want to simplify operations across the hierarchy
    - When operations need to be applied recursively to a group of objects
    
    ---
    
    ## **3. Structure**
    
    ### **Component (Common Interface)**
    
    Defines operations shared by both leaf and composite objects.
    
    ```java
    interface Graphic {
        void draw();
    }
    
    ```
    
    ### **Leaf (Simple Object)**
    
    Simple element that does not contain children.
    
    ```java
    class Circle implements Graphic {
        public void draw() {
            System.out.println("Drawing Circle");
        }
    }
    
    ```
    
    ### **Composite (Container Object)**
    
    Holds a collection of children (leaf or composite).
    
    ```java
    class GraphicGroup implements Graphic {
        private List<Graphic> children = new ArrayList<>();
    
        public void add(Graphic g) {
            children.add(g);
        }
    
        public void remove(Graphic g) {
            children.remove(g);
        }
    
        public void draw() {
            for (Graphic g : children) {
                g.draw();
            }
        }
    }
    
    ```
    
    ### **Client**
    
    ```java
    Graphic circle = new Circle();
    
    GraphicGroup group = new GraphicGroup();
    group.add(new Circle());
    group.add(circle);
    
    group.draw();   // Calls draw() on each child
    
    ```
    
    ---
    
    ## **4. Key Characteristics**
    
    - Common interface for both **Leaf** and **Composite**
    - Composite can hold **multiple children**
    - Client does not care if it’s a single object or a group
    - Often used with **recursive method calls**
    
    ---
    
    ## **5. Real-world Examples**
    
    ### **A. File System**
    
    - File = Leaf
    - Folder = Composite
    - Both implement operations like: open, delete, size, rename
    
    ### **B. UI Components**
    
    - Button, Label, TextBox = Leaf
    - Panels, Frames = Composite
    - Swing/AWT uses composite-based component trees
    
    ### **C. XML/HTML Trees**
    
    - Elements inside elements
    - Text nodes vs container nodes
    
    ### **D. Organization Hierarchy**
    
    - Employee = Component
    - Manager = Composite
    
    ---
    
    ## **6. Composite Pattern in Java**
    
    ### **Swing / AWT**
    
    `Container` vs `Component` with methods like:
    
    ```java
    add(Component comp);
    
    ```
    
    ### **JavaFX**
    
    Scene graph is a tree (composite pattern).
    
    ### **DOM API**
    
    `Node`, `Element`, `Text` implement same interface.
    
    ---
    
    ## **7. Composite Pattern in Spring**
    
    ### **Spring Security**
    
    The `AccessDecisionManager` uses a composite-like pattern:
    
    - `AccessDecisionVoter` (Leaf)
    - `AffirmativeBased`, `ConsensusBased` (Composite)
    
    ### **ApplicationContext Hierarchy**
    
    Parent-child context structure follows composite semantics.
    
    ### **HandlerExecutionChain**
    
    Combines interceptors in chain structure.
    
    ---
    
    ## **8. Advantages**
    
    - Uniform treatment of single/multiple objects
    - Simplifies client code
    - Easy to add new components (Open–Closed Principle)
    - Supports hierarchical structures naturally
    
    ---
    
    ## **9. Disadvantages**
    
    - Harder to restrict operations (e.g., leaves shouldn't have `add()`)
    - Can make design too general
    - Debugging recursive behavior can be difficult
    
    ---
    
    ## **10. Composite Pattern Variants**
    
    ### **A. Transparent Composite**
    
    - Leaf and Composite share *exact same* interface
    - Composite exposes `add()` and `remove()`
    - Simple for client, but leaf methods may throw exceptions
    
    ### **B. Safe Composite**
    
    - Only Composite exposes `add()` and `remove()`
    - Cleaner API but more casting at client side
    
    ---
    
    ## **11. Real-world Analogy**
    
    | Component | Analogy |
    | --- | --- |
    | Leaf | Employee in a company |
    | Composite | Manager handling multiple employees |
    | Client | HR department querying structure |
    | Operation | "Show salary" applied uniformly |
    
    Manager and employee both respond to the same operation.
    
    ---
    
    ## **12. JVM / Compiler Notes**
    
    - Composite operations rely on dynamic dispatch for children.
    - Recursive calls optimized well by JIT when small and predictable.
    - Large composite structures can lead to deep recursion—tail-call elimination not supported in JVM.
    
    ---
    
    ## **13. Important Points**
    
    - Represents part–whole hierarchies using tree structures.
    - Treats leaves and composites uniformly.
    - Often implemented with recursive operations.
    - Very common in UI frameworks, parsers, and hierarchical models.
    - Choose between transparent or safe composite based on design needs.
    
    ---
    
    ## **14. Summary**
    
    | Concept | Description |
    | --- | --- |
    | **Intent** | Represent trees with uniform treatment for components |
    | **Key Idea** | Composition of objects in part–whole structure |
    | **Used In** | File systems, UI trees, DOM, organization charts |
    | **Benefit** | Simplifies client code; supports flexible hierarchies |
    
    **In summary:**
    
    The Composite Pattern provides a clean, recursive structure for representing objects in tree-like hierarchies, enabling uniform treatment of individual and grouped components—essential for UI, filesystem, and structural object modeling in Java.
    

### **5.4 Behavioral Patterns**

- Strategy Pattern
    
    # **Strategy Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Strategy Pattern** is a behavioral design pattern that defines a **family of algorithms**, encapsulates each algorithm separately, and allows them to be **interchanged at runtime**.
    
    Instead of embedding many conditional statements inside a class, Strategy delegates behavior to separate strategy objects.
    
    ### **Purpose**
    
    - Replace multiple `if-else` or `switch` blocks
    - Allow dynamic selection of algorithm
    - Promote flexibility and maintainability
    - Encapsulate behavior cleanly
    - Support Open–Closed Principle (new strategies without modifying existing code)
    
    ---
    
    ## **2. When to Use Strategy Pattern**
    
    - When a class has multiple algorithms/behaviors that can vary
    - When you want to switch logic dynamically (e.g., runtime configuration)
    - When you want to remove big conditional statements
    - When behavior should be easily replaceable and testable
    
    Common uses:
    
    - Payment methods
    - Sorting algorithms
    - Compression/encryption strategies
    - Validation strategies
    - Routing/selection rules
    
    ---
    
    ## **3. Structure**
    
    ### **A. Strategy Interface**
    
    Defines the behavior.
    
    ```java
    interface PaymentStrategy {
        void pay(int amount);
    }
    
    ```
    
    ### **B. Concrete Strategies**
    
    Each class provides different behavior.
    
    ```java
    class CreditCardPayment implements PaymentStrategy {
        public void pay(int amount) {
            System.out.println("Paid " + amount + " using Credit Card");
        }
    }
    
    class UpiPayment implements PaymentStrategy {
        public void pay(int amount) {
            System.out.println("Paid " + amount + " using UPI");
        }
    }
    
    class PayPalPayment implements PaymentStrategy {
        public void pay(int amount) {
            System.out.println("Paid " + amount + " via PayPal");
        }
    }
    
    ```
    
    ### **C. Context Class**
    
    Holds reference to a strategy and delegates work.
    
    ```java
    class PaymentContext {
        private PaymentStrategy strategy;
    
        public void setStrategy(PaymentStrategy strategy) {
            this.strategy = strategy;
        }
    
        public void checkout(int amount) {
            strategy.pay(amount);
        }
    }
    
    ```
    
    ### **Client Usage**
    
    ```java
    PaymentContext context = new PaymentContext();
    
    context.setStrategy(new CreditCardPayment());
    context.checkout(500);
    
    context.setStrategy(new UpiPayment());
    context.checkout(300);
    
    ```
    
    ---
    
    ## **4. Benefits of Strategy Pattern**
    
    - No need for long `switch` or `if-else` chains
    - Algorithms can be added without modifying client code
    - Strategies can be swapped at runtime
    - Improves code readability and maintainability
    - Encourages interface-based design
    
    ---
    
    ## **5. Drawbacks**
    
    - More classes (one per strategy)
    - Client must understand and select appropriate strategy
    - Harder when strategies require shared state
    
    ---
    
    ## **6. Real-world Examples in Java**
    
    ### **A. Java’s Sorting Algorithms**
    
    `Comparator<T>` acts as a Strategy.
    
    ```java
    Collections.sort(list, comparator);
    
    ```
    
    Different comparators = different sorting strategy.
    
    ### **B. Java Streams**
    
    ```java
    list.stream().sorted(Comparator.comparing(Person::getName));
    
    ```
    
    ### **C. Thread Pool Rejection Policies**
    
    `RejectedExecutionHandler` is a strategy.
    
    ### **D. `java.util.Formatter`**
    
    Different formatting strategies.
    
    ---
    
    ## **7. Strategy Pattern in Spring**
    
    Spring uses Strategy Pattern extensively:
    
    ### **A. Spring Bean Injection**
    
    You can inject different strategy implementations using `@Qualifier`.
    
    ### **B. Spring Security – Password Encoding Strategy**
    
    ```java
    PasswordEncoder encoder = new BCryptPasswordEncoder();
    
    ```
    
    ### **C. Spring MVC – View Resolvers**
    
    Each resolver is a strategy for rendering views.
    
    ### **D. Cache Eviction Policies**
    
    Pluggable strategies for expiry, eviction.
    
    ### **E. HandlerMapping, HandlerAdapter**
    
    Different request handling algorithms.
    
    Spring core is filled with pluggable strategies.
    
    ---
    
    ## **8. Removing Conditional Logic with Strategy**
    
    Instead of:
    
    ```java
    if(type.equals("UPI")) payUpi();
    else if(type.equals("CARD")) payCard();
    else if(type.equals("PAYPAL")) payPaypal();
    
    ```
    
    Use strategy objects:
    
    ```java
    PaymentStrategy strategy = strategyMap.get(type);
    strategy.pay(amount);
    
    ```
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Strategy Concept | Analogy |
    | --- | --- |
    | Strategy | Different travel routes (road, train, air) |
    | Context | Your navigation app |
    | Switching strategy | Choosing fastest/cheapest route |
    | Client | User selecting travel mode |
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Strategy objects are often inlined by JVM → low overhead
    - Polymorphism ensures dynamic dispatch of method calls
    - Frequent switching of strategies is lightweight and safe
    
    ---
    
    ## **11. Important Points**
    
    - Defines family of algorithms with interchangeable behavior
    - Eliminates conditionals
    - Promotes modular and maintainable code
    - Used heavily in Java APIs (Comparator, RejectedExecutionHandler)
    - Fits naturally with DI frameworks like Spring
    
    ---
    
    ## **12. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Encapsulate interchangeable behaviors |
    | **Key Idea** | Context delegates execution to a strategy |
    | **Strength** | Flexible, avoids conditionals |
    | **Used In** | Java sorting, Spring strategies, payment logic |
    
    **In summary:**
    
    The Strategy Pattern enables clean and interchangeable algorithm selection at runtime, making it one of the most practical behavioral patterns used in Java and enterprise application development.
    
- Template Method
    
    # **Template Method Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Template Method Pattern** is a behavioral design pattern that defines the **skeleton of an algorithm** in a method (called the *template method*), while allowing subclasses to **override certain steps** of the algorithm without changing its overall structure.
    
    ### **Purpose**
    
    - Establish a fixed algorithm workflow
    - Allow subclasses to customize specific steps
    - Promote code reuse by keeping invariant behavior in superclass
    - Enforce consistent execution order of operations
    
    ---
    
    ## **2. When to Use Template Method Pattern**
    
    - When multiple classes share a common algorithm with variations in certain steps
    - When the overall process remains constant but individual steps differ
    - When enforcing a specific sequence of operations is required
    - When avoiding code duplication across subclasses
    
    Common use cases:
    
    - File processing
    - Authentication flows
    - Data parsing
    - Algorithm pipelines
    - Transaction workflows
    
    ---
    
    ## **3. Structure**
    
    ### **Abstract Class (Template)**
    
    Contains:
    
    - Template method (final)
    - Concrete steps (optional)
    - Abstract or hook methods for subclasses
    
    ```java
    abstract class DataProcessor {
    
        // Template Method
        public final void process() {
            readData();
            processData();
            writeData();
        }
    
        void readData() {
            System.out.println("Reading data...");
        }
    
        abstract void processData();   // must be implemented by subclass
    
        void writeData() {
            System.out.println("Writing data...");
        }
    }
    
    ```
    
    ### **Concrete Subclasses**
    
    Implement variable steps of the algorithm.
    
    ```java
    class CsvDataProcessor extends DataProcessor {
        void processData() {
            System.out.println("Processing CSV data");
        }
    }
    
    class JsonDataProcessor extends DataProcessor {
        void processData() {
            System.out.println("Processing JSON data");
        }
    }
    
    ```
    
    ### **Client Usage**
    
    ```java
    DataProcessor dp = new CsvDataProcessor();
    dp.process();
    
    ```
    
    ---
    
    ## **4. Key Concepts**
    
    ### **Template Method**
    
    Defines a fixed algorithm structure.
    
    Marked `final` to prevent alteration of the workflow.
    
    ### **Abstract Methods**
    
    Steps that subclasses must provide.
    
    ### **Concrete Methods**
    
    Steps already implemented in superclass.
    
    ### **Hook Methods**
    
    Optional methods that subclasses *may* override.
    
    ```java
    void beforeProcessing() { }  // empty hook
    
    ```
    
    ---
    
    ## **5. Benefits**
    
    - Enforces consistent algorithm structure
    - Promotes code reuse
    - Removes duplication among subclasses
    - Supports variations through subclassing
    - Ensures correct sequence of operations
    
    ---
    
    ## **6. Drawbacks**
    
    - Requires inheritance (less flexible than composition-based patterns)
    - Subclass explosion possible if many variations exist
    - Tight coupling between superclass and subclass
    
    ---
    
    ## **7. Real-world Examples in Java**
    
    ### **A. Java Collections Framework**
    
    `AbstractList`, `AbstractSet`, `AbstractMap`
    
    Provide template methods:
    
    - `add()`
    - `remove()`
    - Iteration logic
    
    Subclasses override necessary behaviors.
    
    ### **B. Java Servlet API**
    
    `HttpServlet` uses Template Method Pattern:
    
    - `service()` is the template method
    - Calls `doGet()`, `doPost()`, etc. (subclasses override)
    
    ### **C. `InputStream` / `OutputStream`**
    
    Framework defines template read/write method sequence.
    
    ### **D. Spring Framework**
    
    `JdbcTemplate` uses template-style processing:
    
    - Fixed structure for DB operations
    - Allows customizable callbacks
    - Provides hooks like `RowMapper`, `PreparedStatementSetter`
    
    ---
    
    ## **8. Template Method vs Strategy Pattern**
    
    | Aspect | Template Method | Strategy |
    | --- | --- | --- |
    | Based on | Inheritance | Composition |
    | Customization | Subclasses override methods | Inject different strategy objects |
    | Algorithm structure | Fixed in superclass | Defined entirely by strategies |
    | Flexibility | Less flexible | More flexible |
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Template Method | A cooking recipe with fixed steps |
    | Subclass customizations | Changing spice mixture or garnish |
    | Client | Person preparing the dish |
    
    The recipe controls the overall flow; individual steps can vary.
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Template method often declared `final` to ensure inlining.
    - Subclass overriding resolved through dynamic dispatch.
    - Hook methods with empty bodies incur minimal overhead.
    
    ---
    
    ## **11. Important Points**
    
    - Defines fixed algorithm steps in a superclass.
    - Subclasses override specific steps without changing structure.
    - Promotes reuse and consistency.
    - Frequently used in frameworks and abstract class hierarchies.
    - Works well when algorithm workflow must remain controlled.
    
    ---
    
    ## **12. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Define algorithm framework while allowing step variation |
    | **Key Idea** | Fixed template + customizable steps |
    | **Used In** | Java I/O, Servlets, Collections, Spring templates |
    | **Strength** | Encourages reuse and uniform algorithms |
    
    The Template Method Pattern provides a stable algorithm structure while offering subclass-level customization, making it ideal for consistent workflows in frameworks and complex systems.
    
- Observer Pattern
    
    # **Observer Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Observer Pattern** is a behavioral design pattern where an object (called the **Subject** or **Observable**) maintains a list of dependents (called **Observers**) and **automatically notifies** them of state changes.
    
    ### **Purpose**
    
    - Establish a **one-to-many** relationship
    - Ensure observers are updated **automatically**
    - Support event-driven systems
    - Reduce tight coupling between subject and observers
    
    ---
    
    ## **2. When to Use Observer Pattern**
    
    - When multiple objects must react to changes in another object
    - When you want to avoid manually updating dependent objects
    - When designing event-driven or reactive systems
    - When a state change must trigger associated actions
    
    Common use cases:
    
    - GUI frameworks (button click listeners)
    - Messaging/event systems
    - Stock price update notifications
    - Publisher–Subscriber architectures
    - MVC architecture (View observes Model)
    
    ---
    
    ## **3. Structure**
    
    ### **Subject Interface**
    
    ```java
    interface Subject {
        void attach(Observer o);
        void detach(Observer o);
        void notifyObservers();
    }
    
    ```
    
    ### **Observer Interface**
    
    ```java
    interface Observer {
        void update(String message);
    }
    
    ```
    
    ### **Concrete Subject**
    
    ```java
    class NewsAgency implements Subject {
        private List<Observer> observers = new ArrayList<>();
        private String news;
    
        public void setNews(String news) {
            this.news = news;
            notifyObservers();
        }
    
        public void attach(Observer o) { observers.add(o); }
        public void detach(Observer o) { observers.remove(o); }
    
        public void notifyObservers() {
            for (Observer o : observers) {
                o.update(news);
            }
        }
    }
    
    ```
    
    ### **Concrete Observer**
    
    ```java
    class NewsChannel implements Observer {
        private String channelNews;
    
        public void update(String news) {
            this.channelNews = news;
            System.out.println("Channel received: " + news);
        }
    }
    
    ```
    
    ### **Client Usage**
    
    ```java
    NewsAgency agency = new NewsAgency();
    NewsChannel c1 = new NewsChannel();
    NewsChannel c2 = new NewsChannel();
    
    agency.attach(c1);
    agency.attach(c2);
    
    agency.setNews("Breaking: Java 22 Released!");
    
    ```
    
    ---
    
    ## **4. Key Characteristics**
    
    - Establishes event propagation automatically
    - Supports dynamic subscription
    - Decouples subject from observers (low coupling)
    - Observers react independently
    
    ---
    
    ## **5. Push vs Pull Model**
    
    ### **Push Model**
    
    Subject sends full data to observers:
    
    ```java
    observer.update(fullData);
    
    ```
    
    ### **Pull Model**
    
    Observer fetches the data it needs:
    
    ```java
    public void update() {
        String data = subject.getData();
    }
    
    ```
    
    Pull model is preferred for flexibility and efficiency.
    
    ---
    
    ## **6. Java Implementations**
    
    ### **A. Java’s Legacy Observer API**
    
    - `java.util.Observable` (Subject)
    - `java.util.Observer` (Observer)
        
        **Deprecated** due to design issues.
        
    
    ### **Issues**
    
    - Observable was a class (not interface) — prevented flexibility
    - Pull/push tightly coupled
    - Not thread-safe
    
    ---
    
    ### **B. Modern Java Alternatives**
    
    Observer pattern is implemented using:
    
    ### **PropertyChangeListener**
    
    ```java
    PropertyChangeSupport support = new PropertyChangeSupport(this);
    support.addPropertyChangeListener(listener);
    support.firePropertyChange("prop", oldVal, newVal);
    
    ```
    
    ### **Flow API (Java 9+) – Reactive Streams**
    
    Publisher–Subscriber model:
    
    ```java
    Flow.Publisher
    Flow.Subscriber
    Flow.Subscription
    
    ```
    
    This is essentially an asynchronous observer pattern.
    
    ---
    
    ## **7. Observer Pattern in Spring**
    
    Spring heavily relies on Observer Pattern:
    
    ### **A. ApplicationEvent & ApplicationListener**
    
    ```java
    @Component
    class EmailListener implements ApplicationListener<OrderPlacedEvent> {
        public void onApplicationEvent(OrderPlacedEvent event) {
            System.out.println("Sending email for order: " + event.getId());
        }
    }
    
    ```
    
    Publish:
    
    ```java
    applicationEventPublisher.publishEvent(new OrderPlacedEvent(orderId));
    
    ```
    
    ### **B. Spring Cloud / Messaging**
    
    - KafkaListener
    - RabbitListener
    - EventBus systems
    
    ### **C. Spring Security**
    
    Authentication events follow observer style.
    
    ### **D. ContextRefreshedEvent, ContextClosedEvent**
    
    Spring lifecycle events.
    
    ---
    
    ## **8. Observer Pattern vs Pub–Sub Pattern**
    
    | Aspect | Observer | Pub–Sub |
    | --- | --- | --- |
    | Coupling | Subject knows observers | Publisher doesn’t know subscribers |
    | Communication | Direct | Broker mediated |
    | Use cases | In-process events | Distributed messaging |
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Observer Concept | Analogy |
    | --- | --- |
    | Subject | You posting an update |
    | Observers | Followers who subscribed |
    | Notification | Followers receive the update automatically |
    | Detach | Unfollow |
    
    ---
    
    ## **10. JVM / Compiler Notes**
    
    - Observer notifications rely on dynamic dispatch (virtual calls).
    - Frequent observer updates are cheap but large observer lists may incur overhead.
    - Thread safety is the developer’s responsibility (synchronization, concurrent collections).
    - Asynchronous observers in Java (CompletableFuture, Flow API) use fork-join pools internally.
    
    ---
    
    ## **11. Important Points**
    
    - Defines a 1-to-many dependency automatically.
    - Observers are decoupled from the subject.
    - Observer list is dynamic (add/remove anytime).
    - Push and pull models offer flexibility.
    - Used extensively in GUIs, Spring events, messaging systems.
    - Subject notifies observers without knowing full implementation details.
    
    ---
    
    ## **12. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Notify multiple objects automatically when state changes |
    | **Key Idea** | One-to-many dependency using event propagation |
    | **Used In** | GUI listeners, Spring events, reactive streams |
    | **Strength** | Decoupling + automatic updates |
    
    **In summary:**
    
    The Observer Pattern provides a clean, event-driven mechanism for communication between objects, enabling automatic notifications and low coupling. It is foundational in GUI frameworks, Spring’s event system, and reactive programming models.
    
- Command Pattern
    
    # **Command Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Command Pattern** is a behavioral design pattern that encapsulates a **request as an object**, allowing the request to be:
    
    - Executed
    - Queued
    - Logged
    - Rolled back (undoable operations)
    
    It separates the **request invoker** from the **logic that performs the request**, promoting loose coupling and flexible command execution.
    
    ### **Purpose**
    
    - Parameterize objects with actions
    - Queue, schedule, or log operations
    - Support undo/redo functionality
    - Decouple invoker from receiver
    - Allow dynamic composition of commands
    
    ---
    
    ## **2. When to Use the Command Pattern**
    
    - When you need to support undo/redo features
    - When multiple operations must be queued or batched
    - When requests should be handled by different objects at runtime
    - When you want to decouple user actions from the execution logic
    - When macros or command sequences are required
    
    Common use cases:
    
    - GUI actions (menu items, buttons)
    - Transactional operations
    - Remote commands (RPC)
    - Task scheduling systems
    - Logging-based replay of operations
    
    ---
    
    ## **3. Structure**
    
    ### **Command Interface**
    
    ```java
    interface Command {
        void execute();
    }
    
    ```
    
    ### **Receiver**
    
    The actual business logic.
    
    ```java
    class Light {
        void turnOn() {
            System.out.println("Light ON");
        }
    
        void turnOff() {
            System.out.println("Light OFF");
        }
    }
    
    ```
    
    ### **Concrete Commands**
    
    Each encapsulates a request.
    
    ```java
    class TurnOnCommand implements Command {
        private Light light;
    
        TurnOnCommand(Light light) {
            this.light = light;
        }
    
        public void execute() {
            light.turnOn();
        }
    }
    
    class TurnOffCommand implements Command {
        private Light light;
    
        TurnOffCommand(Light light) {
            this.light = light;
        }
    
        public void execute() {
            light.turnOff();
        }
    }
    
    ```
    
    ### **Invoker**
    
    Triggers the command.
    
    ```java
    class RemoteControl {
        private Command command;
    
        void setCommand(Command command) {
            this.command = command;
        }
    
        void pressButton() {
            command.execute();
        }
    }
    
    ```
    
    ### **Client Usage**
    
    ```java
    Light light = new Light();
    Command on = new TurnOnCommand(light);
    Command off = new TurnOffCommand(light);
    
    RemoteControl remote = new RemoteControl();
    remote.setCommand(on);
    remote.pressButton();
    remote.setCommand(off);
    remote.pressButton();
    
    ```
    
    ---
    
    ## **4. Key Characteristics**
    
    - Request becomes an object
    - Allows storing command history
    - Enforces separation between invoker and receiver
    - Commands are reusable and testable
    - Easy to extend with new commands
    
    ---
    
    ## **5. Advanced Concepts**
    
    ### **A. Undo/Redo Support**
    
    Add an `undo()` method.
    
    ```java
    interface Command {
        void execute();
        void undo();
    }
    
    ```
    
    Concrete command:
    
    ```java
    class IncreaseVolumeCommand implements Command {
        private Speaker speaker;
    
        public void execute() { speaker.increase(); }
        public void undo() { speaker.decrease(); }
    }
    
    ```
    
    ### **B. Command Queueing**
    
    Commands can be queued for later execution:
    
    ```java
    Queue<Command> commandQueue = new LinkedList<>();
    commandQueue.add(new TurnOnCommand(light));
    
    ```
    
    ### **C. Macro Commands (Composite Command)**
    
    A command composed of other commands.
    
    ```java
    class MacroCommand implements Command {
        private List<Command> commands;
    
        public void execute() {
            for (Command c : commands) c.execute();
        }
    }
    
    ```
    
    ---
    
    ## **6. Command Pattern in Java**
    
    ### **A. Runnable and Callable**
    
    Java’s `Runnable` is essentially a command:
    
    ```java
    Runnable task = () -> System.out.println("Task executed");
    executor.submit(task);
    
    ```
    
    ### **B. Swing Action**
    
    `ActionListener` objects act as commands executed on events.
    
    ### **C. JavaFX EventHandlers**
    
    ```java
    button.setOnAction(event -> doSomething());
    
    ```
    
    ### **D. Spring**
    
    - `JdbcTemplate`, `RestTemplate` use callback objects (command-like actions)
    - `@Scheduled` tasks encapsulate commands
    - Messaging frameworks treat messages as commands
    
    ---
    
    ## **7. Real-world Analogy**
    
    | Command Element | Analogy |
    | --- | --- |
    | Command | Order slip with work instructions |
    | Receiver | Chef executing the order |
    | Invoker | Waiter passing the order to kitchen |
    | Client | Customer placing the request |
    
    The waiter doesn’t cook; the command contains everything needed.
    
    ---
    
    ## **8. Command Pattern vs Strategy vs Chain of Responsibility**
    
    | Pattern | Purpose |
    | --- | --- |
    | Command | Encapsulate request as object |
    | Strategy | Replace one algorithm with another |
    | Chain of Responsibility | Pass request along handlers |
    
    ---
    
    ## **9. JVM / Compiler Notes**
    
    - Commands are small objects; JVM optimizes them well.
    - Lambda-based commands lead to very efficient bytecode.
    - Commands stored in collections are handled through interfaces enabling polymorphic behavior.
    
    ---
    
    ## **10. Important Points**
    
    - Encapsulates a request as an object.
    - Supports undo, redo, logging, and queuing.
    - Decouples invoker from receiver logic.
    - Can create macro commands for batch operations.
    - Very common in GUI frameworks, job schedulers, and asynchronous systems.
    
    ---
    
    ## **11. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Wrap a request in an object |
    | **Key Idea** | Decouple sender from executor |
    | **Used In** | GUIs, tasks, undo systems, schedulers |
    | **Strength** | Supports extensibility, undo, macros |
    
    The Command Pattern provides a flexible way to encapsulate actions, enabling powerful features such as undo/redo, task queues, and decoupled architectures widely used in modern Java applications.
    
- Chain of Responsibility
    
    # **Chain of Responsibility Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Chain of Responsibility (CoR)** Pattern is a behavioral design pattern in which a **request is passed along a chain of handlers**.
    
    Each handler decides either to:
    
    - **Process the request**, or
    - **Pass it to the next handler** in the chain
    
    This eliminates the need for the sender to know which handler will ultimately process the request.
    
    ### **Purpose**
    
    - Decouple the request sender from its receiver
    - Allow multiple handlers to process or reject requests
    - Support flexible, dynamic reconfiguration of processing chains
    - Promote loose coupling and single responsibility
    
    ---
    
    ## **2. When to Use Chain of Responsibility**
    
    Use CoR when:
    
    - Multiple handlers can process a request
    - Handler order may change at runtime
    - You want to avoid complex if-else/switch chains
    - A request must go through a sequence of filters/validation layers
    
    Common use cases:
    
    - Logging pipelines
    - Event processing
    - HTTP filters
    - Validation chains
    - ATM withdrawal processing
    - Middleware pipelines (web frameworks)
    
    ---
    
    ## **3. Structure**
    
    ### **Handler (Interface or Abstract Class)**
    
    ```java
    interface Handler {
        void setNext(Handler next);
        void handle(Request req);
    }
    
    ```
    
    ### **Concrete Handler**
    
    ```java
    class AuthHandler implements Handler {
        private Handler next;
    
        public void setNext(Handler next) { this.next = next; }
    
        public void handle(Request req) {
            if (!req.isAuthenticated()) {
                System.out.println("Authentication failed.");
                return;
            }
            System.out.println("Authentication successful.");
            if (next != null) next.handle(req);
        }
    }
    
    ```
    
    ### **Another Handler**
    
    ```java
    class AuthorizationHandler implements Handler {
        private Handler next;
    
        public void setNext(Handler next) { this.next = next; }
    
        public void handle(Request req) {
            if (!req.hasRole("ADMIN")) {
                System.out.println("Access denied.");
                return;
            }
            System.out.println("Authorization successful.");
            if (next != null) next.handle(req);
        }
    }
    
    ```
    
    ### **Client Code (Building Chain)**
    
    ```java
    Handler auth = new AuthHandler();
    Handler authorization = new AuthorizationHandler();
    
    auth.setNext(authorization);
    
    auth.handle(new Request("Ayush", true, "ADMIN"));
    
    ```
    
    ---
    
    ## **4. Key Characteristics**
    
    - Handlers are linked dynamically
    - Request may be fully processed, partially processed, or ignored
    - Each handler has **one job** (single responsibility)
    - Chain order is highly flexible
    - Client calls the chain’s head and does not know internal flow
    
    ---
    
    ## **5. Types of Chains**
    
    ### **A. Pure CoR**
    
    A handler must pass the request or stop; only one might handle it.
    
    ### **B. Impure CoR**
    
    Multiple handlers in the chain may process the request.
    
    ---
    
    ## **6. Advanced Concepts**
    
    ### **A. Breaking the Chain**
    
    A handler may stop the chain:
    
    ```java
    if (conditionFails) return;
    
    ```
    
    ### **B. Asynchronous Chains**
    
    Useful for reactive processing pipelines (CompletableFuture, Reactor).
    
    ### **C. Chain Configuration via Builders**
    
    Construct chains through configuration files or Spring Beans.
    
    ---
    
    ## **7. Chain of Responsibility in Java**
    
    ### **A. Servlet Filters (Most Common Example)**
    
    `FilterChain` in Java EE / Spring Boot:
    
    ```java
    doFilter(request, response, filterChain);
    
    ```
    
    Every filter:
    
    - Processes the request
    - Passes to next filter
    
    ### **B. Logging Frameworks**
    
    Log4j, JUL, and Logback use chained appenders.
    
    ### **C. java.util.logging.Handler**
    
    This tree-like structure passes log messages to handlers.
    
    ---
    
    ## **8. Chain of Responsibility in Spring**
    
    ### **A. Spring Security Filter Chain**
    
    Every HTTP request flows through filters:
    
    - `UsernamePasswordAuthenticationFilter`
    - `BasicAuthenticationFilter`
    - `AuthorizationFilter`
    - ...
    
    All are linked in a chain (`FilterChainProxy`).
    
    ### **B. Spring Interceptors**
    
    HandlerInterceptor chain in Spring MVC.
    
    ### **C. BeanPostProcessor Chain**
    
    Each BeanPostProcessor has a chance to modify a bean before and after initialization.
    
    ### **D. HandlerExecutionChain**
    
    Used in request mapping and handling.
    
    ---
    
    ## **9. Real-world Analogy**
    
    | Component | Analogy |
    | --- | --- |
    | Handler | Employee in a support team |
    | Chain | Escalation hierarchy |
    | Request | Customer complaint |
    | Flow | Every employee decides: resolve or escalate |
    
    ---
    
    ## **10. Chain vs Other Patterns**
    
    | Pattern | Difference |
    | --- | --- |
    | **Strategy** | Client selects strategy; CoR selects handler dynamically |
    | **Decorator** | Adds behavior; CoR passes request along handlers |
    | **Observer** | Broadcasts to all; CoR passes sequentially |
    | **Mediator** | Central controller; CoR is decentralized |
    
    ---
    
    ## **11. JVM / Performance Notes**
    
    - Each handler call is a virtual method call → fast
    - Long chains may increase latency
    - Precomputing/optimizing chain order improves performance
    - Spring uses cached, prebuilt chains for efficiency
    
    ---
    
    ## **12. Important Points**
    
    - Removes conditional complexity
    - Highly flexible chain order and composition
    - Handlers operate independently
    - Great for request filtering, validation, logging
    - Forms backbone of many Java and Spring frameworks
    
    ---
    
    ## **13. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Pass request through a chain of handlers |
    | **Key Idea** | Each handler processes or passes request |
    | **Used In** | Filters, validation, logging, Spring security |
    | **Benefit** | Decoupled, flexible, scalable request handling |
    
    The **Chain of Responsibility Pattern** creates clean, modular request pipelines that are widely used in enterprise systems, especially in web frameworks and processing engines.
    

### **5.5 Patterns Used in Spring**

- Dependency Injection Pattern
    
    # **Dependency Injection (DI) Pattern**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    **Dependency Injection (DI)** is a design pattern where an object’s dependencies are **provided (injected)** to it from the outside rather than the object creating them internally.
    
    The object does **not** control the creation of its dependencies; instead, a **caller, framework, or container** supplies them.
    
    ### **Purpose**
    
    - Reduce tight coupling between classes
    - Improve testability, maintainability, and modularity
    - Follow the **Inversion of Control (IoC)** principle
    - Promote reusable and swappable components
    
    DI is a key principle behind modern frameworks like **Spring**, **Jakarta EE**, and **Micronaut**.
    
    ---
    
    ## **2. Why Dependency Injection?**
    
    Without DI:
    
    ```java
    class Service {
        private Repository repo = new Repository(); // tightly coupled
    }
    
    ```
    
    Problems:
    
    - Hard to test
    - Hard to replace or mock dependencies
    - Violates SOLID principles (especially DIP)
    
    With DI, the dependency is provided externally:
    
    ```java
    class Service {
        private Repository repo;
    
        Service(Repository repo) {
            this.repo = repo;
        }
    }
    
    ```
    
    ---
    
    ## **3. Types of Dependency Injection**
    
    ### **A. Constructor Injection (Preferred)**
    
    Dependencies are injected via the constructor.
    
    ```java
    class Service {
        private final Repository repo;
    
        public Service(Repository repo) {
            this.repo = repo;
        }
    }
    
    ```
    
    **Benefits**
    
    - Promotes immutability
    - Ensures dependency availability
    - Enables clean testing
    
    ---
    
    ### **B. Setter Injection**
    
    Dependencies provided via setter methods.
    
    ```java
    class Service {
        private Repository repo;
    
        public void setRepository(Repository repo) {
            this.repo = repo;
        }
    }
    
    ```
    
    **Use When**
    
    - Optional dependencies
    - Reconfigurable objects
    
    ---
    
    ### **C. Field Injection (Not recommended except in frameworks)**
    
    Injected directly into fields.
    
    ```java
    class Service {
        @Inject
        private Repository repo;
    }
    
    ```
    
    **Limitations**
    
    - Hard to test
    - Hides dependencies
    - Breaks immutability
    
    ---
    
    ## **4. Dependency Injection Example (Plain Java)**
    
    ### **Interfaces**
    
    ```java
    interface MessageService {
        void send(String msg);
    }
    
    class EmailService implements MessageService {
        public void send(String msg) {
            System.out.println("Email: " + msg);
        }
    }
    
    ```
    
    ### **Client with DI**
    
    ```java
    class NotificationController {
        private final MessageService service;
    
        public NotificationController(MessageService service) {
            this.service = service;
        }
    
        public void notify(String msg) {
            service.send(msg);
        }
    }
    
    ```
    
    ### **Injecting Dependency**
    
    ```java
    MessageService email = new EmailService();
    NotificationController controller = new NotificationController(email);
    controller.notify("Hello");
    
    ```
    
    ---
    
    ## **5. Dependency Injection in Spring**
    
    Spring is **built** on the DI principle.
    
    ### **A. Constructor Injection**
    
    ```java
    @Service
    public class OrderService {
        private final PaymentService ps;
    
        public OrderService(PaymentService ps) {
            this.ps = ps;
        }
    }
    
    ```
    
    ### **B. Autowiring**
    
    Spring automatically detects and injects beans:
    
    ```java
    @Component
    class EmailService { }
    
    @Component
    class NotificationService {
        @Autowired
        private EmailService emailService;
    }
    
    ```
    
    ### **C. Injection Scopes**
    
    - Singleton
    - Prototype
    - Request
    - Session
    
    Spring IoC container manages lifecycle + injection.
    
    ---
    
    ## **6. DI and Inversion of Control (IoC)**
    
    DI is one implementation of IoC.
    
    - **DI**: Framework injects dependencies into objects.
    - **IoC**: Framework controls object creation instead of the application.
    
    ### **Flow**
    
    ```
    Your code asks Spring for dependencies → ❌
    Spring gives dependencies to your code → ✔️
    
    ```
    
    ---
    
    ## **7. Benefits of DI**
    
    - **Loose coupling** (program to interfaces)
    - **Easy unit testing** (mock dependencies)
    - **Better maintainability**
    - **Improved flexibility** (swap implementations easily)
    - **Encourages clean architecture (SOLID principles)**
    
    ---
    
    ## **8. Drawbacks**
    
    - Slightly more initial setup
    - Indirection may confuse beginners
    - Requires a DI container (like Spring) for large applications
    - Overuse can lead to too many small classes
    
    ---
    
    ## **9. Use Cases**
    
    - Enterprise applications
    - Microservices
    - Modular systems
    - Plugin architectures
    - Any code requiring replacement/test of components
    
    ---
    
    ## **10. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Dependency | Electricity |
    | Object | Laptop |
    | DI Container | Power socket |
    | DI Process | Plugging laptop into socket |
    
    Laptop **doesn’t generate electricity**; it receives it.
    
    Similarly, objects shouldn’t instantiate their own dependencies—DI provides them.
    
    ---
    
    ## **11. Comparison With Other Patterns**
    
    ### **DI vs Service Locator**
    
    | Aspect | DI | Service Locator |
    | --- | --- | --- |
    | Visibility | Dependencies explicit | Hidden (bad for clarity) |
    | Coupling | Low | Higher |
    | Testing | Easy | Hard |
    | Recommended | ✔️ | ❌ (mostly discouraged) |
    
    ### **DI vs Factory Pattern**
    
    Factory creates objects → DI injects already-created objects.
    
    ---
    
    ## **12. JVM / Performance Notes**
    
    - DI adds minimal overhead after initialization
    - Spring caches bean instances → fast access
    - Constructor injection is compiler-friendly due to immutability
    - Reflection-based injection in Spring is optimized via caching
    
    ---
    
    ## **13. Important Points**
    
    - DI separates **creation** from **use**.
    - Constructor injection is the **best practice**.
    - Essential for testability using mocks.
    - Spring is built entirely around DI/IoC.
    - Promotes clean architecture and SOLID compliance.
    
    ---
    
    ## **14. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Inject dependencies instead of creating them |
    | **Key Idea** | Separate object creation from object usage |
    | **Benefits** | Loose coupling, testability, flexibility |
    | **Used In** | Spring, Jakarta EE, microservices |
    | **Pattern Type** | Architectural pattern (IoC) |
    
    **Dependency Injection** is one of the most important patterns in modern Java development, forming the foundation of frameworks like Spring and enabling scalable, flexible, and testable applications.
    
- MVC Pattern
    
    # **MVC Pattern (Model–View–Controller)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Model–View–Controller (MVC)** Pattern is an architectural pattern that divides an application into **three interconnected components**:
    
    1. **Model** — Represents data, business logic, and state
    2. **View** — Represents UI or presentation
    3. **Controller** — Handles user input and coordinates between Model and View
    
    Its primary purpose is to **separate concerns**, improving maintainability, testability, and scalability.
    
    ---
    
    ## **2. Purpose of MVC**
    
    - Decouple UI from business logic
    - Allow independent development of Model, View, and Controller
    - Support multiple views of the same data
    - Make the application easier to test
    - Encourage clean separation between concerns
    
    ---
    
    ## **3. Components in Detail**
    
    ---
    
    ## **A. Model**
    
    ### **Role**
    
    - Encapsulates application **data**, **state**, and **business logic**
    - Not concerned with UI or user interactions
    - Notifies View(s) when data changes (Observer-like behavior)
    
    ### **Responsibilities**
    
    - Maintain data
    - Perform validation and business rules
    - Trigger events/notifications to observers
    
    ### **Example**
    
    ```java
    class User {
        private String name;
        private int age;
    
        // getters, setters, validation logic
    }
    
    ```
    
    ---
    
    ## **B. View**
    
    ### **Role**
    
    - Responsible for **presentation** (UI rendering)
    - Displays data from Model
    - Receives minimal user input (delegated to Controller)
    
    ### **Responsibilities**
    
    - Render output (HTML, GUI components, JSON, etc.)
    - Subscribe to Model changes (optional)
    
    ### **Example**
    
    ```java
    public class UserView {
        public void showUser(String name, int age) {
            System.out.println("User: " + name + ", Age: " + age);
        }
    }
    
    ```
    
    ---
    
    ## **C. Controller**
    
    ### **Role**
    
    - Handles **user input**
    - Interprets commands (clicks, requests, keystrokes)
    - Updates Model or selects View
    
    ### **Responsibilities**
    
    - Communicate with Model
    - Select which View to render
    - Process incoming requests
    
    ### **Example**
    
    ```java
    class UserController {
        private User model;
        private UserView view;
    
        public UserController(User model, UserView view) {
            this.model = model;
            this.view = view;
        }
    
        public void setUserName(String name) { model.setName(name); }
        public void setUserAge(int age) { model.setAge(age); }
    
        public void updateView() {
            view.showUser(model.getName(), model.getAge());
        }
    }
    
    ```
    
    ---
    
    ## **4. MVC Workflow**
    
    ### **Typical Flow**
    
    ```
    User → Controller → Model → View → User
    
    ```
    
    Detailed:
    
    1. User interacts with View (button click/input)
    2. Controller receives the event
    3. Controller updates Model
    4. Model changes state
    5. View retrieves updated model data and re-renders
    6. User sees updated output
    
    ---
    
    ## **5. MVC in Java (Swing / JavaFX)**
    
    ### **Swing Example**
    
    - **Model**: TableModel
    - **View**: JTable
    - **Controller**: TableModelListener or event handlers
    
    Swing uses Observer behavior to update Views when the Model changes.
    
    ---
    
    ## **6. MVC in Web Applications**
    
    ### **Spring MVC Architecture**
    
    | MVC Component | Spring Implementation |
    | --- | --- |
    | **Model** | Java objects, Entities, DTOs |
    | **View** | JSP, Thymeleaf, JSON responses |
    | **Controller** | `@Controller`, `@RestController` |
    
    ### **Spring MVC Flow**
    
    ```
    Client Request
           ↓
    DispatcherServlet (Front Controller)
           ↓
    Handler Mapping
           ↓
    Controller
           ↓
    Model
           ↓
    View Resolver
           ↓
    View (UI or JSON)
    
    ```
    
    This uses **Front Controller** + **MVC** + **Strategy** internally.
    
    ---
    
    ## **7. MVC vs MVVM vs MVP**
    
    | Pattern | Description |
    | --- | --- |
    | **MVC** | View delegates user actions to Controller; Model notifies View |
    | **MVP** | Presenter updates View explicitly; View is passive |
    | **MVVM** | ViewModel provides observable data bindings to View |
    
    JavaFX uses **MVVM-style** with bindings.
    
    ---
    
    ## **8. Benefits of MVC**
    
    - Clear separation of concerns
    - Easy to maintain and scale
    - Reusability of Model and Controller
    - Multiple views supported
    - Encourages clean architecture
    - Good for large applications
    
    ---
    
    ## **9. Drawbacks**
    
    - More classes → more structure and complexity
    - Requires discipline in team development
    - Controller can become too large if misused (God object)
    - Improper design may lead to tightly coupled components
    
    ---
    
    ## **10. Real-world Analogy**
    
    | Layer | Analogy |
    | --- | --- |
    | **Model** | Data stored in a warehouse |
    | **View** | Display shelf showing products |
    | **Controller** | Worker fetching items based on user requests |
    
    Users never directly interact with the warehouse—everything goes through the worker (controller).
    
    ---
    
    ## **11. JVM / Architectural Notes**
    
    - MVC promotes separation → optimized testing and modularization
    - MVC layers can run on separate threads (e.g., UI thread + background thread)
    - Spring MVC uses DI + proxies + handler mappings internally
    
    ---
    
    ## **12. Important Points**
    
    - MVC is an architectural (not design) pattern
    - Enforces separation between data, presentation, and control logic
    - Controller mediates between Model and View
    - Widely used in both desktop and web applications
    - Foundation for modern frameworks like Spring MVC, Django, Ruby on Rails
    
    ---
    
    ## **13. Summary**
    
    | Component | Responsibility |
    | --- | --- |
    | **Model** | Handles data + business logic |
    | **View** | Renders UI |
    | **Controller** | Handles input and coordinates flow |
    
    **In summary:**
    
    The MVC Pattern cleanly separates application logic into Model, View, and Controller, making applications more maintainable, testable, and scalable—especially foundational in frameworks like Spring MVC.
    
- Template Pattern (JdbcTemplate)
    
    # **Template Pattern (JdbcTemplate)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Template Pattern** in Spring's `JdbcTemplate` provides a **predefined workflow** for executing database operations while allowing developers to supply only the variable parts (SQL, parameters, mappings).
    
    The fixed algorithm steps—connection handling, statement creation, execution, exception translation, and resource cleanup—are defined by the template, while custom logic is inserted via callbacks.
    
    ### **Purpose**
    
    - Encapsulate boilerplate JDBC code
    - Ensure consistent resource management
    - Provide a reusable and robust algorithm for DB operations
    - Allow developers to focus on business logic, not JDBC mechanics
    - Prevent connection leaks and reduce errors
    
    ---
    
    ## **2. JDBC Without Template Pattern (Problem)**
    
    Traditional JDBC requires:
    
    - Loading drivers
    - Getting a connection
    - Creating statements
    - Executing queries
    - Processing `ResultSet`
    - Handling SQL exceptions
    - Closing `ResultSet`, `Statement`, and `Connection`
    
    This leads to:
    
    - Repetitive boilerplate
    - Frequent resource leaks
    - Hard-to-maintain code
    - Error-prone exception handling
    
    ---
    
    ## **3. Template Pattern Solution — JdbcTemplate**
    
    `JdbcTemplate` defines a **fixed algorithm** for JDBC operations:
    
    ```
    1. Acquire connection
    2. Prepare statement
    3. Execute SQL
    4. Process results
    5. Handle exceptions
    6. Close resources automatically
    
    ```
    
    Developers provide only:
    
    - SQL query
    - Query parameters
    - Row-mapping logic
    
    Everything else is handled by the template.
    
    ---
    
    ## **4. Structure**
    
    ### **Fixed Steps (Handled by JdbcTemplate)**
    
    - Obtaining a connection from DataSource
    - Preparing statement
    - Executing queries
    - Managing transactions (when integrated with Spring TX)
    - Converting SQLExceptions into DataAccessException
    - Ensuring proper resource cleanup
    
    ### **Variable Steps (Provided by Developer)**
    
    - SQL string
    - Input parameters
    - Row mapping logic (`RowMapper`)
    - Callback functions (`PreparedStatementSetter`, `ResultSetExtractor`)
    
    ---
    
    ## **5. Common Operations Using JdbcTemplate**
    
    ### **1. Query for Single Object**
    
    ```java
    String sql = "SELECT name FROM users WHERE id=?";
    String name = jdbcTemplate.queryForObject(sql, String.class, 1);
    
    ```
    
    ### **2. Query for List**
    
    ```java
    List<String> names = jdbcTemplate.queryForList("SELECT name FROM users", String.class);
    
    ```
    
    ### **3. Query Using RowMapper**
    
    ```java
    String sql = "SELECT id, name FROM users";
    
    List<User> users = jdbcTemplate.query(sql, (rs, rowNum) ->
        new User(rs.getInt("id"), rs.getString("name"))
    );
    
    ```
    
    ### **4. Insert/Update/Delete**
    
    ```java
    String sql = "UPDATE users SET name=? WHERE id=?";
    jdbcTemplate.update(sql, "Ayush", 1);
    
    ```
    
    ### **5. Batch Update**
    
    ```java
    jdbcTemplate.batchUpdate(
        "INSERT INTO users (name) VALUES (?)",
        Arrays.asList("A", "B", "C"),
        10,
        (ps, name) -> ps.setString(1, name)
    );
    
    ```
    
    ---
    
    ## **6. Internal Algorithm (Template Method Pattern)**
    
    ### **Fixed by Template:**
    
    ```
    doExecute() {
        Connection con = getConnection();
        PreparedStatement ps = con.prepareStatement();
        applyParameters(ps);
        executeStatement(ps);
        handleExceptions();
        closeResources();
    }
    
    ```
    
    ### **Hooks for User:**
    
    - Provide SQL query
    - Provide mapping logic via callbacks
    
    The template enforces a strict order, ensuring robustness and consistency.
    
    ---
    
    ## **7. Callback Interfaces in JdbcTemplate**
    
    ### **A. RowMapper**
    
    Maps each row of ResultSet to an object:
    
    ```java
    class UserMapper implements RowMapper<User> {
        public User mapRow(ResultSet rs, int rowNum) {
            return new User(rs.getInt("id"), rs.getString("name"));
        }
    }
    
    ```
    
    ### **B. ResultSetExtractor**
    
    Extract entire result set at once:
    
    ```java
    User extract(ResultSet rs);
    
    ```
    
    ### **C. PreparedStatementSetter**
    
    Allows custom parameter binding.
    
    ---
    
    ## **8. Exception Handling in Template Pattern**
    
    JdbcTemplate uses **Spring’s DataAccessException hierarchy**, providing:
    
    - Consistent exceptions across different databases
    - No need to catch `SQLException` manually
    
    Example translation:
    
    - `SQLException` → `DataIntegrityViolationException`, etc.
    
    ---
    
    ## **9. JdbcTemplate and Transactions**
    
    When used with Spring Transaction Management:
    
    - Template automatically uses transactional connection
    - No need for manual commit/rollback
    
    Transaction boundaries are defined by:
    
    ```java
    @Transactional
    public void method() { ... }
    
    ```
    
    ---
    
    ## **10. Benefits of Template Pattern in JdbcTemplate**
    
    - Eliminates boilerplate JDBC code
    - Prevents resource leaks
    - Makes DB access cleaner and more readable
    - Provides reusable higher-level APIs
    - Improves maintainability
    - Safe exception translation
    - Encourages separation of concerns
    
    ---
    
    ## **11. Limitations**
    
    - Limited control over very low-level JDBC operations
    - Can become verbose for complex mappings (use JPA ORMs instead)
    - Heavy use of callbacks may reduce readability for beginners
    
    ---
    
    ## **12. Real-world Analogy**
    
    | Template Concept | Analogy |
    | --- | --- |
    | Template Method | Fixed recipe of baking bread |
    | Hook methods | Custom filling or toppings |
    | JdbcTemplate | Bakery machine doing all repetitious steps |
    | Developer | Person providing only ingredients |
    
    The machine always follows the same baking sequence.
    
    ---
    
    ## **13. Summary**
    
    | Aspect | Description |
    | --- | --- |
    | **Intent** | Encapsulate fixed JDBC execution workflow |
    | **Template** | Connection → Statement → Execution → Cleanup |
    | **Variable Parts** | SQL, parameters, RowMapper |
    | **Benefit** | Clean, safe, reusable JDBC access |
    | **Used In** | Spring JDBC module |
    
    **JdbcTemplate implements the Template Method Pattern** by defining a fixed database operation workflow while allowing variation through callbacks. This ensures robust, clean, and efficient data access in Spring applications.
    
- Proxy Pattern (AOP & Transactions)
    
    # **Proxy Pattern (AOP & Transactions Focused)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Proxy Pattern** provides a **surrogate object** that controls access to another object (the *target*).
    
    In Spring, AOP (Aspect-Oriented Programming) and Transaction Management rely entirely on **dynamic proxies** that intercept method calls and apply cross-cutting behaviors such as:
    
    - Transaction boundaries (`@Transactional`)
    - Security checks (`@Secured`)
    - Logging, metrics
    - Caching (`@Cacheable`)
    
    The proxy wraps the original bean and executes additional logic **before**, **after**, or **around** target method execution.
    
    ---
    
    ## **2. Why Proxy Pattern for AOP and Transactions?**
    
    ### **Reasons**
    
    - Allows adding cross-cutting logic without modifying business code
    - Keeps application classes clean
    - Centralizes concerns like logging, caching, and tx-management
    - Enables runtime weaving (no bytecode modification)
    - Supports aspect chaining (multiple advisors)
    
    ### **Problem Solved**
    
    Without proxies, each method would require boilerplate like:
    
    ```java
    connection.beginTransaction();
    try {
        doBusinessLogic();
        connection.commit();
    } catch (Exception e) {
        connection.rollback();
    }
    
    ```
    
    Spring proxies automate this logic transparently.
    
    ---
    
    ## **3. Types of Proxies Used by Spring**
    
    ### **1. JDK Dynamic Proxy (Interface-based)**
    
    - Used when the target bean implements an interface
    - Proxy class implements the same interfaces
    - Uses `java.lang.reflect.Proxy` and `InvocationHandler`
    
    ### **2. CGLIB Proxy (Class-based)**
    
    - Used when:
        - No interface is present, OR
        - `proxyTargetClass = true`
    - Creates subclass at runtime using bytecode generation
    - Cannot proxy:
        - final classes
        - final methods
    
    Spring Boot defaults to CGLIB in many cases.
    
    ---
    
    ## **4. How AOP Uses Proxy Pattern**
    
    ### **Execution Flow**
    
    ```
    Client → Proxy → Advice Chain → Target Bean
    
    ```
    
    ### **Example**
    
    Call:
    
    ```java
    paymentService.processPayment();
    
    ```
    
    Actual flow:
    
    ```
    1. Proxy intercepts method call
    2. Applies AOP advices:
          - @Before
          - @Around
          - @AfterReturning
          - @AfterThrowing
    3. Calls target method
    4. Returns output to client
    
    ```
    
    The client never interacts with the target directly.
    
    ---
    
    ## **5. How Transactions Use Proxy Pattern (`@Transactional`)**
    
    ### **Transaction Interceptor**
    
    `@Transactional` creates a transactional proxy around the bean.
    
    ### **Transaction Workflow**
    
    ```
    1. Proxy begins transaction
    2. Calls target method
    3. On success → commit
    4. On exception → rollback
    5. Clean up resources
    
    ```
    
    ### **Conceptual Flow**
    
    ```java
    @Transactional
    public void transferMoney() {
        // business logic
    }
    
    ```
    
    ### **Proxy Behavior**
    
    ```
    → Start transaction
    → Execute transferMoney()
    → Commit or rollback
    
    ```
    
    ---
    
    ## **6. Code-Level Illustration**
    
    ### **Target Bean**
    
    ```java
    @Service
    public class AccountService {
        @Transactional
        public void transfer() {
            // business logic
        }
    }
    
    ```
    
    ### **Proxy Generated Internally**
    
    ```
    class AccountService$$Proxy {
        void transfer() {
            txManager.begin();
            try {
                target.transfer();
                txManager.commit();
            } catch (Exception e) {
                txManager.rollback();
                throw e;
            }
        }
    }
    
    ```
    
    This proxy is created dynamically at runtime.
    
    ---
    
    ## **7. Self-Invocation Problem (Important)**
    
    ### **Scenario**
    
    ```java
    @Service
    public class ServiceA {
        @Transactional      // Intercepted by proxy
        public void methodA() { ... }
    
        public void internalCall() {
            methodA();      // NOT intercepted, NO transaction
        }
    }
    
    ```
    
    ### **Reason**
    
    - Proxy intercepts **external** calls
    - Internal method calls bypass proxy → AOP/transactions not applied
    
    ### **Solution**
    
    - Move method to another bean
    - Use AspectJ weaving
    - Avoid internal transactional calls
    
    ---
    
    ## **8. Proxy Chain (Multiple Advices)**
    
    Spring supports chaining:
    
    ```
    Security Proxy
    → Transaction Proxy
    → Logging Proxy
    → Target Method
    
    ```
    
    Each proxy wraps the previous one using a layered structure.
    
    ---
    
    ## **9. AOP Advice Types Implemented via Proxies**
    
    | Advice Type | Proxy Behavior |
    | --- | --- |
    | **@Before** | Runs before target method |
    | **@After** | Always runs after method |
    | **@AfterReturning** | Runs only on success |
    | **@AfterThrowing** | Runs only on exception |
    | **@Around** | Full control: before + after + exception handling |
    
    All implemented inside proxy’s invocation handler.
    
    ---
    
    ## **10. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Proxy | Security guard verifying entry |
    | Target | Actual office employee |
    | AOP Advice | Guard checking ID, logging entry |
    | Transaction Proxy | Guard ensuring rules followed before entry/exit |
    
    The employee does not change; the guard adds behavior.
    
    ---
    
    ## **11. JDK Proxy vs CGLIB Proxy**
    
    | Aspect | JDK Dynamic Proxy | CGLIB |
    | --- | --- | --- |
    | Based on | Interfaces | Subclasses |
    | Can proxy classes? | No | Yes |
    | Works with final classes? | Yes | No |
    | Works with final methods? | Yes | No |
    | Speed | Slightly slower | Faster after warm-up |
    | Default in Spring | Yes (if interfaces exist) | Yes (in many Boot cases) |
    
    ---
    
    ## **12. Important Points**
    
    - All Spring AOP and `@Transactional` features rely on **runtime proxies**
    - Proxies wrap beans and intercept method calls
    - Self-invocation bypasses proxy → AOP not applied
    - Use `@EnableAspectJAutoProxy` (or enabled via Spring Boot) for proxy creation
    - CGLIB is required for class-based proxying
    - Transaction boundaries are implemented exclusively in the proxy layer
    
    ---
    
    ## **13. Summary**
    
    | Concept | Description |
    | --- | --- |
    | **Intent** | Intercept method calls using proxy wrappers |
    | **Used For** | AOP, transactions, security, caching |
    | **Mechanism** | JDK Dynamic Proxy / CGLIB Proxy |
    | **Transactional Flow** | Start → Execute → Commit/Rollback |
    | **Benefits** | Cross-cutting concerns without modifying business code |
    
    **In summary:**
    
    The Proxy Pattern powers Spring’s AOP and Transaction Management by wrapping beans with dynamic proxies that transparently handle cross-cutting concerns such as transactions, logging, and security while keeping business logic clean and decoupled.
