## Module 14 — Hibernate / JPA ORM

### **14.1 ORM Fundamentals**

- What is ORM?
    
    # **Object-Relational Mapping (ORM)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ORM (Object-Relational Mapping) is a **programming technique** that allows Java objects to interact with relational database tables **without writing SQL directly**.
    
    ORM maps:
    
    - **Classes → Tables**
    - **Objects → Rows**
    - **Fields → Columns**
    
    This automates database persistence and retrieval, enabling developers to work with objects instead of manual SQL queries.
    
    ORM frameworks (like Hibernate, JPA, Spring Data JPA) act as a **translator** between the object-oriented world (Java) and relational databases (SQL).
    
    ---
    
    ---
    
    ## **2. Purpose of ORM**
    
    - Simplify database operations.
    - Reduce boilerplate JDBC code.
    - Allow developers to think in terms of objects, not tables.
    - Improve maintainability and productivity.
    - Provide automatic CRUD support.
    
    ---
    
    ---
    
    ## **3. How ORM Works Internally**
    
    ORM frameworks:
    
    1. Inspect entity classes and their annotations.
    2. Map those to table metadata.
    3. Auto-generate SQL for CRUD operations.
    4. Convert DB rows back into Java objects.
    5. Manage relationships (one-to-many, many-to-many).
    6. Handle caching, transactions, lazy loading, dirty checking.
    
    Example Mapping:
    
    ```java
    @Entity
    @Table(name = "employee")
    public class Employee {
    
        @Id
        private int id;
    
        private String name;
        private double salary;
    }
    
    ```
    
    ORM understands that:
    
    - Table: employee
    - Columns: id, name, salary
    
    ---
    
    ---
    
    ## **4. Benefits of ORM**
    
    ### **A. Productivity**
    
    No need to manually write SQL for every CRUD operation.
    
    ### **B. Maintainability**
    
    Less code duplication, fewer JDBC concerns.
    
    ### **C. Portability**
    
    Database switching is easier — change driver, not queries.
    
    ### **D. Caching & Performance**
    
    ORM manages caching and query optimization internally.
    
    ### **E. Object-oriented Thinking**
    
    Work with objects, relationships, inheritance.
    
    ---
    
    ---
    
    ## **5. ORM Features**
    
    - Automatic CRUD operations
    - Transaction management
    - Query generation and abstraction
    - Lazy loading of data
    - Entity relationships mapping
    - Schema generation
    - Caching
    - JPQL/HQL query language support
    
    ---
    
    ---
    
    ## **6. ORM Framework Examples**
    
    - Hibernate
    - JPA (Java Persistence API)
    - Spring Data JPA
    - EclipseLink
    - MyBatis (semi-ORM)
    
    ---
    
    ---
    
    ## **7. Real-world Analogy**
    
    ORM is like a **translator** between two languages:
    
    - Java speaks object language.
    - Databases speak SQL language.
    
    ORM converts one form into the other—seamlessly—so the programmer doesn't need to write translation code.
    
    ---
    
    ---
    
    ## **8. Important Points**
    
    - ORM hides SQL but still generates it behind the scenes.
    - It is best suited for business applications with complex object models.
    - For highly optimized queries, developers may still write SQL.
    - ORM improves development speed but requires understanding of how it translates operations to SQL.
    
    ---
    
    ---
    
    ## **9. Summary**
    
    ORM is a technique that maps Java objects to database tables so applications can interact with databases using objects rather than SQL.
    
    It simplifies persistence, improves productivity, and enables object-oriented database programming through frameworks like Hibernate and JPA.
    
- JPA vs Hibernate
    
    # **JPA vs Hibernate**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **JPA (Java Persistence API)**
    
    JPA is a **specification (interface guideline)** provided by Java for ORM (Object-Relational Mapping).
    
    It defines how Java applications should interact with databases but does **not provide implementation**.
    
    It includes:
    
    - Interfaces (`EntityManager`, `PersistenceContext`)
    - Annotations (`@Entity`, `@Id`, `@Table`, etc.)
    - Standards for entity mapping, queries, life cycle rules
    
    JPA is like a **rulebook or contract**.
    
    ---
    
    ### **Hibernate**
    
    Hibernate is a **framework (implementation)** of JPA.
    
    It provides the actual ORM engine and implements JPA interfaces along with many additional features.
    
    Hibernate = JPA implementation + extra proprietary features.
    
    ---
    
    ---
    
    # **2. Relationship Between JPA and Hibernate**
    
    - JPA defines **what** should be done.
    - Hibernate implements **how** it should be done.
    
    Hibernate is the most widely used JPA provider, but there are others like EclipseLink and OpenJPA.
    
    ---
    
    ---
    
    # **3. Comparison Table**
    
    | --- | --- | --- |
    
    ---
    
    ---
    
    # **4. JPA Example (Standard Code)**
    
    Entity:
    
    ```java
    @Entity
    public class Employee {
        @Id
        private int id;
        private String name;
    }
    
    ```
    
    Persistence call:
    
    ```java
    EntityManager em = emf.createEntityManager();
    em.persist(emp);
    
    ```
    
    This code is portable across **any JPA provider**, not tied to Hibernate.
    
    ---
    
    ---
    
    # **5. Hibernate Example (Native API Code)**
    
    Entity:
    
    ```java
    Session session = sessionFactory.openSession();
    session.save(emp);
    
    ```
    
    Hibernate-specific methods like:
    
    - `save()`
    - `update()`
    - `merge()`
    - `getCurrentSession()`
    
    These are not portable outside Hibernate.
    
    ---
    
    ---
    
    # **6. Key Concepts in JPA**
    
    - Entities
    - EntityManager
    - Persistence Context
    - JPQL (Java Persistence Query Language)
    - Entity lifecycle
    - Mapping rules
    
    JPA gives standards but **does not execute anything itself**.
    
    ---
    
    ---
    
    # **7. Key Concepts in Hibernate**
    
    Hibernate provides:
    
    - Implementation of JPA interfaces (`EntityManager`, `PersistenceContext`)
    - Hibernate Session API
    - Hibernate Query Language (HQL)
    - Automatic dirty checking
    - First-level and second-level caching
    - Fetch strategies
    - Auditing support
    - Batch operations and tuning
    
    ---
    
    ---
    
    # **8. Real-world Analogy**
    
    - **JPA is like an electrical standard** — defines plug layout, voltage, safety spec.
    - **Hibernate is like a specific appliance manufacturer** — creates devices that follow those standards and adds extra features (remote control, energy saver mode).
    
    ---
    
    ---
    
    # **9. Which Should You Use?**
    
    ### Recommended Approach:
    
    Write code using **JPA interfaces/annotations**
    
    Run it using **Hibernate implementation**
    
    This gives:
    
    - Standard portable code
    - Hibernate’s powerful extra features behind the scenes
    
    ---
    
    ---
    
    # **10. Important Points**
    
    - Every JPA implementation is ORM; Hibernate is the most widely used.
    - Hibernate existed before JPA and inspired JPA standards.
    - JPA code is portable across providers; Hibernate-specific code is not.
    - You can mix JPA and Hibernate APIs, but best practice is to stay JPA-compliant unless special features are required.
    
    ---
    
    ---
    
    # **11. Summary**
    
    JPA is the **specification** that defines how ORM should work in Java, while Hibernate is the **implementation** that executes ORM operations.
    
    - Use **JPA** for standard API-level development.
    - Use **Hibernate** when deeper ORM features or optimizations are required.
    
    Together, they provide a powerful and flexible persistence solution for enterprise Java applications.
    
- Entities & persistence context
    
    # **Entities & Persistence Context**
    
    ---
    
    ## **1. Entities**
    
    ---
    
    ### **1.1 Definition and Explanation**
    
    An **Entity** in JPA is a Java class that represents a table in the relational database.
    
    Each object of that class corresponds to a row in the table.
    
    Entities are annotated using `@Entity`, making them persistable by ORM frameworks like Hibernate or JPA.
    
    ### **1.2 Entity Mapping Example**
    
    ```java
    @Entity
    @Table(name = "employee")
    public class Employee {
    
        @Id
        private int id;
        private String name;
        private double salary;
    }
    
    ```
    
    Mapping Interpretation:
    
    - `Employee` class → `employee` table
    - `id` field → primary key column
    - Objects of Employee → table rows
    
    ---
    
    ### **1.3 Purpose of Entities**
    
    - Represent database records as Java objects
    - Allow object-oriented coding instead of SQL-based coding
    - Enable ORM to map Java fields to DB columns
    - Support relationships like:
    
    ---
    
    ### **1.4 Entity Lifecycle States**
    
    | --- | --- |
    
    ORM frameworks transition entities between these states.
    
    ---
    
    ---
    
    ## **2. Persistence Context**
    
    ---
    
    ### **2.1 Definition and Explanation**
    
    Persistence Context is a **shared memory store (cache)** managed by JPA/Hibernate, where Entity instances are tracked and synchronized with the database.
    
    When an entity is managed inside persistence context:
    
    - Changed fields are automatically detected
    - Updates are written to database at commit time (dirty checking)
    - Identity is ensured — same entity is not loaded twice
    
    ### **2.2 Key Responsibilities**
    
    - Manage lifecycle of entities
    - Maintain 1st-level cache
    - Track changes in entities
    - Ensure entity uniqueness
    - Propagate updates to DB automatically
    
    ---
    
    ---
    
    ## **3. How Entities Interact with Persistence Context**
    
    ### **Step-by-step Flow**
    
    1. Application requests EntityManager.
    2. EntityManager creates Persistence Context.
    3. Entity is loaded or persisted — added to context.
    4. Any field changes tracked automatically.
    5. When transaction commits, context writes to DB.
    
    ---
    
    ---
    
    ## **4. EntityManager and Persistence Context Relationship**
    
    - EntityManager is the API.
    - Persistence Context is the internal storage and tracking mechanism.
    
    Calling methods like:
    
    - `find()`
    - `persist()`
    - `merge()`
    
    adds entities into the persistence context.
    
    ---
    
    ---
    
    ## **5. Behaviour of Persistence Context**
    
    ### **A. First-Level Cache**
    
    When retrieving the same entity twice:
    
    ```java
    Employee e1 = em.find(Employee.class, 101);
    Employee e2 = em.find(Employee.class, 101);
    
    ```
    
    Only **one** SQL SELECT executes — the second fetch returns from cache.
    
    ### **B. Dirty Checking**
    
    ```java
    Employee e = em.find(Employee.class, 101);
    e.setSalary(60000);  // change tracked
    em.getTransaction().commit(); // auto update to DB
    
    ```
    
    Hibernate generates `UPDATE` automatically.
    
    ### **C. Object Identity Guarantee**
    
    Two employees loaded from DB will refer to **same instance** in persistence context.
    
    ---
    
    ---
    
    ## **6. Entity State Transitions**
    
    | --- | --- |
    
    Persistence context manages these transitions.
    
    ---
    
    ---
    
    ## **7. Real-world Analogy**
    
    Entity is like a **form filled out for a user**.
    
    Persistence context is like a **registry office’s workspace** where:
    
    - Records are tracked
    - Updated values are reconciled
    - Everything is synced with master records (database)
    
    ---
    
    ---
    
    ## **8. Key Differences: Entity vs Persistence Context**
    
    | --- | --- | --- |
    
    ---
    
    ---
    
    ## **9. Important Points**
    
    - Entities model database tables.
    - Persistence context manages entities at runtime.
    - Persistence context ensures caching, change detection, and synchronization.
    - EntityManager interacts with persistence context, not database directly.
    - Entities become “persistent” only after joining persistence context.
    
    ---
    
    ---
    
    ## **10. Summary**
    
    Entities are Java classes mapped to database tables, representing application data.
    
    Persistence context is the runtime environment managed by JPA/Hibernate that keeps track of entity instances, caches them, monitors changes, and synchronizes updates with the database.
    
    Together, they form the backbone of ORM behavior and JPA-based persistence.
    
- Sessions & SessionFactory
    
    # **Sessions & SessionFactory (Hibernate Core Concepts)**
    
    ---
    
    ## **1. SessionFactory**
    
    ---
    
    ### **1.1 Definition and Explanation**
    
    `SessionFactory` is a **thread-safe, heavyweight Hibernate object** responsible for:
    
    - Creating and managing database sessions
    - Reading Hibernate configuration and mapping metadata
    - Acting as the factory for `Session` instances
    - Serving as a second-level cache provider
    
    It is usually created **once per application** at startup and reused throughout the application lifecycle.
    
    ---
    
    ### **1.2 Purpose**
    
    - Central configuration point
    - Provides Sessions for database operations
    - Manages entity metadata
    - Supports second-level caching
    
    ---
    
    ### **1.3 Creation of SessionFactory**
    
    Typically configured in Hibernate config XML or via JPA Bootstrapping:
    
    ```java
    SessionFactory factory = new Configuration()
        .configure("hibernate.cfg.xml")
        .buildSessionFactory();
    
    ```
    
    ---
    
    ---
    
    ## **2. Session**
    
    ---
    
    ### **2.1 Definition and Explanation**
    
    A `Session` represents a **single connection unit** between the application and the database.
    
    It:
    
    - Is **lightweight**
    - Is **not thread-safe**
    - Maps to a database connection
    - Is used for CRUD operations
    - Manages persistence context (first-level cache)
    
    Every Session communicates with Hibernate’s persistence mechanism.
    
    ---
    
    ### **2.2 Typical Usage**
    
    ```java
    Session session = factory.openSession();
    Transaction tx = session.beginTransaction();
    
    Employee emp = session.get(Employee.class, 101);
    session.save(new Employee(102, "John", 50000));
    
    tx.commit();
    session.close();
    
    ```

### **14.2 Entity Mapping**

- Overview
    
    # **Entity Mapping (JPA/Hibernate)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Entity Mapping refers to the process of **linking Java objects (entities) to database tables**.
    
    This is the core of ORM functionality where:
    
    - **Entity class → Table**
    - **Entity field → Column**
    - **Relationships → Foreign keys / Join tables**
    
    Entity Mapping enables applications to perform persistence operations using entities instead of writing raw SQL.
    
    ---
    
    ---
    
    ## **2. Why Entity Mapping?**
    
    - Avoid manual SQL handling
    - Automate database CRUD operations
    - Promote clean object-oriented modeling
    - Help maintain consistency between Java domain model and DB schema
    
    ---
    
    ---
    
    ## **3. Basic Entity Mapping**
    
    ### **Example Entity**
    
    ```java
    @Entity
    @Table(name = "employee")
    public class Employee {
    
        @Id
        private int id;
        private String name;
        private double salary;
    }
    
    ```
    
    ### Mapping Interpretation
    
    | --- | --- |
    
    ORM understands this mapping and auto-generates queries like:
    
    ```sql
    SELECT * FROM employee;
    INSERT INTO employee(id, name, salary) VALUES (?, ?, ?);
    
    ```
    
    ---
    
    ---
    
    ## **4. Mapping Annotations**
    
    ### **4.1 Entity Annotation**
    
    ```java
    @Entity
    
    ```
    
    Marks class as persistent.
    
    ### **4.2 Table Annotation**
    
    ```java
    @Table(name = "employee")
    
    ```
    
    Defines table name.
    
    ### **4.3 Id Annotation**
    
    ```java
    @Id
    
    ```
    
    Marks primary key field.
    
    ### **4.4 Column Mapping**
    
    ```java
    @Column(name = "emp_name", nullable = false)
    private String name;
    
    ```
    
    ### **4.5 Generated Values**
    
    ```java
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    ```
    
    ---
    
    ---
    
    ## **5. Relationships Mapping**
    
    ### **5.1 One-to-One**
    
    ```java
    @OneToOne
    @JoinColumn(name = "address_id")
    private Address address;
    
    ```
    
    ### **5.2 One-to-Many**
    
    ```java
    @OneToMany(mappedBy = "dept")
    private List<Employee> employees;
    
    ```
    
    ### **5.3 Many-to-One**
    
    ```java
    @ManyToOne
    @JoinColumn(name = "dept_id")
    private Department dept;
    
    ```
    
    ### **5.4 Many-to-Many**
    
    ```java
    @ManyToMany
    @JoinTable(
        name = "student_course",
        joinColumns = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private List<Course> courses;
    
    ```
    
    These annotations translate object relationships into database foreign-key relationships.

    ### **6.2 Inheritance Mapping**

    - `@Inheritance(strategy = InheritanceType.SINGLE_TABLE)`
    - `JOINED`
    - `TABLE_PER_CLASS`

- @Entity, @Id, @GeneratedValue
    
    # **@Entity, @Id, @GeneratedValue**
    
    ---
    
    ## **1. @Entity**
    
    ---
    
    ### **1.1 Definition and Explanation**
    
    `@Entity` is a JPA annotation used to declare a Java class as a **persistent entity**, meaning it represents a database table.
    
    When a class is marked with `@Entity`, JPA/Hibernate:
    
    - Maps the class to a table
    - Manages lifecycle, persistence, retrieval, and transaction synchronization
    - Treats each object as a row in the table
    
    ---
    
    ### **1.2 Rules for @Entity**
    
    - Class must have a no-arg constructor.
    - Class must not be final (Hibernate enhances bytecode).
    - Fields should be private and accessed via getters/setters.
    - Must have exactly one `@Id` annotated field.
    
    ---
    
    ### **1.3 Example**
    
    ```java
    @Entity
    public class Employee {
        @Id
        private int id;
        private String name;
    }
    
    ```
    
    Automatically maps:
    
    - `Employee` class → `employee` table
    - Fields → columns
    
    ---
    
    ---
    
    ## **2. @Id**
    
    ---
    
    ### **2.1 Definition and Explanation**
    
    `@Id` marks a field as the **primary key** of an entity.
    
    JPA requires every entity to have a primary key to:
    
    - Uniquely identify records
    - Track entity state in persistence context
    - Perform updates and deletes reliably
    
    ---
    
    ### **2.2 Example**
    
    ```java
    @Entity
    public class Employee {
        @Id
        private int id;
    }
    
    ```
    
    ---
    
    ### **2.3 Characteristics**
    
    - Must be unique across table.
    - May be manually assigned or auto-generated.
    - Can be primitive or wrapper type (int, Long, UUID, etc.).
    
    ---
    
    ---
    
    ## **3. @GeneratedValue**
    
    ---
    
    ### **3.1 Definition and Explanation**
    
    `@GeneratedValue` defines **how the primary key will be generated**.
    
    It works with `@Id`, enabling automatic key assignment rather than manual key setting.
    
    ---
    
    ### **3.2 Generation Strategies**
    
    | --- | --- | --- |
    
    ---
    
    ### **3.3 Syntax**
    
    ```java
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    ```
    
    ---
    
    ### **3.4 Example with AUTO**
    
    ```java
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Integer id;
    
    ```
    
    Hibernate chooses appropriate strategy based on DB dialect.
    
    ---
    
    ---
    
    ## **4. Complete Example**
    
    ```java
    @Entity
    public class Product {
    
        @Id
        @GeneratedValue(strategy = GenerationType.IDENTITY)
        private Long id;
    
        private String name;
        private double price;
    }
    
    ```
    
    Mapping:
    
    - Table: product
    - id column auto-generated (AUTO_INCREMENT)
    - name, price mapped as columns
    
    ---
    
    ---
    
    ## **5. Internal Behavior / Persistence Context Interaction**
    
    ### When saving entity:
    
    ```java
    Product p = new Product();
    p.setName("Phone");
    p.setPrice(15000);
    
    em.persist(p);
    
    ```
    
    Hibernate:
    
    1. Inserts row
    2. Retrieves generated id
    3. Assigns id to entity instance
    4. Tracks it in persistence context
    
    ---
    
    ---
    
    ## **6. Real-world Analogy**
    
    - `@Entity` → Declaring something is a **registered citizen**
    - `@Id` → Assigning a **unique ID number**
    - `@GeneratedValue` → Automatic **government ID issuance**
    
    ---
    
    ---
    
    ## **7. Important Points**
    
    - Every entity must have an `@Id`.
    - `@GeneratedValue` removes need for manual id assignment.
    - Incorrect mapping of id generation strategy can cause failure.
    - Hibernate may use sequences, tables, or identity based on dialect.
    
    ---
    
    ---
    
    ## **8. Summary**
    
    - `@Entity` marks a class as persistable.
    - `@Id` defines the primary key.
    - `@GeneratedValue` defines how the key is auto-generated.
    
    Together, these form the foundational annotations for entity persistence and identity management in JPA/Hibernate.
    
- Table/column mapping
    
    # **Table and Column Mapping in JPA/Hibernate**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Table/Column Mapping describes how **Java entity classes and their fields** relate to **database tables and columns**.
    
    This is a fundamental ORM feature that allows Hibernate/JPA to translate objects into relational data.
    
    Mapping tells ORM:
    
    - Which table an entity belongs to
    - Which database column stores each property
    
    Without mapping, Hibernate would not know how to read or write entity values from/to the database.
    
    ---
    
    ---
    
    ## **2. Table Mapping**
    
    ### **2.1 Annotation: `@Table`**
    
    `@Table` is applied on a class annotated with `@Entity`.
    
    It specifies:
    
    - Table name
    - Schema name
    - Unique constraints
    
    ### **Syntax**
    
    ```java
    @Entity
    @Table(name = "employee")
    public class Employee {
        ...
    }
    
    ```
    
    If `@Table` is omitted, JPA uses the class name as table name by default.

    ```java
    @Entity
    @Table(
        name = "employee_master",
        schema = "hr",

    public class Employee { ... }
    
    ```

    ### **3.1 Annotation: `@Column`**
    
    `@Column` specifies a field → database column mapping.
    
    It is optional because Hibernate maps field names to column names if matching exists, but explicit mapping improves control.
    
    ### **Basic Example**
    
    ```java
    @Column(name = "emp_name")
    private String name;
    
    ```

    | --- | --- |

    ```java

    ```

    ## **4. Default Mapping Behavior**
    
    Without `@Table` or `@Column`:
    
    ```java
    @Entity

    }
    
    ```

    - Table name = class name (`product`)
    - Column name = field name (`name`)

### **14.4 Fetching Strategies**

# **Fetching Strategies in JPA/Hibernate**

---

## **1. Definition and Explanation**

Fetching strategy determines **when** related entities are loaded from the database:

- Immediately when the parent loads (**Eager** fetch)
- Only when accessed later (**Lazy** fetch)

Choosing correct fetch strategy is essential for:

- Performance
- Memory usage
- Avoiding unnecessary database queries

JPA defines fetching using two primary annotations:

```java
FetchType.LAZY
FetchType.EAGER

```

---

---

# **2. Lazy Loading**

---

## **2.1 Definition**

Lazy loading defers loading of associated entities until they are explicitly accessed.

### Code Example

```java
@Entity
public class Department {

    @OneToMany(mappedBy="department", fetch = FetchType.LAZY)
    private List<Employee> employees;
}

```

### Behavior

- Department loads
- Employees list is represented as a **proxy**
- When `.getEmployees().size()` or `.get(0)` accessed, Hibernate fetches data

---

### Benefits of Lazy Loading

- Efficient memory usage
- Faster initial loading
- Avoids unnecessary joins

---

### When It Is Default

| --- | --- |

---

---

# **3. Eager Loading**

---

## **3.1 Definition**

Eager loading fetches associated data **immediately during parent entity load**.

### Code Example

```java
@ManyToOne(fetch = FetchType.EAGER)
private Department department;

```

### Behavior

- Loads department and its related data instantly

---

### Benefits of Eager Loading

- Convenience for frequently used, small relationships
- Avoids LazyInitializationException outside persistence context

---

### When It Is Default

| --- | --- |

---

---

# **4. Which Fetching Strategy Should You Prefer?**

| --- | --- |

Best practice:

- Prefer **Lazy loading**
- Fetch explicitly using joins, fetch profiles, DTO queries, etc.

# **5. The N+1 Problem**

---

## **5.1 What Is the N+1 Query Problem?**

Occurs when:

- One query loads N parent records.
- For each parent, an additional query loads children.
- Result: **1 + N queries instead of 2 queries**

### Example Scenario

Code:

```java
List<Department> depts = em.createQuery("from Department", Department.class).getResultList();

for (Department d : depts) {
    d.getEmployees().size();
}

```

Queries executed:

```
SELECT * FROM department;
SELECT * FROM employee WHERE dept_id=1;
SELECT * FROM employee WHERE dept_id=2;
...

```

If N = 50 departments → 51 queries executed → slow performance.

---

---

## **5.2 Why It Happens**

Because of:

- LAZY loading default behavior
- ORM accessing child entities one-by-one

---

---

# **6. Solutions to N+1 Problem**

---

### **Solution 1: Fetch Join (Recommended)**

```java
SELECT d FROM Department d
LEFT JOIN FETCH d.employees

```

JPA Example:

```java
List<Department> depts =
    em.createQuery(
        "SELECT d FROM Department d JOIN FETCH d.employees",
        Department.class
    ).getResultList();

```

This loads:

- Departments
- Employees
- In **one single query**

---

---

### **Solution 2: Use Entity Graphs**

```java
@EntityGraph(attributePaths = "employees")
List<Department> findAll();

```

Spring Data JPA supports this declaratively.

---

---

### **Solution 3: Batch Size Fetching**

Hibernate optimization — batch load collections:

```java
@BatchSize(size = 10)
private List<Employee> employees;

```

Hibernate loads employee records in fixed batches instead of one-by-one.

---

---

### **Solution 4: Second-Level Cache**

Enables repeated queries to be avoided for cached entities.

---

---

### **Solution 5: DTO Projections**

Instead of loading full entities:

```java
SELECT new com.tech.EmployeeDTO(e.name, d.name)
FROM Employee e JOIN e.department d

```

Load only required data — no child queries.

---

---

# **7. Fetching Guidelines / Best Practices**

- Prefer **LAZY** by default
- Use **JOIN FETCH** for aggregating child data
- Avoid EAGER on collections
- Avoid Lazy fetching outside persistence context (may cause `LazyInitializationException`)
- Design service layer boundaries that load what is required

---

---

# **8. Real-world Analogy**

- **Lazy Loading**: Only open detailed student files **when** required.
- **Eager Loading**: Open all student files **immediately**, whether needed or not.
- **N+1 Problem**: Opening each student file separately after already opening the cabinet — extremely inefficient.

---

---

# **9. Summary**

- Fetch strategies govern when related data loads.
- **Lazy loading** delays loading until access; efficient for large relationships.
- **Eager loading** loads immediately; useful for small/highly used references.
- The **N+1 problem** occurs when many lazy children trigger extra queries.
- Solutions include fetch joins, batching, projections, caching, and entity graphs.

Choosing appropriate fetching strategy significantly improves performance and scalability of JPA/Hibernate applications.
