## **Module 16 — Spring Core (IoC & DI)**

### **16.1 Spring Framework Introduction**

- Overview
    
    # **Spring Framework Introduction**
    
    ---
    
    ## **1. Definition and Overview**
    
    The **Spring Framework** is a comprehensive, lightweight, open-source framework for **building modern Java enterprise applications**.
    
    It promotes loose coupling, modular structure, and clean architecture through dependency injection, aspect-oriented programming, and layered design principles.
    
    Spring simplifies development of:
    
    - Web applications
    - REST APIs
    - Microservices
    - Enterprise systems
    
    It provides a consistent programming model and integrates with other technologies such as Hibernate/JPA, JDBC, and messaging systems.
    
    ---
    
    ---
    
    ## **2. Why Spring Framework?**
    
    Spring was created to address the complexity of Java EE (J2EE) and to offer:
    
    - Better modularity
    - Reduced boilerplate code
    - POJO-based programming
    - Reusable infrastructure support
    
    Spring encourages writing business logic as simple Java classes that the framework manages.
    
    ---
    
    ---
    
    ## **3. Core Concepts**
    
    ### **A. Inversion of Control (IoC)**
    
    Spring manages object creation and wiring rather than the application doing it directly.
    
    ### **B. Dependency Injection (DI)**
    
    Dependencies are wired into classes via constructor, setter, or field injection — improving testability and modularity.
    
    ### **C. Aspect-Oriented Programming (AOP)**
    
    Separates cross-cutting concerns like logging, security, and transactions from business logic.
    
    ---
    
    ---
    
    ## **4. Spring Modules / Architecture**
    
    Spring is modular, meaning developers choose only required components.
    
    Key modules include:
    
    | Layer | Modules |
    | --- | --- |
    | Core Container | Core, Beans, Context |
    | AOP | Spring AOP |
    | Data Access | JDBC, ORM, Transaction |
    | Web | Web MVC, WebFlux |
    | Testing | Spring Test |
    | Cloud/Microservices | Spring Boot, Spring Cloud |
    
    ---
    
    ---
    
    ## **5. Spring Benefits**
    
    - Lightweight and modular
    - Loose coupling
    - Easy unit testing
    - Declarative transaction management
    - Integration with frameworks (Hibernate, JPA, Kafka, security, cloud)
    - Container-managed dependency wiring
    
    ---
    
    ---
    
    ## **6. Spring Bean and IoC Container**
    
    A **Spring Bean** is an object managed by the Spring IoC container.
    
    Configuration options include:
    
    - XML configuration
    - Java-based annotations
    - Java Config classes
    
    Spring creates beans, initializes them, injects dependencies, and manages their lifecycle.
    
    ---
    
    ---
    
    ## **7. Dependency Injection Types**
    
    - **Constructor Injection**
    - **Setter Injection**
    - **Field Autowiring (not recommended without awareness)**
    
    Example:
    
    ```java
    @Component
    public class OrderService {
        private final PaymentService payment;
    
        @Autowired
        public OrderService(PaymentService payment) {
            this.payment = payment;
        }
    }
    
    ```
    
    ---
    
    ---
    
    ## **8. Spring MVC Overview**
    
    Spring MVC is a web framework in Spring for building request-driven applications.
    
    It follows MVC pattern:
    
    - Controller → Handles request
    - Service → Business logic
    - DAO/Repository → Data access
    - View (JSP/Thymeleaf) → Presentation
    
    ---
    
    ---
    
    ## **9. Spring Boot**
    
    Spring Boot builds on Spring Framework to:
    
    - Auto-configure components
    - Reduce boilerplate configuration
    - Package applications as executable jars
    
    Most modern Spring projects use Spring Boot due to ease of development.
    
    ---
    
    ---
    
    ## **10. Real-World Analogy**
    
    Spring works like **a factory and supervisor**:
    
    - Factory builds objects (beans)
    - Supervisor injects dependencies
    - System functions smoothly without each worker assembling their own tools
    
    ---
    
    ---
    
    ## **11. Best Practices with Spring**
    
    - Design interfaces for services
    - Prefer constructor-based injection
    - Keep configuration centralized
    - Avoid business logic in controllers
    - Apply layered architecture (Controller → Service → DAO)
    
    ---
    
    ---
    
    ## **12. Summary**
    
    Spring Framework is a powerful ecosystem enabling scalable, modular, and loosely coupled enterprise Java applications through IoC, DI, and AOP principles.
    
    It supports web development, security, messaging, data access, and microservices while significantly reducing configuration complexity in traditional Java EE development.
    
- IoC container
    
    # **IoC Container in Spring**
    
    ---
    
    ## **1. Definition and Explanation**
    
    The **Inversion of Control (IoC) Container** is the **core engine** of the Spring Framework responsible for:
    
    - Creating objects (beans)
    - Managing their lifecycles
    - Resolving their dependencies
    - Wiring them together
    
    Instead of applications constructing and managing objects manually, Spring’s IoC container **controls object creation and dependency management**—this is called Inversion of Control.
    
    ---
    
    ---
    
    # **2. Purpose of IoC Container**
    
    The IoC container exists to:
    
    - Decouple application components
    - Improve reusability and maintainability
    - Enable easy testing and modular design
    
    Using container-based dependency injection, developers focus on business logic while Spring manages wiring.
    
    ---
    
    ---
    
    # **3. Key Responsibilities**
    
    1. **Instantiate objects (beans)**
    2. **Inject required dependencies**
    3. **Configure objects as per Java/annotation/XML metadata**
    4. **Manage bean lifecycle (init/destroy callbacks)**
    5. **Provide beans upon request to the application**
    
    ---
    
    ---
    
    # **4. Major IoC Container Implementations**
    
    Spring provides two main container types:
    
    | Container | Implementation Class |
    | --- | --- |
    | **BeanFactory** | `org.springframework.beans.factory.BeanFactory` |
    | **ApplicationContext** | `org.springframework.context.ApplicationContext` |
    
    ---
    
    ---
    
    # **5. BeanFactory vs ApplicationContext**
    
    | Aspect | BeanFactory | ApplicationContext |
    | --- | --- | --- |
    | Basic container | Yes | Yes + more features |
    | Lazy initialization | Yes | Eager by default |
    | AOP integration | Limited | Full |
    | Event handling | No | Yes |
    | Application area | Lightweight/embedded apps | Enterprise apps |
    
    ApplicationContext is recommended for most use cases.
    
    ---
    
    ---
    
    # **6. IoC Container Workflow**
    
    ### Step 1: Load Configuration
    
    - XML file
    - Java annotations
    - Java configuration class
    
    ### Step 2: Scan and construct beans
    
    ### Step 3: Inject dependencies
    
    ### Step 4: Manage bean lifecycle
    
    - init methods
    - destroy methods
    
    ### Step 5: Serve beans on request
    
    ---
    
    ---
    
    # **7. Example Using Annotation Configuration**
    
    ### Bean definitions
    
    ```java
    @Component
    public class PaymentService {
        public void process() {
            System.out.println("Processing payment...");
        }
    }
    
    ```
    
    ### Obtain bean via container
    
    ```java
    ApplicationContext context =
            new AnnotationConfigApplicationContext(AppConfig.class);
    
    PaymentService service = context.getBean(PaymentService.class);
    service.process();
    
    ```
    
    ---
    
    ---
    
    # **8. Configuration Types for IoC**
    
    ### **XML-based**
    
    ```xml
    <bean id="paymentService" class="com.tech.PaymentService" />
    
    ```
    
    ### **Annotation-based**
    
    `@Component`, `@Service`, `@Repository`, `@Controller`
    
    ### **Java Config**
    
    ```java
    @Configuration
    public class AppConfig {
        @Bean
        public PaymentService paymentService() {
            return new PaymentService();
        }
    }
    
    ```
    
    ---
    
    ---
    
    # **9. Bean Lifecycle Inside IoC Container**
    
    1. Instantiate bean
    2. Populate properties
    3. Call lifecycle callbacks (`@PostConstruct`, init-method)
    4. Ready for use
    5. On shutdown, destroy callback executed
    
    ---
    
    ---
    
    # **10. Dependency Injection Mechanisms**
    
    - Constructor injection
    - Setter injection
    - Field injection (discouraged unless aware)
    
    Example:
    
    ```java
    @Component
    public class OrderService {
    
        private final PaymentService payment;
    
        @Autowired
        public OrderService(PaymentService payment) {
            this.payment = payment;
        }
    }
    
    ```
    
    ---
    
    ---
    
    # **11. Real-world Analogy**
    
    The IoC container works like **a factory and supply chain manager**:
    
    - It creates tools (beans)
    - Configures them
    - Supplies them when workers (application objects) request them
    
    Workers just use tools; they do not craft or maintain them.
    
    ---
    
    ---
    
    # **12. Benefits of IoC Container**
    
    - Loose coupling
    - Improved testability
    - Centralized configuration
    - Controlled lifecycle
    - Easier scalability and maintenance
    
    ---
    
    ---
    
    # **13. Summary**
    
    The IoC Container is Spring’s foundational engine that constructs, configures, wires, and manages beans.
    
    It embodies inversion of control, letting Spring—not application code—handle dependency management, improving flexibility and modularity in enterprise applications.
    
- BeanFactory vs ApplicationContext
    
    # **BeanFactory vs ApplicationContext**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Spring provides two main IoC container implementations:
    
    1. **BeanFactory** – the basic IoC container
    2. **ApplicationContext** – an advanced, enterprise-ready container
    
    Both manage beans, perform dependency injection, and enable IoC.
    
    However, ApplicationContext extends BeanFactory and includes additional enterprise support.
    
    ---
    
    ---
    
    # **2. BeanFactory**
    
    ---
    
    ## **2.1 Definition**
    
    `BeanFactory` is the **core and simplest IoC container** in Spring.
    
    It provides:
    
    - Bean creation
    - Dependency injection
    - Lazy initialization
    
    It loads bean definitions only when requested.
    
    ---
    
    ## **2.2 When to Use BeanFactory?**
    
    - Lightweight applications
    - Memory-constrained environments
    - Framework-level services
    
    ---
    
    ---
    
    # **3. ApplicationContext**
    
    ---
    
    ## **3.1 Definition**
    
    `ApplicationContext` is a **more feature-rich IoC container** built on BeanFactory.
    
    It provides all BeanFactory functions plus:
    
    - Eager initialization
    - Event handling
    - Internationalization
    - AOP support
    - Automatic bean post-processing
    - Injection of common enterprise services
    
    It is designed for **enterprise and web applications**.
    
    ---
    
    ---
    
    # **4. Comparison Table**
    
    | Feature | BeanFactory | ApplicationContext |
    | --- | --- | --- |
    | Type | Basic IoC container | Advanced container |
    | Bean initialization | Lazy by default | Eager by default |
    | Event publishing | No | Yes |
    | AOP support | Limited | Full lifecycle support |
    | Internationalization | No | Yes |
    | Message resource support | No | Yes |
    | Auto Bean Post Processing | Manual | Automatic |
    | Annotation scanning | Must be explicit | Built-in support |
    | Context refresh | Manual | Automatic |
    | Use Case | Simple apps | Large enterprise apps |
    
    ---
    
    ---
    
    # **5. Code Usage Differences**
    
    ### BeanFactory
    
    ```java
    BeanFactory factory =
        new XmlBeanFactory(new ClassPathResource("beans.xml"));
    
    PaymentService service = factory.getBean(PaymentService.class);
    
    ```
    
    ### ApplicationContext
    
    ```java
    ApplicationContext context =
        new ClassPathXmlApplicationContext("beans.xml");
    
    PaymentService service = context.getBean(PaymentService.class);
    
    ```
    
    ---
    
    ---
    
    # **6. Practical Implications**
    
    - BeanFactory initializes beans **only when first requested**, which conserves memory.
    - ApplicationContext initializes beans **during startup**, ensuring faster access later and better readiness for enterprise behaviors.
    
    ---
    
    ---
    
    # **7. Real-world Analogy**
    
    - **BeanFactory** is like a **simple hand-operated machine** — minimal, functional but manual work needed.
    - **ApplicationContext** is like an **automated assembly line** — performs extra tasks without manual intervention.
    
    ---
    
    ---
    
    # **8. Best Practices**
    
    - Prefer `ApplicationContext` for real-world development.
    - Use BeanFactory only for internal frameworks or where resource footprint must be extremely low.
    
    ---
    
    ---
    
    # **9. Summary**
    
    BeanFactory is the foundational container providing basic IoC, while ApplicationContext is its advanced version offering enterprise services.
    
    In modern Spring development, **ApplicationContext is the standard choice**, due to richer features and better integration with Spring’s ecosystem.
