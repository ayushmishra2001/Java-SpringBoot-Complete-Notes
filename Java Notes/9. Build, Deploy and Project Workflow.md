## **Module 9 — Build, Deploy & Project Workflow**

### **9.1 Maven Essentials**

- Maven
    
    # **Maven Essentials**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    **Maven** is a powerful **build automation and dependency management tool** used primarily for Java projects. It provides a standardized way to build, test, package, and manage project dependencies through a set of conventions and configuration files.
    
    ### **Purpose**
    
    - Automate build processes
    - Manage external libraries (dependencies)
    - Ensure consistent project structure
    - Provide reproducible builds
    - Simplify integration with CI/CD tools
    - Standardize project lifecycle
    
    ---
    
    ## **2. Key Terminology**
    
    ### **A. POM (Project Object Model)**
    
    Central configuration file: `pom.xml`.
    
    Contains:
    
    - Project metadata
    - Dependencies
    - Plugins
    - Build settings
    - Versioning
    - Modules
    - Profiles
    
    ### **B. Maven Repository**
    
    Stores project dependencies.
    
    Three types:
    
    - **Local Repository**: Located in your machine (`~/.m2/repository`)
    - **Central Repository**: Default public repo used by Maven
    - **Remote Repository**: Custom corporate or third-party repos
    
    ### **C. Coordinates**
    
    Each dependency is identified uniquely by:
    
    ```
    groupId, artifactId, version, packaging
    
    ```
    
    Example:
    
    ```xml
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>6.0.0</version>
    </dependency>
    
    ```
    
    ---
    
    ## **3. Standard Maven Project Structure**
    
    ```
    project/
    │
    ├── src/
    │   ├── main/
    │   │   ├── java/         → Source code
    │   │   ├── resources/    → Config files
    │   │   └── webapp/       → Web files (for web apps)
    │   │
    │   └── test/
    │       ├── java/         → Test cases
    │       └── resources/
    │
    └── pom.xml               → Maven configuration
    
    ```
    
    Follows convention-over-configuration.
    
    ---
    
    ## **4. Maven Build Lifecycle**
    
    ### **A. Clean Lifecycle**
    
    - `clean` → deletes target directory
    
    ### **B. Default (Build) Lifecycle**
    
    Important phases:
    
    - `validate`
    - `compile`
    - `test`
    - `package`
    - `verify`
    - `install`
    - `deploy`
    
    Example:
    
    ```bash
    mvn clean install
    
    ```
    
    ### **C. Site Lifecycle**
    
    - Generates documentation
    
    ---
    
    ## **5. Dependency Management**
    
    ### **A. Adding Dependencies**
    
    ```xml
    <dependencies>
        <dependency>
            <groupId>mysql</groupId>
            <artifactId>mysql-connector-java</artifactId>
            <version>8.0.33</version>
        </dependency>
    </dependencies>
    
    ```
    
    ### **B. Transitive Dependencies**
    
    Maven automatically pulls dependent libraries required by other libraries.
    
    Example:
    
    Spring Web → brings Jackson, Spring Core, etc.
    
    ### **C. Dependency Scopes**
    
    | Scope | Used In | Available In |
    | --- | --- | --- |
    | `compile` | Default | Everywhere |
    | `provided` | Servlet API, JDK libs | Compile, not packaged |
    | `runtime` | JDBC drivers | Only runtime |
    | `test` | JUnit | Only test |
    | `system` | Local JARs | Must specify path |
    
    ### **D. Exclusions**
    
    Remove unwanted transitive dependencies:
    
    ```xml
    <exclusion>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-log4j12</artifactId>
    </exclusion>
    
    ```
    
    ---
    
    ## **6. Maven Plugins**
    
    Plugins perform tasks such as compiling, packaging, testing, etc.
    
    ### **Common Plugins**
    
    ### **Compiler Plugin**
    
    ```xml
    <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-compiler-plugin</artifactId>
        <version>3.10.1</version>
        <configuration>
            <source>17</source>
            <target>17</target>
        </configuration>
    </plugin>
    
    ```
    
    ### **Surefire Plugin (Testing)**
    
    ```xml
    <maven-surefire-plugin>
    
    ```
    
    ### **Jar Plugin**
    
    ```xml
    <maven-jar-plugin>
    
    ```
    
    ### **Spring Boot Plugin**
    
    ```xml
    <plugin>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-maven-plugin</artifactId>
    </plugin>
    
    ```
    
    ---
    
    ## **7. Multi-Module Projects**
    
    Structure:
    
    ```
    root
    ├── module1
    └── module2
    
    ```
    
    Parent `pom.xml` manages common dependencies:
    
    ```xml
    <modules>
        <module>module1</module>
        <module>module2</module>
    </modules>
    
    ```
    
    Child modules use:
    
    ```xml
    <parent>
        <groupId>com.example</groupId>
        <artifactId>project-parent</artifactId>
        <version>1.0.0</version>
    </parent>
    
    ```
    
    ---
    
    ## **8. Version Management**
    
    ### **Dependency Management Section**
    
    ```xml
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-context</artifactId>
                <version>6.0.0</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    ```
    
    Child modules inherit versions without specifying them.
    
    ### **Properties for Versions**
    
    ```xml
    <properties>
        <spring.version>6.0.0</spring.version>
    </properties>
    
    ```
    
    Use:
    
    ```xml
    <version>${spring.version}</version>
    
    ```
    
    ---
    
    ## **9. Profiles**
    
    Enable environment-specific builds.
    
    Example:
    
    ```xml
    <profiles>
        <profile>
            <id>dev</id>
            <properties>
                <db.url>jdbc:mysql://localhost/dev</db.url>
            </properties>
        </profile>
    </profiles>
    
    ```
    
    Run:
    
    ```
    mvn clean install -Pdev
    
    ```
    
    ---
    
    ## **10. Maven Wrapper (mvnw)**
    
    Provides consistent Maven version across machines.
    
    Generate:
    
    ```
    mvn -N io.takari:maven:wrapper
    
    ```
    
    Run:
    
    ```
    ./mvnw clean install
    
    ```
    
    ---
    
    ## **11. Real-world Analogy**
    
    | Maven Concept | Analogy |
    | --- | --- |
    | pom.xml | Blueprint of a building |
    | Dependencies | Raw materials |
    | Repositories | Warehouses storing materials |
    | Plugins | Specialized machines |
    | Lifecycle | Steps in construction |
    | Multi-module project | Apartment complex with multiple buildings |
    
    ---
    
    ## **12. Important Points**
    
    - Maven standardizes Java project builds
    - pom.xml is the heart of configuration
    - Maven automatically manages transitive dependencies
    - Build lifecycle simplifies complex build tasks
    - Profiles support environment-specific builds
    - Wrapper ensures consistent Maven version
    - Plugins extend functionality significantly
    
    ---
    
    ## **13. Summary**
    
    | Feature | Description |
    | --- | --- |
    | **Intent** | Automate build and dependency management |
    | **Core File** | `pom.xml` |
    | **Benefits** | Standardization, automation, dependency resolution |
    | **Project Structure** | Convention-driven |
    | **Plugins** | Extend build capabilities |
    | **Repositories** | Store artifacts |
    
    **Maven** is essential for modern Java development, providing a robust, consistent, and automated build and dependency management system that integrates seamlessly with enterprise frameworks like Spring.
    
- POM structure
    
    # **POM Structure (Project Object Model)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **POM (Project Object Model)** is the fundamental XML configuration file used by Maven.
    
    It defines the project’s:
    
    - Metadata
    - Dependencies
    - Build configurations
    - Plugins
    - Profiles
    - Modules
    - Versioning information
    
    The POM follows a **hierarchical and declarative structure**, allowing Maven to understand and manage the project’s lifecycle.
    
    ---
    
    ## **2. Standard POM Structure Overview**
    
    A typical `pom.xml` includes:
    
    ```xml
    <project>
        <modelVersion>4.0.0</modelVersion>
    
        <groupId>com.example</groupId>
        <artifactId>my-app</artifactId>
        <version>1.0.0</version>
        <packaging>jar</packaging>
    
        <name>My Application</name>
        <description>Sample Maven project</description>
    
        <properties>...</properties>
    
        <dependencies>...</dependencies>
    
        <build>...</build>
    
        <repositories>...</repositories>
    
        <dependencyManagement>...</dependencyManagement>
    
        <modules>...</modules>
    
        <profiles>...</profiles>
    </project>
    
    ```
    
    Each section has a specific purpose explained below.
    
    ---
    
    ## **3. Detailed Components of POM**
    
    ---
    
    ## **A. `<modelVersion>`**
    
    - Defines the POM model version
    - Always `"4.0.0"` for modern Maven projects
    
    ```xml
    <modelVersion>4.0.0</modelVersion>
    
    ```
    
    ---
    
    ## **B. Project Coordinates (GAV)**
    
    ### **1. `<groupId>`**
    
    Uniquely identifies the project’s group or organization.
    
    Example:
    
    ```xml
    <groupId>com.techacademy</groupId>
    
    ```
    
    ### **2. `<artifactId>`**
    
    Name of the project (generated artifact name).
    
    Example:
    
    ```xml
    <artifactId>employee-service</artifactId>
    
    ```
    
    ### **3. `<version>`**
    
    Project’s version.
    
    Example:
    
    ```xml
    <version>1.0.0-SNAPSHOT</version>
    
    ```
    
    ### **4. `<packaging>`**
    
    Defines how the project is packaged.
    
    Possible values:
    
    - `jar`
    - `war`
    - `pom`
    - `ear`
    
    Example:
    
    ```xml
    <packaging>jar</packaging>
    
    ```
    
    ---
    
    ## **C. Project Information**
    
    ```xml
    <name>Employee Service</name>
    <description>Handles employee operations</description>
    <url>https://example.com/employee-service</url>
    
    ```
    
    Optional sections:
    
    - `licenses`
    - `developers`
    - `organization`
    - `issueManagement`
    
    ---
    
    ## **D. `<properties>`**
    
    Defines reusable values (e.g., Java version, dependency versions).
    
    ```xml
    <properties>
        <java.version>17</java.version>
        <spring.version>6.0.0</spring.version>
    </properties>
    
    ```
    
    Used inside POM:
    
    ```xml
    <version>${spring.version}</version>
    
    ```
    
    ---
    
    ## **E. `<dependencies>` Section**
    
    Contains list of libraries required by the project.
    
    ```xml
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>6.0.0</version>
            <scope>compile</scope>
        </dependency>
    </dependencies>
    
    ```
    
    Maven resolves transitive dependencies automatically.
    
    ---
    
    ## **F. `<dependencyManagement>`**
    
    Defines dependency versions for child modules.
    
    ```xml
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>com.fasterxml.jackson.core</groupId>
                <artifactId>jackson-databind</artifactId>
                <version>2.15.0</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    ```
    
    Child POMs can then reference without specifying version.
    
    ---
    
    ## **G. `<build>` Section**
    
    Controls build configuration.
    
    ```xml
    <build>
        <finalName>employee-service</finalName>
    
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.10.1</version>
                <configuration>
                    <source>17</source>
                    <target>17</target>
                </configuration>
            </plugin>
        </plugins>
    </build>
    
    ```
    
    Contains:
    
    - plugins
    - pluginManagement
    - sourceDirectory
    - resources
    - testResources
    
    ---
    
    ## **H. `<repositories>`**
    
    Defines additional remote repositories.
    
    ```xml
    <repositories>
        <repository>
            <id>jitpack</id>
            <url>https://jitpack.io</url>
        </repository>
    </repositories>
    
    ```
    
    ---
    
    ## **I. `<modules>` (For Multi-Module Projects)**
    
    Used in parent POM.
    
    ```xml
    <modules>
        <module>service-api</module>
        <module>service-impl</module>
    </modules>
    
    ```
    
    ---
    
    ## **J. `<profiles>`**
    
    Used for environment-based configurations (dev, test, prod).
    
    ```xml
    <profiles>
    
        <profile>
            <id>dev</id>
            <properties>
                <db.url>jdbc:mysql://localhost/dev</db.url>
            </properties>
        </profile>
    
        <profile>
            <id>prod</id>
            <properties>
                <db.url>jdbc:mysql://prod.company.com/prod</db.url>
            </properties>
        </profile>
    
    </profiles>
    
    ```
    
    Execute using:
    
    ```
    mvn clean install -Pdev
    
    ```
    
    ---
    
    ## **4. Full Example POM (Well-Structured)**
    
    ```xml
    <project xmlns="http://maven.apache.org/POM/4.0.0"
             xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
             xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
             http://maven.apache.org/xsd/maven-4.0.0.xsd">
    
        <modelVersion>4.0.0</modelVersion>
    
        <groupId>com.techacademy</groupId>
        <artifactId>employee-service</artifactId>
        <version>1.0.0</version>
        <packaging>jar</packaging>
    
        <name>Employee Service</name>
    
        <properties>
            <java.version>17</java.version>
            <spring.version>6.0.0</spring.version>
        </properties>
    
        <dependencies>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-context</artifactId>
                <version>${spring.version}</version>
            </dependency>
        </dependencies>
    
        <build>
            <plugins>
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-compiler-plugin</artifactId>
                    <version>3.10.1</version>
                    <configuration>
                        <source>${java.version}</source>
                        <target>${java.version}</target>
                    </configuration>
                </plugin>
            </plugins>
        </build>
    
    </project>
    
    ```
    
    ---
    
    ## **5. Real-world Analogy**
    
    | POM Concept | Analogy |
    | --- | --- |
    | POM File | Blueprint of a building |
    | Dependencies | Materials needed |
    | Plugins | Construction tools |
    | Profiles | Different designs for dev/prod |

    ## **6. Important Points**
    
    - POM is the heart of any Maven project
    - Defines dependencies, plugins, build settings
    - Supports inheritance and multi-module structure
    - Promotes consistency through convention
    - Enables version management and reusable configurations

    ## **7. Summary**
    
    | Section | Purpose |

- Build lifecycle
    
    # **Maven Build Lifecycle**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    The **Maven Build Lifecycle** is a sequence of well-defined **phases** that govern how a Maven project is built.
    
    Each phase represents a stage in the build process, such as compiling code, running tests, packaging artifacts, installing them locally, or deploying them to a remote repository.
    
    ### **Purpose**
    
    - Standardize the build process
    - Provide predictable, repeatable builds
    - Automate complex tasks via lifecycle phases
    - Allow plugins to attach goals at specific phases
    - Ensure consistent build behavior across projects
    
    ---
    
    ## **2. Maven’s Three Built-in Lifecycles**
    
    Maven provides **3 default lifecycles**:
    
    1. **clean** – Cleans previous build outputs
    2. **default (build)** – Handles full project build
    3. **site** – Generates project site/documentation
    
    Each lifecycle contains multiple **sequential phases**.
    
    ---
    
    # **3. CLEAN LIFECYCLE**
    
    ### **Phases**
    
    1. **pre-clean**
    2. **clean**
    3. **post-clean**
    
    ### **Primary Phase**
    
    - **clean** → Deletes the `target/` directory and all build artifacts.
    
    ### **Example**
    
    ```bash
    mvn clean
    
    ```
    
    ---
    
    # **4. DEFAULT (BUILD) LIFECYCLE**
    
    This is the most important lifecycle.
    
    It contains **23 phases**, but the main ones are:
    
    ---
    
    ## **Core Phases (Commonly Used)**
    
    ### **1. validate**
    
    - Ensures the project is correct and all required information is available.
    - Checks `pom.xml` structure.
    
    ---
    
    ### **2. compile**
    
    - Compiles source code from `src/main/java`
    - Outputs `.class` files to `target/classes`
    
    Equivalent plugin goal:
    
    ```bash
    mvn compiler:compile
    
    ```
    
    ---
    
    ### **3. test**
    
    - Compiles and runs tests in `src/test/java`
    - Uses Surefire Plugin by default
    - Does **not** package or deploy anything yet
    
    ---
    
    ### **4. package**
    
    - Creates a distributable artifact:
        - JAR (default)
        - WAR (for web apps)
        - EAR (enterprise apps)
    
    Example:
    
    ```bash
    mvn package
    
    ```
    
    ---
    
    ### **5. verify**
    
    - Runs additional checks on packaged artifact
    - Could be integration tests, checksum validation, etc.
    
    ---
    
    ### **6. install**
    
    - Installs the packaged artifact into the **local Maven repository** (`~/.m2/repository`)
    - Makes it available for other projects on your machine
    
    ```bash
    mvn install
    
    ```
    
    ---
    
    ### **7. deploy**
    
    - Deploys the artifact to a **remote repository** (Nexus, Artifactory, etc.)
    - Used in CI/CD pipelines for team-level sharing
    
    ```bash
    mvn deploy
    
    ```
    
    ---
    
    # **5. Complete List of Default Lifecycle Phases**
    
    Below is the full ordered list (for completeness):
    
    ```
    validate
    initialize
    generate-sources
    process-sources
    generate-resources
    process-resources
    compile
    process-classes
    generate-test-sources
    process-test-sources
    generate-test-resources
    process-test-resources
    test-compile
    process-test-classes
    test
    prepare-package
    package
    pre-integration-test
    integration-test
    post-integration-test
    verify
    install
    deploy
    
    ```
    
    Each phase triggers all previous phases before executing.
    
    For example:
    
    ```
    mvn package → runs everything up to package
    mvn install → runs everything up to install
    
    ```
    
    ---
    
    # **6. SITE LIFECYCLE**
    
    ### **Phases**
    
    1. **pre-site**
    2. **site** – generates documentation and reports
    3. **post-site**
    4. **site-deploy** – deploys site to remote server
    
    ### Example
    
    ```bash
    mvn site
    
    ```
    
    ---
    
    # **7. How Plugins Attach to Phases**
    
    Plugins define **goals**, which can be bound to lifecycle phases.
    
    ### Example: Binding compiler plugin
    
    ```xml
    <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-compiler-plugin</artifactId>
        <version>3.10.1</version>
        <configuration>
            <source>17</source>
            <target>17</target>
        </configuration>
    </plugin>
    
    ```
    
    ### Example: Surefire plugin executing tests during `test` phase
    
    ```bash
    mvn test
    
    ```
    
    ---
    
    # **8. Execution Behavior**
    
    ### **Sequential Execution**
    
    If you run:
    
    ```
    mvn install
    
    ```
    
    Maven executes in order:
    
    ```
    validate → compile → test → package → verify → install
    
    ```
    
    ### **Phase Skipping via Direct Goal Execution**
    
    Running:
    
    ```
    mvn compiler:compile
    
    ```
    
    Skips the lifecycle and runs only the goal.
    
    ---
    
    # **9. Real-world Analogy**
    
    | Maven Lifecycle | Analogy |
    | --- | --- |
    | validate | Check building plan |
    | compile | Cut building materials |
    | test | Inspect materials |
    | package | Assemble final parts |
    | install | Store in local warehouse |
    | deploy | Send to client warehouse |
    
    ---
    
    # **10. Important Points**
    
    - Lifecycles standardize all Maven builds
    - Each phase runs in order
    - Plugins attach to phases to execute tasks
    - You rarely run individual plugin goals manually
    - `install` places jar/war into local repo
    - `deploy` publishes artifact to remote repo
    - Clean lifecycle is separate from build lifecycle
    
    ---
    
    # **11. Summary**
    
    | Lifecycle | Purpose | Key Phases |
    | --- | --- | --- |
    | **clean** | Remove previous builds | clean |
    | **default** | Build & manage project | validate → deploy |
    | **site** | Generate documentation | site → site-deploy |
    
    **In summary:**
    
    The Maven Build Lifecycle defines how Maven builds a project, offering standardized phases that plugins hook into. It ensures reproducible, consistent builds, making Maven a powerful tool for Java development and CI/CD pipelines.
    
- Dependencies and scopes
    
    # **Dependencies and Scopes (Maven)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    In Maven, **dependencies** are external libraries your project needs, declared inside the `<dependencies>` section of `pom.xml`.
    
    Each dependency may have a **scope**, which controls:
    
    - **Where** the dependency is available (compile, test, runtime)
    - **Whether** it gets packaged into your artifact
    - **Which classpath** it is included in
    
    Scopes allow precise control of dependency usage throughout the build lifecycle.
    
    ---
    
    ## **2. Adding a Dependency (Basic Structure)**
    
    ```xml
    <dependency>
        <groupId>org.slf4j</groupId>
        <artifactId>slf4j-api</artifactId>
        <version>2.0.9</version>
        <scope>compile</scope>
    </dependency>
    
    ```
    
    ---
    
    # **3. Maven Dependency Scopes (All Six)**
    
    Maven provides **6 main scopes**:
    
    1. **compile** (default)
    2. **provided**
    3. **runtime**
    4. **test**
    5. **system**
    6. **import** (for `dependencyManagement`)
    
    ---
    
    # **4. Scope: compile (DEFAULT)**
    
    ### **Usage**
    
    - Standard libraries needed during *compile, test, run*.
    
    ### **Classpath availability**
    
    | Phase | Available? |
    | --- | --- |
    | Compile | ✔ |
    | Test | ✔ |
    | Runtime | ✔ |
    | Packaging | ✔ |
    
    ### **Packaged into JAR/WAR?**
    
    ✔ Yes
    
    ### **Example**
    
    ```xml
    <dependency>
        <groupId>com.google.code.gson</groupId>
        <artifactId>gson</artifactId>
        <version>2.10.1</version>
    </dependency>
    
    ```
    
    ---
    
    # **5. Scope: provided**
    
    ### **Usage**
    
    Dependencies needed for **compilation**, but **provided at runtime by the container** (e.g., Tomcat, JBoss, Servlet container).
    
    ### **Classpath availability**
    
    | Phase | Available? |
    | --- | --- |
    | Compile | ✔ |
    | Test | ✔ |
    | Runtime | ✘ |
    | Packaging | ✘ |
    
    ### **Packaged into JAR/WAR?**
    
    ✘ No (container supplies it)
    
    ### **Examples**
    
    ```xml
    <dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>javax.servlet-api</artifactId>
        <version>4.0.1</version>
        <scope>provided</scope>
    </dependency>
    
    ```
    
    Used for:
    
    - Servlet API
    - JSP API
    - Java EE APIs
    - Spring Boot starter (embedded Tomcat auto-provides Servlet API)
    
    ---
    
    # **6. Scope: runtime**
    
    ### **Usage**
    
    Libraries not needed for compilation but required during execution.
    
    Examples:
    
    - JDBC drivers
    - Logging backends
    - Database connectors
    
    ### **Classpath availability**
    
    | Phase | Available? |
    | --- | --- |
    | Compile | ✘ |
    | Test | ✔ |
    | Runtime | ✔ |
    
    ### **Packaged into JAR/WAR?**
    
    ✔ Yes
    
    ### **Example**
    
    ```xml
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>8.0.33</version>
        <scope>runtime</scope>
    </dependency>
    
    ```
    
    ---
    
    # **7. Scope: test**
    
    ### **Usage**
    
    Dependencies required only for testing.
    
    ### **Classpath availability**
    
    | Phase | Available? |
    | --- | --- |
    | Compile | ✘ |
    | Test | ✔ |
    | Runtime | ✘ |
    
    ### **Packaged into JAR/WAR?**
    
    ✘ No
    
    ### **Example**
    
    ```xml
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter-engine</artifactId>
        <version>5.9.3</version>
        <scope>test</scope>
    </dependency>
    
    ```
    
    Used for:
    
    - JUnit
    - Mockito
    - Testcontainers
    
    ---
    
    # **8. Scope: system** (RARE & NOT RECOMMENDED)
    
    ### **Usage**
    
    Similar to `provided` but requires specifying the **JAR file’s absolute path**.
    
    ### **Classpath availability**
    
    | Phase | Available? |
    | --- | --- |
    | Compile | ✔ |
    | Runtime | ✘ |
    | Test | ✔ |
    
    ### **Packaged?**
    
    ✘ No
    
    ### **Example**
    
    ```xml
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>my-lib</artifactId>
        <version>1.0</version>
        <scope>system</scope>
        <systemPath>${project.basedir}/lib/mylib.jar</systemPath>
    </dependency>
    
    ```
    
    **Issues**
    
    - Breaks portability
    - Hardcodes local file paths
    - Better to use local or remote repository instead
    
    ---
    
    # **9. Scope: import** (Special Scope)
    
    Used only inside `<dependencyManagement>` to **import BOMs** (Bill of Materials).
    
    ### **Example: Spring Boot BOM**
    
    ```xml
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>3.1.0</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    ```
    
    This allows version management for:
    
    - Spring Core
    - Spring MVC
    - Spring JPA
    - Jackson
    - Tomcat
    - Hibernate
    
    ---
    
    # **10. Transitive Dependencies**
    
    If A depends on B, and B depends on C → A automatically gets C.
    
    To exclude unwanted transitive dependency:
    
    ```xml
    <dependency>
        <groupId>org.hibernate</groupId>
        <artifactId>hibernate-core</artifactId>
        <exclusions>
            <exclusion>
                <groupId>org.slf4j</groupId>
                <artifactId>slf4j-log4j12</artifactId>
            </exclusion>
        </exclusions>
    </dependency>
    
    ```
    
    ---
    
    # **11. How Scopes Influence Transitive Dependencies**
    
    | Scope | Passed to child dependencies? |
    | --- | --- |
    | `compile` | ✔ propagated |
    | `provided` | ✘ not propagated |
    | `runtime` | ✔ propagated |
    | `test` | ✘ not propagated |
    | `system` | ✘ not propagated |
    | `import` | N/A (used only in DM section) |
    
    ---
    
    # **12. Real-world Analogy**
    
    | Dependency Scope | Analogy |
    | --- | --- |
    | compile | Tools always needed (hammer, screwdriver) |
    | provided | Tools provided by job site (drill on-site) |
    | runtime | Tools needed only during operation (soldering iron) |
    | test | Tools used only for inspection (voltmeter) |
    | system | Bringing your own non-standard tool |
    | import | Global tool configuration for all projects |
    
    ---
    
    # **13. Important Points**
    
    - **compile** is default scope
    - Use **provided** when container supplies library (servlet & JSP APIs)
    - Use **runtime** for connectors and drivers
    - **test** dependencies never go in the final artifact
    - Avoid `system` scope
    - `import` is essential for BOM-based projects like Spring Boot
    - Dependency scopes control inclusion across lifecycle phases
    
    ---
    
    # **14. Summary Table**
    
    | Scope | Compile | Test | Runtime | Packaged | Transitive | Typical Use |
    | --- | --- | --- | --- | --- | --- | --- |
    | compile | ✔ | ✔ | ✔ | ✔ | ✔ | Spring, JPA, Utils |
    | provided | ✔ | ✔ | ✘ | ✘ | ✘ | Servlet API, Java EE |
    | runtime | ✘ | ✔ | ✔ | ✔ | ✔ | JDBC Drivers |
    | test | ✘ | ✔ | ✘ | ✘ | ✘ | JUnit, Mockito |
    | system | ✔ | ✔ | ✘ | ✘ | ✘ | Local external JAR |
    | import | - | - | - | - | - | BOM version control |
    
    ---
    
    **Dependencies and scopes** are core to Maven’s power, allowing precise control of library availability across build and runtime environments, ensuring clean and efficient project setup.

    # **Maven Plugins**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    **Maven Plugins** are modules that perform specific tasks during a Maven build.
    
    Every phase of the Maven lifecycle is executed through plugins.
    
    Plugins define **goals**, and these goals execute the actual actions during build phases such as:
    
    - Compiling source code
    - Running tests
    - Packaging JAR/WAR
    - Creating documentation
    - Deploying artifacts
    
    ### **Purpose**
    
    - Extend Maven functionality
    - Bind specific operations to lifecycle phases
    - Automate repeated development tasks
    - Customize build behavior
    
    ---
    
    ## **2. How Plugins Work**
    
    Each plugin contains one or more **goals**.
    
    A goal represents a specific task.
    
    Example:
    
    ```
    maven-compiler-plugin:compile
    
    ```
    
    Plugins are bound to phases via:
    
    - Default lifecycle mappings
    - Explicit configuration in `pom.xml`
    
    ---
    
    ## **3. Basic Plugin Structure**
    
    ```xml
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.10.1</version>
    
                <configuration>
                    <source>17</source>
                    <target>17</target>
                </configuration>
            </plugin>
        </plugins>
    </build>
    
    ```
    
    ---
    
    # **4. Categories of Maven Plugins**
    
    Maven has two main plugin groups:
    
    ---
    
    ## **A. Build Plugins**
    
    Executed during the **build lifecycle phases**.
    
    Examples:
    
    - `maven-compiler-plugin`
    - `maven-surefire-plugin`
    - `maven-jar-plugin`
    - `maven-war-plugin`
    - `maven-install-plugin`
    - `maven-deploy-plugin`
    
    ---
    
    ## **B. Reporting Plugins**
    
    Used during **site** lifecycle to generate reports.
    
    Examples:
    
    - `maven-project-info-reports-plugin`
    - `maven-surefire-report-plugin`
    - `maven-javadoc-plugin`
    
    ---
    
    # **5. Essential Maven Plugins (Most Used)**
    
    ---

- Multi-module structure (simple)
    
    # **Multi-Module Maven Structure (Simple Overview)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    A **multi-module Maven project** is a structure where a single **parent project** manages one or more **child modules**.
    
    The parent defines **shared configuration**, and each module represents a **separate Maven project**, usually compiled and built together.
    
    ### **Purpose**
    
    - Centralized dependency and plugin management
    - Shared versioning and build configuration
    - Logical separation of layers (API, service, persistence, web)
    - Easier maintenance and cleaner architecture
    
    ---
    
    ## **2. Basic Multi-Module Structure**
    
    ```
    root-project/
    │
    ├── pom.xml          (parent POM)
    │
    ├── module-a/
    │   └── pom.xml
    │
    └── module-b/
        └── pom.xml
    
    ```
    
    ---
    
    ## **3. Parent POM Structure**
    
    Parent POM uses:
    
    ```xml
    <packaging>pom</packaging>
    
    ```
    
    ### **Example: parent pom.xml**
    
    ```xml
    <project>
        <modelVersion>4.0.0</modelVersion>
    
        <groupId>com.example</groupId>
        <artifactId>parent-project</artifactId>
        <version>1.0.0</version>
        <packaging>pom</packaging>
    
        <modules>
            <module>module-a</module>
            <module>module-b</module>
        </modules>
    
        <dependencyManagement>
            <dependencies>
                <dependency>
                    <groupId>org.springframework</groupId>
                    <artifactId>spring-context</artifactId>
                    <version>6.0.0</version>
                </dependency>
            </dependencies>
        </dependencyManagement>
    
        <properties>
            <java.version>17</java.version>
        </properties>
    </project>
    
    ```
    
    Key points:
    
    - Declares modules
    - Shares versions and plugin configs
    - Doesn’t generate .jar — it's a parent aggregator
    
    ---
    
    ## **4. Child Module POM Structure**
    
    Child modules **inherit** from the parent.
    
    ### **module-a/pom.xml**
    
    ```xml
    <project>
        <parent>
            <groupId>com.example</groupId>
            <artifactId>parent-project</artifactId>
            <version>1.0.0</version>
        </parent>
    
        <artifactId>module-a</artifactId>
    
        <dependencies>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-context</artifactId>
            </dependency>
        </dependencies>
    </project>
    
    ```
    
    ### **module-b/pom.xml**
    
    ```xml
    <project>
        <parent>
            <groupId>com.example</groupId>
            <artifactId>parent-project</artifactId>
            <version>1.0.0</version>
        </parent>
    
        <artifactId>module-b</artifactId>
    
        <dependencies>
            <dependency>
                <groupId>com.example</groupId>
                <artifactId>module-a</artifactId>
            </dependency>
        </dependencies>
    </project>
    
    ```
    
    ---
    
    ## **5. How Modules Interact**
    
    ### **module-b depends on module-a**
    
    Maven automatically builds in the correct order:
    
    ```
    module-a → module-b
    
    ```
    
    ---
    
    ## **6. Build Commands**
    
    Run **entire project**:
    
    ```
    mvn clean install
    
    ```
    
    Build only one module:
    
    ```
    mvn -pl module-a clean install
    
    ```
    
    ---
    
    ## **7. Common Use Cases**
    
    - API Module
    - Service Module
    - Web Module
    - Persistence Module
    - Shared Utilities
    
    Example structure:
    
    ```
    project/
    ├── api/
    ├── service/
    └── web/
    
    ```
    
    ---
    
    ## **8. Real-world Analogy**
    
    | Component | Analogy |
    | --- | --- |
    | Parent Project | Master blueprint |
    | Child Modules | Independent buildings sharing specifications |
    | Dependency Management | Common material list |
    
    ---
    
    ## **9. Important Points**
    
    - Parent POM uses `<packaging>pom</packaging>`
    - Child modules inherit configuration
    - Shared dependencies managed in `<dependencyManagement>`
    - Modules are built in order based on dependency graph
    - Encourages modular architecture
    
    ---
    
    ## **10. Summary**
    
    | Item | Meaning |
    | --- | --- |
    | Parent POM | Aggregates and manages modules |
    | Module | Independent Maven project |
    | `<modules>` | Declares child modules |
    | Inheritance | Children reuse parent config |
    
    **A multi-module project** provides a clean, maintainable layout where multiple Maven subprojects share configuration but remain modular and independent.
    

### **9.2 Deployable Artifacts**

- JAR vs WAR
    
    # **JAR vs WAR**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **JAR (Java ARchive)**
    
    A **JAR** file is a compressed archive that contains:
    
    - Compiled `.class` files
    - Resources (properties, XML, images)
    - Optional `MANIFEST.MF`
    - Optional dependencies (if using shaded/FAT JAR)
    
    Used for:
    
    - Standalone Java applications
    - Libraries
    - Microservices (Spring Boot executable JARs)
    
    ---
    
    ### **WAR (Web Application ARchive)**
    
    A **WAR** file is a special archive format structured to package **web applications** containing:
    
    - Servlets
    - JSPs
    - Web resources
    - `WEB-INF` directory
    - Deployment descriptor (`web.xml`)
    - Libraries packaged under `WEB-INF/lib`
    
    WAR files are deployed to an **external servlet container** such as:
    
    - Apache Tomcat
    - Jetty
    - WildFly / JBoss
    
    ---
    
    # **2. Packaging Type in Maven**
    
    ### JAR
    
    ```xml
    <packaging>jar</packaging>
    
    ```
    
    ### WAR
    
    ```xml
    <packaging>war</packaging>
    
    ```
    
    ---
    
    # **3. Directory Structure Differences**
    
    ---
    
    ## **JAR Structure**
    
    ```
    myapp.jar
     ├── META-INF/
     │     └── MANIFEST.MF
     ├── com/example/App.class
     └── application.properties
    
    ```
    
    Used like:
    
    ```
    java -jar myapp.jar
    
    ```
    
    ---
    
    ## **WAR Structure**
    
    ```
    myapp.war
     ├── index.jsp
     ├── images/
     ├── css/
     ├── js/
     ├── WEB-INF/
     │    ├── web.xml
     │    ├── classes/  (compiled classes)
     │    └── lib/      (JAR dependencies)
    
    ```
    
    Deployed to:
    
    ```
    tomcat/webapps/
    
    ```
    
    ---
    
    # **4. JAR vs WAR (Side-by-Side Comparison)**
    
    | Feature | JAR | WAR |
    | --- | --- | --- |
    | Full form | Java ARchive | Web Application ARchive |
    | Purpose | General-purpose libraries or standalone applications | Web applications (Servlet/JSP/Spring MVC) |
    | Contains | Classes, resources | Web assets, web.xml, classes, libs |
    | Deployment | Run via JVM (`java -jar`) | Deploy to container (Tomcat) |
    | Web server needed? | No (embedded server optional) | Yes, must deploy to external server |
    | Used in Spring Boot | Yes (default executable JAR) | Rare (custom setup) |
    | Structure | Simple | Special directory structure |
    | Packaging | Lightweight | Heavier |
    | Includes dependencies? | Only if fat/shaded JAR | Yes (under `WEB-INF/lib`) |
    | Suitable for Microservices | ✔ | ✘ |
    | Suitable for Monolithic Java EE apps | ✘ | ✔ |
    
    ---
    
    # **5. WAR and Java EE**
    
    WAR is designed for:
    
    - Servlet-based applications
    - JSP pages
    - Filters
    - Listeners
    - Web deployment descriptor
    
    Java EE servers manage the lifecycles of WAR deployments.
    
    ---
    
    # **6. JAR in Modern Applications (Spring Boot)**
    
    Spring Boot uses **embedded servers** (Tomcat/Jetty) to package everything into one runnable JAR:
    
    ```
    myapp.jar
    
    ```
    
    With entry point in MANIFEST:
    
    ```
    Main-Class: org.springframework.boot.loader.JarLauncher
    
    ```
    
    Advantages:
    
    - No need for external container
    - Easy deployment (just run the JAR)
    - Perfect for cloud and microservices
    
    ---
    
    # **7. WAR in Legacy Applications**
    
    Used when:
    
    - Application must run inside a centralized application server
    - Organization mandates WAR deployment strategy
    - There are legacy servlets/JSP-based architectures
    
    ---
    
    # **8. When to Use JAR vs WAR**
    
    ### **Use JAR When:**
    
    - Building Spring Boot microservices
    - Need standalone application
    - Want simpler deployment
    - Using REST APIs / modern applications
    - Using embedded server
    
    ### **Use WAR When:**
    
    - Deploying into enterprise WebServers
    - Using classic Java EE architecture
    - Have multiple WARs inside the same server
    - Need server-managed features (JNDI, JMS, clustering)
    
    ---
    
    # **9. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | JAR | A standalone program that you can run directly |
    | WAR | A web application that must be placed inside a web server |
    
    ---
    
    # **10. Important Points**
    
    - JAR is lightweight; WAR is web-specific.
    - JAR can be executable; WAR cannot (without server).
    - WAR contains structured web files and metadata.
    - JAR-based deployment is preferred in cloud-native architecture.
    - Maven packaging type decides output format.
    
    ---
    
    # **11. Summary**
    
    | Item | JAR | WAR |
    | --- | --- | --- |
    | Application Type | Standalone / Microservice | Web app for server |
    | Server Included | Optional (embedded) | No |
    | Structure | Simple | Web-specific |
    | Deployment | Directly run | Deploy to container |
    
    **JAR vs WAR**:
    
    JAR is best for modern, standalone, cloud-native Java apps, while WAR suits classical server-based enterprise applications with servlet containers.
    
- Fat JAR creation
    
    # **Fat JAR Creation (Maven)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    A **Fat JAR** (also called **Uber JAR**, **Shaded JAR**, or **Self-Contained JAR**) is a single executable JAR file that contains:
    
    - Your application’s compiled classes
    - All dependency JARs merged inside
    
    This enables running your application with **one single JAR**, without needing external libraries on the classpath.
    
    ```
    java -jar app-fat.jar
    
    ```
    
    ### **Purpose**
    
    - Simplifies deployment
    - Ideal for microservices, CLI tools, and standalone apps
    - Ensures all dependencies ship together
    - Commonly used in cloud-native and container-based deployments
    
    ---
    
    ## **2. Why Fat JARs Are Needed**
    
    Normal JAR:
    
    ```
    myapp.jar
    
    ```
    
    ❌ Cannot run alone
    
    ❌ Missing dependencies
    
    ❌ Must manage classpath manually
    
    Fat JAR:
    
    ```
    myapp-fat.jar
    
    ```
    
    ✔ Contains everything
    
    ✔ Single file
    
    ✔ `java -jar` works directly
    
    ---
    
    ## **3. Tools for Creating Fat JAR in Maven**
    
    There are **three common ways**:
    
    1. **Maven Shade Plugin** (Most flexible)
    2. **Spring Boot Maven Plugin** (For Spring Boot apps)
    3. **Maven Assembly Plugin** (Older but still used)
    
    ---
    
    # **4. Fat JAR Using Maven Shade Plugin (Recommended)**
    
    ### **Add plugin to `pom.xml`:**
    
    ```xml
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-shade-plugin</artifactId>
                <version>3.3.0</version>
    
                <executions>
                    <execution>
                        <phase>package</phase>
                        <goals>
                            <goal>shade</goal>
                        </goals>
                        <configuration>
                            <transformers>
                                <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                                    <mainClass>com.example.Main</mainClass>
                                </transformer>
                            </transformers>
                        </configuration>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
    
    ```
    
    ### **Command**
    
    ```
    mvn clean package
    
    ```
    
    ### **Result**
    
    ```
    target/myapp-1.0-SNAPSHOT-shaded.jar
    
    ```
    
    ### **Key Features of Shade Plugin**
    
    - Merges dependencies
    - Handles resource conflicts
    - Allows package relocation (solve "jar hell")
    - Customizable manifest
    
    ---
    
    # **5. Fat JAR Using Spring Boot Plugin (If using Spring Boot)**
    
    Spring Boot automatically creates executable JARs with dependencies.
    
    ### **Add plugin**
    
    ```xml
    <build>
        <plugins>
            <plugin>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-maven-plugin</artifactId>
            </plugin>
        </plugins>
    </build>
    
    ```
    
    ### **Command**
    
    ```
    mvn clean package
    
    ```
    
    ### **Output**
    
    ```
    target/myapp-0.0.1-SNAPSHOT.jar  (Fat JAR)
    
    ```
    
    Spring Boot uses its own classloader structure (`BOOT-INF/`) for dependencies.
    
    ---
    
    # **6. Fat JAR Using Assembly Plugin (Older Approach)**
    
    ### **Add plugin**
    
    ```xml
    <build>
        <plugins>
            <plugin>
                <artifactId>maven-assembly-plugin</artifactId>
                <configuration>
                    <archive>
                        <manifest>
                            <mainClass>com.example.Main</mainClass>
                        </manifest>
                    </archive>
                    <descriptorRefs>
                        <descriptorRef>jar-with-dependencies</descriptorRef>
                    </descriptorRefs>
                </configuration>
                <executions>
                    <execution>
                        <phase>package</phase>
                        <goals>
                            <goal>single</goal>
                        </goals>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
    
    ```
    
    ### **Command**
    
    ```
    mvn clean package
    
    ```
    
    ### **Output**
    
    ```
    target/myapp-jar-with-dependencies.jar
    
    ```
    
    While widely used, it has downsides like possible resource merging issues.
    
    ---
    
    # **7. Running the Fat JAR**
    
    ```
    java -jar target/myapp-fat.jar
    
    ```
    
    If you set `mainClass` correctly, the app runs immediately.
    
    ---
    
    # **8. Structure of a Fat JAR**
    
    ### **Shade Plugin Output**
    
    ```
    META-INF/
    com/example/App.class
    org/apache/...   (from dependencies)
    
    ```
    
    ### **Spring Boot Output**
    
    ```
    BOOT-INF/classes/
    BOOT-INF/lib/    ← All dependencies
    META-INF/
    
    ```
    
    ---
    
    # **9. When to Use Fat JARs**
    
    ### Ideal for:
    
    - CLI tools
    - Microservices
    - Spring Boot apps
    - Deploying to cloud / Docker
    - Shipping standalone utilities
    
    ### Not ideal for:
    
    - Large monoliths requiring shared dependencies
    - Application servers where dependencies are provided by container

    # **10. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Normal JAR | A car without fuel; you must bring fuel jars separately |
    | Fat JAR | Car with fuel tank full — everything ready to go |

    # **11. Important Points**
    
    - Shade plugin is most flexible and modern
    - Spring Boot plugin automatically builds fat JARs
    - Assembly plugin uses older “jar-with-dependencies” style
    - Ensure `Main-Class` is set in manifest
    - Useful for simplified deployments
    
    ---
    
    # **12. Summary**
    
    | Method | Best For | Pros |
    | --- | --- | --- |
    | **Shade Plugin** | Any Java app | Flexible, relocations, resource fixes |
    | **Spring Boot Plugin** | Spring Boot apps | Auto fat JAR, simple |
    | **Assembly Plugin** | Legacy builds | Simple descriptor |
    
    **Fat JARs** provide a single-file solution for packaging your entire application with all required dependencies, making deployment easy and reliable, especially in cloud-native environments.

    # **Environment Configurations (Environment Configs)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    ### **Definition**
    
    **Environment configurations** refer to application settings that vary by environment such as:
    
    - **dev** (local development)
    - **test / QA**
    - **staging / pre-prod**
    - **production**
    
    These configurations include:
    
    - Database credentials
    - API URLs
    - Logging levels
    - Feature flags
    - Security settings
    - External service endpoints
    
    Environment config management ensures an application behaves correctly in each environment **without changing code**.
    
    ---
    
    ## **2. Why Environment Configs Are Needed**
    
    - Different environments require different settings
    - Avoid hardcoding sensitive values
    - Enable clean deployments
    - Improve maintainability
    - Allow separation between dev/test/production behaviors
    - Facilitate CI/CD pipelines
    
    ---
    
    ## **3. Types of Environment Configurations**
    
    ---
    
    ## **A. JVM System Properties**
    
    Pass environment variables at runtime:
    
    ### Run:
    
    ```
    java -Denv=prod -jar app.jar
    
    ```
    
    ### Inside Java:
    
    ```java
    String env = System.getProperty("env");
    
    ```
    
    Good for simple settings.
    
    ---
    
    ## **B. Environment Variables (OS-Level)**
    
    Set variables at OS or container level:
    
    ### Linux
    
    ```bash
    export DB_USER=root
    export DB_PASS=secret
    
    ```
    
    ### Java Access:
    
    ```java
    System.getenv("DB_USER");
    
    ```
    
    Used heavily in Docker + Kubernetes.
    
    ---
    
    ## **C. Properties Files**
    
    Different `.properties` files for each environment:
    
    ```
    application-dev.properties
    application-test.properties
    application-prod.properties
    
    ```
    
    ### Example:
    
    ```
    db.url=jdbc:mysql://localhost/devdb
    logging.level=DEBUG
    
    ```
    
    ---
    
    ## **D. YAML Configurations (Spring Boot)**
    
    Spring Boot supports environment-specific YAML:
    
    ```
    application.yml
    application-dev.yml
    application-test.yml
    application-prod.yml
    
    ```
    
    ### Profile selection:
    
    ```
    spring.profiles.active=dev
    
    ```
    
    ---
    
    ## **E. Maven Profiles (Build-Time Config)**
    
    Define environment-based builds in `pom.xml`:
    
    ```xml
    <profiles>
        <profile>
            <id>dev</id>
            <properties>
                <config.env>dev</config.env>
            </properties>
        </profile>
    
        <profile>
            <id>prod</id>
            <properties>
                <config.env>prod</config.env>
            </properties>
        </profile>
    </profiles>
    
    ```
    
    ### Run:
    
    ```
    mvn clean package -Pprod
    
    ```
    
    Used mainly for build differences (not runtime).
    
    ---
    
    ## **F. Externalized Config (Spring Cloud Config)**
    
    Centralized repository-based configuration:
    
    - Git config storage
    - Dynamic refresh
    - Secure secrets
    
    Useful for microservices.
    
    ---
    
    ## **G. Vault / Secret Managers**
    
    Secure management for sensitive configurations:
    
    - HashiCorp Vault
    - AWS Secrets Manager
    - Azure Key Vault
    - GCP Secret Manager
    
    Used for:
    
    - API keys
    - DB passwords
    - Certificates
    
    ---
    
    # **4. Environment Config in Spring Boot (Most Common)**
    
    ---
    
    ## **A. application.properties / application.yml**
    
    Base configuration:
    
    ```
    server.port=8080
    
    ```
    
    ---
    
    ## **B. Profile-specific files**
    
    ```
    application-dev.properties
    application-prod.properties
    
    ```
    
    ### Example:
    
    **Dev**
    
    ```
    server.port=8081
    logging.level.root=DEBUG
    db.url=jdbc:mysql://localhost/devdb
    
    ```
    
    **Prod**
    
    ```
    server.port=8080
    logging.level.root=INFO
    db.url=jdbc:mysql://prod.company.com/db
    
    ```
    
    Activate profiles using:
    
    ```
    spring.profiles.active=dev
    
    ```
    
    ---
    
    ## **C. @Value and @ConfigurationProperties**
    
    Inject environment values:
    
    ```java
    @Value("${db.url}")
    private String url;
    
    ```
    
    Or bind structured configs:
    
    ```java
    @ConfigurationProperties(prefix = "app")
    public class AppConfig {
        private String name;
        private String version;
    }
    
    ```
    
    ---
    
    ## **D. Command-Line Arg Overrides**
    
    ```
    java -jar app.jar --server.port=9090 --spring.profiles.active=prod
    
    ```
    
    Overrides everything else.
    
    ---
    
    ## **5. Managing Environments in CI/CD**
    
    ### 1. Separate config for:
    
    - Jenkins
    - GitHub Actions
    - Docker
    - Kubernetes
    
    ### 2. Kubernetes Config Maps
    
    ```
    kind: ConfigMap
    
    ```
    
    ### 3. Kubernetes Secrets
    
    ```
    kind: Secret
    
    ```
    
    These inject environment settings into pods.
    
    ---
    
    ## **6. File Structure Example**
    
    ```
    src/main/resources/
    │
    ├── application.properties
    ├── application-dev.properties
    ├── application-test.properties
    └── application-prod.properties
    
    ```
    
    ---
    
    ## **7. Best Practices**
    
    ### ✔ Externalize All Configurations
    
    No hardcoding in code.
    
    ### ✔ Separate secrets from configs
    
    Use:
    
    - Vault
    - Environment variables
    - Secret Manager
    
    ### ✔ Use profiles for environment differences
    
    Keep dev/test/prod separate.
    
    ### ✔ Avoid environment-specific builds
    
    Prefer *same JAR*, different configs.
    
    ### ✔ Log appropriate levels per environment
    
    Dev → DEBUG
    
    Prod → INFO/WARN
    
    ### ✔ Use YAML for complex configs
    
    Easier to read/structure.
    
    ### ✔ Use placeholders
    
    ```
    db.username=${DB_USER}
    
    ```
    
    ---
    
    ## **8. Real-world Analogy**
    
    | Concept | Analogy |
    | --- | --- |
    | Environment config | Car dashboard settings |
    | Dev mode | Sport mode |
    | Prod mode | Eco mode |
    | Config switching | Changing car mode without modifying engine |
    
    ---
    
    ## **9. Important Points**
    
    - Environment configs allow code to stay constant across environments
    - Profiles allow switching configuration at runtime
    - Sensitive config should never be inside source code
    - Cloud-native apps heavily rely on environment variables
    - Spring Boot offers the richest environment config model
    
    ---
    
    ## **10. Summary**
    
    | Method | Purpose |
    | --- | --- |
    | Properties/YAML | Basic environment configs |
    | Spring Profiles | Environment switching |
    | Environment Variables | Container/cloud settings |
    | Maven Profiles | Build-time configs |
    | External Config Server | Centralized config |
    | Secret Manager | Secure sensitive values |
    
    **Environment Configurations** ensure applications run correctly in different environments while keeping code clean, secure, and environment-independent.
