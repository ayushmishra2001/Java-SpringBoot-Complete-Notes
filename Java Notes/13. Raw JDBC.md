### **13.1 JDBC Architecture**

- Architecture
    
    # **JDBC Architecture**
    
    ---
    
    ## **1. Definition and Explanation**
    
    JDBC (Java Database Connectivity) is a **Java API** that enables Java applications to interact with relational databases.
    
    It provides a standard way to:
    
    - Connect to a database
    - Send SQL queries
    - Retrieve and process results
    - Manage transactions
    
    JDBC acts as a **bridge** between Java code and the underlying database by using database-specific drivers.
    
    The architecture is designed to be flexible, database-independent, and easy to integrate with enterprise applications.
    
    ---
    
    # **2. Components of JDBC Architecture**
    
    JDBC architecture consists of two major layers:
    
    1. **JDBC API Layer** (Java side)
    2. **JDBC Driver Layer** (Database side)
    
    Together, they enable seamless communication between Java applications and databases.
    
    ---
    
    ## **2.1 JDBC API Layer (Java Application Layer)**
    
    This layer contains classes and interfaces provided by **java.sql** and **javax.sql** packages.
    
    Key interfaces/classes include:
    
    - `DriverManager`
    - `Connection`
    - `Statement`
    - `PreparedStatement`
    - `CallableStatement`
    - `ResultSet`
    - `SQLException`
    - `DataSource`
    
    These form the standard API used by developers.
    
    ---
    
    ## **2.2 JDBC Driver Layer**
    
    This layer contains the **database-specific JDBC driver**.
    
    Each database vendor (MySQL, Oracle, PostgreSQL) provides its own driver implementing the JDBC interfaces.
    
    Driver Types:
    
    - **Type 4 (Pure Java Driver)** → Most commonly used today
    - (Type 1/2/3 are legacy and rarely used)
    
    The driver translates Java API calls into database-specific protocol calls.
    
    ---
    
    # **3. JDBC Architecture Diagram (Conceptual)**
    
    ```
    ----------------------------------------
    |       Java Application (Your App)    |
    |--------------------------------------|
    | JDBC API (Connection, Statement...)  |
    ----------------------------------------
    |  JDBC Driver (MySQL/Oracle Driver)   |
    ----------------------------------------
    |       Database (MySQL, Oracle)       |
    ----------------------------------------
    
    ```
    
    ---
    
    # **4. Flow of JDBC Architecture**
    
    ### **Step 1: Load/Re-register Driver**
    
    The driver is loaded using:
    
    ```java
    Class.forName("com.mysql.cj.jdbc.Driver");
    
    ```
    
    Or automatically by Service Provider Mechanism.
    
    ---
    
    ### **Step 2: Establish Connection**
    
    ```java
    Connection con = DriverManager.getConnection(url, user, pass);
    
    ```
    
    DriverManager checks registered drivers and finds the suitable one.
    
    ---
    
    ### **Step 3: Create Statement Object**
    
    ```java
    Statement st = con.createStatement();
    
    ```
    
    Or better:
    
    ```java
    PreparedStatement ps = con.prepareStatement(sql);
    
    ```
    
    ---
    
    ### **Step 4: Execute SQL**
    
    ```java
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    ```
    
    ---
    
    ### **Step 5: Process Results**
    
    ```java
    while (rs.next()) {
        System.out.println(rs.getInt(1) + " " + rs.getString(2));
    }
    
    ```
    
    ---
    
    ### **Step 6: Close Resources**
    
    ```java
    rs.close();
    st.close();
    con.close();
    
    ```
    
    ---
    
    # **5. JDBC Core Interfaces Explained**
    
    ### **A. DriverManager**
    
    - Manages available JDBC drivers.
    - Selects appropriate driver for a given URL.
    - Creates database connections.
    
    ### **B. Connection**
    
    - Represents a session with the database.
    - Controls transactions.
    - Creates statements.
    
    ### **C. Statement**
    
    - Executes static SQL queries.
    
    ### **D. PreparedStatement**
    
    - Precompiled SQL statements.
    - Prevent SQL injection.
    - Faster performance.
    
    ### **E. CallableStatement**
    
    - For executing stored procedures.
    
    ### **F. ResultSet**
    
    - Holds the data returned from queries.
    - Cursor-based navigation.

    # **6. Types of JDBC Architecture**
    
    ### **6.1 Two-Layer Architecture (Direct Connection)**
    
    Application → JDBC Driver → Database
    
    Used when working directly with the database.
    
    ### **6.2 Three-Layer Architecture (Application Server in Between)**
    
    Application → Middleware Server → JDBC Driver → Database
    
    Used in enterprise applications (Spring Boot uses DataSource).

    # **8. Real-world Analogy**
    
    Think of JDBC as a **translator** between two languages:
    
    - Java speaks “Java”.
    - Databases speak “SQL and database protocol”.
    
    JDBC driver translates Java commands into database language and translates results back.

    - Database-independent standard API.

    # **10. Summary**
    
    JDBC Architecture provides a standardized communication bridge between Java applications and relational databases.
    
    It consists of the JDBC API and the database-specific driver layer.
    
    By offering interfaces like Connection, Statement, PreparedStatement, and ResultSet, JDBC enables secure, efficient interaction with databases.
    
    Understanding this architecture is essential for writing reliable data-driven applications and integrating Java with enterprise-level databases.
    
- JDBC drivers & DriverManager
    
    # **JDBC Drivers & DriverManager**
    
    ---
    
    ## **1. JDBC Drivers**
    
    ---
    
    ## **1.1 Definition and Explanation**
    
    A **JDBC Driver** is a software component that enables Java applications to communicate with a specific database.
    
    It implements the JDBC interfaces such as:
    
    - `Driver`
    - `Connection`
    - `Statement`
    - `ResultSet`, etc.
    
    Each database vendor (MySQL, Oracle, PostgreSQL, SQL Server) provides its own driver.
    
    The driver converts **high-level JDBC calls** into **low-level database protocol calls**.
    
    Without a driver, Java cannot understand how to communicate with a database.
    
    ---
    
    ## **1.2 Types of JDBC Drivers**
    
    JDBC drivers are categorized into **four types**.
    
    | --- | --- | --- | --- |
    
    ### **Most Used Today: Type 4 (Pure Java Driver)**
    
    Examples:
    
    - `com.mysql.cj.jdbc.Driver` (MySQL)
    - `oracle.jdbc.driver.OracleDriver` (Oracle)
    - `org.postgresql.Driver` (PostgreSQL)
    - `com.microsoft.sqlserver.jdbc.SQLServerDriver` (SQL Server)
    
    These drivers are platform-independent and do not require native OS libraries.
    
    ---
    
    ## **1.3 Loading the Driver**
    
    ### **Old Approach (before JDBC 4.0)**
    
    ```java
    Class.forName("com.mysql.cj.jdbc.Driver");
    
    ```
    
    ### **Modern Approach (JDBC 4.0 onward)**
    
    No need to load manually because of **Service Provider Mechanism (SPI)**.
    
    Driver is auto-registered when found in classpath.
    
    ---
    
    ## **1.4 How Drivers Work Internally**
    
    1. Java application calls JDBC APIs.
    2. JDBC driver translates calls into database-specific protocol.
    3. Database executes query.
    4. Driver converts database result into Java objects (`ResultSet`).
    5. Driver returns data back to application.
    
    ---
    
    ## **1.5 Example of Using a Driver in Connection**
    
    ```java
    String url = "jdbc:mysql://localhost:3306/test";
    Connection conn = DriverManager.getConnection(url, "root", "root");
    
    ```
    
    The driver matching the URL prefix (`jdbc:mysql:`) will handle the request.
    
    ---
    
    ## **1.6 JVM / Internal Behavior**
    
    - Driver class registers itself using:
    - When the driver JAR loads, this block runs automatically.
    - DriverManager adds the driver to its internal list of drivers.
    - No explicit class loading needed (JDBC 4.0+).

    # **2. DriverManager**
    
    ---
    
    ## **2.1 Definition and Explanation**
    
    `DriverManager` is a **central service class** in JDBC that:
    
    - Registers all available drivers
    - Manages the list of drivers
    - Selects the appropriate driver based on URL
    - Establishes database connections
    
    It acts as a factory for `Connection` objects.
    
    ---
    
    ## **2.2 Role of DriverManager**
    
    - Maintains static list of registered JDBC drivers
    - For every `getConnection()` call:
    
    ---
    
    ## **2.3 Syntax**
    
    ### **Establishing Connection using DriverManager**
    
    ```java
    Connection con = DriverManager.getConnection(
        "jdbc:mysql://localhost:3306/test",
        "root",
        "root"
    );
    
    ```
    
    ---
    
    ## **2.4 Internal Working of DriverManager**
    
    ### **Step-by-step Flow**
    
    1. Driver JAR loads → Driver registers via static block.
    2. Application calls `DriverManager.getConnection(url, user, pass)`.
    3. DriverManager loops through all registered drivers.
    4. Compares the URL prefix:
    5. Finds the matching driver (MySQL driver).
    6. Delegates connection request to it.
    7. Driver creates and returns `Connection` object.
    
    ---
    
    ## **2.5 Methods in DriverManager**
    
    | --- | --- |
    
    ---
    
    ## **2.6 Example: Manual Driver Registration**
    
    ```java
    DriverManager.registerDriver(new com.mysql.cj.jdbc.Driver());
    
    ```
    
    Rarely used today because auto-loading is enabled.
    
    ---
    
    ## **2.7 Why Use DriverManager?**
    
    - Simplifies connection management
    - Automatically detects drivers
    - Provides a unified method to obtain connections
    - Ensures database independence
    - Handles login timeout and error handling
    
    ---
    
    ## **2.8 Real-world Analogy**
    
    ### **JDBC Driver**
    
    Like a translator for a specific foreign language (each database has its translator).
    
    ### **DriverManager**
    
    Like a **hotel reception desk**:
    
    - Keeps track of all translators (drivers).
    - When someone wants communication (database connection), it chooses the right translator based on language (URL).

    # **4. Important Points**
    
    - Type 4 drivers are widely used today.
    - JDBC 4.0+ does not require `Class.forName()` manually.
    - DriverManager uses the JDBC URL to select correct driver.
    - Avoid using DriverManager in production; prefer **DataSource** for pooling.
    - Drivers must be on classpath to be auto-registered.

    # **5. Summary**
    
    JDBC Drivers are database-specific components that enable Java to communicate with relational databases by converting JDBC calls into database commands.
    
    DriverManager is the core JDBC class that maintains a registry of these drivers and selects the appropriate one based on the JDBC URL to create database connections.
    
    Together, JDBC Driver + DriverManager provide the foundation for all database operations in Java applications.
    
- Connection, Statement, PreparedStatement
    
    # **Connection, Statement, PreparedStatement**
    
    ---
    
    ## **1. Connection**
    
    ---
    
    ## **1.1 Definition and Explanation**
    
    `Connection` represents an active link between a Java application and the database.
    
    It is the **gateway** for executing SQL commands, managing transactions, and creating Statement objects.
    
    A `Connection` is created using `DriverManager` or `DataSource`.
    
    It is the most critical object in JDBC because **without a connection, no SQL operation can be performed**.
    
    ---
    
    ## **1.2 Purpose**
    
    - Establish communication with the database.
    - Manage transactions (commit, rollback).
    - Configure connection settings.
    - Create `Statement`, `PreparedStatement`, and `CallableStatement`.
    
    ---
    
    ## **1.3 Syntax**
    
    ```java
    Connection con = DriverManager.getConnection(url, user, pass);
    
    ```
    
    ---
    
    ## **1.4 Common Methods**
    
    | --- | --- |
    
    ---
    
    ## **1.5 Example**
    
    ```java
    Connection con = DriverManager.getConnection(
        "jdbc:mysql://localhost:3306/test", "root", "root"
    );
    
    ```
    
    ---
    
    ---
    
    ## **2. Statement**
    
    ---
    
    ## **2.1 Definition and Explanation**
    
    `Statement` is used to execute **static SQL queries**.
    
    These are SQL statements that do NOT accept parameters.
    
    It is suitable for simple, one-time queries.
    
    ---
    
    ## **2.2 Characteristics**
    
    - SQL is sent directly to database each time.
    - Not precompiled → slower.
    - Prone to SQL injection if values are concatenated.
    
    ---
    
    ## **2.3 Syntax**
    
    ```java
    Statement st = con.createStatement();
    
    ```
    
    ---
    
    ## **2.4 Methods**
    
    | --- | --- |
    
    ---
    
    ## **2.5 Example**
    
    ```java
    Statement st = con.createStatement();
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    ```
    
    ---
    
    ## **2.6 Drawbacks**
    
    - cannot accept parameters
    - risk of SQL injection
    - slower than PreparedStatement
    
    ---
    
    ---
    
    ## **3. PreparedStatement**
    
    ---
    
    ## **3.1 Definition and Explanation**
    
    `PreparedStatement` is a **precompiled SQL statement** that accepts **parameters**.
    
    It is safer, faster, and more flexible than `Statement`.
    
    PreparedStatement prevents SQL injection and improves performance due to pre-compilation and caching.
    
    ---
    
    ## **3.2 Characteristics**
    
    - Uses `?` placeholders for parameters
    - SQL is compiled only once
    - Efficient for repeated queries
    - Safe against SQL injection
    - Supports dynamic input
    
    ---
    
    ## **3.3 Syntax**
    
    ```java
    PreparedStatement ps = con.prepareStatement(
        "INSERT INTO employee(id,name,salary) VALUES(?,?,?)"
    );
    
    ```
    
    ---
    
    ## **3.4 Setting Parameters**
    
    ```java
    ps.setInt(1, 101);
    ps.setString(2, "John");
    ps.setDouble(3, 45000.0);
    
    ```
    
    ---
    
    ## **3.5 Executing**
    
    ```java
    ps.executeUpdate();
    
    ```
    
    ---
    
    ## **3.6 Example**
    
    ```java
    String sql = "SELECT * FROM employee WHERE id = ?";
    PreparedStatement ps = con.prepareStatement(sql);
    ps.setInt(1, 101);
    ResultSet rs = ps.executeQuery();
    
    ```
    
    ---
    
    ## **3.7 Advantages**
    
    - Prevents SQL injection
    - Faster for repeated execution
    - Easier to read and maintain
    - Allows dynamic parameters
    
    ---
    
    # **4. Differences Table**
    
    | --- | --- | --- |
    
    ---
    
    # **5. Interaction Between Connection, Statement, PreparedStatement**
    
    ```
    Connection → creates → Statement / PreparedStatement
    Statement / PreparedStatement → executes → SQL
    
    ```
    
    Flow:
    
    1. Get Connection →
    2. Create Statement or PreparedStatement →
    3. Execute SQL →
    4. Process ResultSet →
    5. Close resources
    
    ---
    
    # **6. Real-world Analogy**
    
    ### **Connection**
    
    A phone line that connects you to another person.
    
    ### **Statement**
    
    Speaking a fixed sentence every time — no changes allowed.
    
    ### **PreparedStatement**
    
    Speaking a prepared sentence where you can fill blanks dynamically.
    
    Example:
    
    “Order ID is ___ and customer is ___.”
    
    ---
    
    # **7. Important Points**
    
    - Always close `Connection`, `Statement`, and `PreparedStatement`.
    - Prefer `PreparedStatement` over `Statement` for most operations.
    - Use `Connection` to manage transactions and commit/rollback.
    - Use connection pooling for performance.
    
    ---
    
    # **8. Summary**
    
    `Connection` manages communication with the database and controls transactions.
    
    `Statement` executes simple SQL without parameters.
    
    `PreparedStatement` executes parameterized and precompiled SQL, providing better security and performance.
    
    Together, these JDBC components form the backbone of all database interactions in Java applications.
    
- ResultSet navigation & types
    
    # **ResultSet Navigation & Types**
    
    ---
    
    ## **1. ResultSet: Definition and Explanation**
    
    `ResultSet` is an object that represents the **tabular result** returned by executing a SQL query using JDBC.
    
    It acts like a **cursor** pointing to rows fetched from the database, allowing the application to read data row by row.
    
    `ResultSet` provides methods to:
    
    - Navigate through rows
    - Retrieve column values
    - Check cursor position
    - Handle different scroll behaviors
    
    ---
    
    # **2. ResultSet Navigation**
    
    ---
    
    ## **2.1 Cursor Movement**
    
    When a ResultSet is created, the cursor is positioned **before the first row**.
    
    Navigation methods include:
    
    | --- | --- |
    
    ### **Basic Example**
    
    ```java
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    while (rs.next()) {
        System.out.println(rs.getInt("id") + " " + rs.getString("name"));
    }
    
    ```
    
    ---
    
    ## **2.2 Retrieving Column Values**
    
    ```java
    int id = rs.getInt(1);
    String name = rs.getString("name");
    double salary = rs.getDouble("salary");
    
    ```
    
    Columns can be accessed by:
    
    - Column index (1-based)
    - Column name
    
    ---
    
    ## **2.3 Cursor Positions**
    
    | --- | --- |
    
    ---
    
    ## **2.4 Limitations of Default ResultSet**
    
    The default ResultSet is **forward-only** and **read-only**.
    
    Advanced navigation requires a different ResultSet type.
    
    ---
    
    # **3. ResultSet Types**
    
    ---
    
    ## **3.1 Overview**
    
    ResultSet types define:
    
    - Whether the cursor can move backward
    - Whether data is read-only or updatable
    - Whether ResultSet detects database changes
    
    To specify a type, create Statement using:
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ```
    
    ---
    
    # **4. Scrollability Types**
    
    ---
    
    ## **4.1 TYPE_FORWARD_ONLY (Default)**
    
    - Cursor moves forward only
    - Best performance
    - Most common
    
    Suitable for simple loops.
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_FORWARD_ONLY,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ```
    
    ---
    
    ## **4.2 TYPE_SCROLL_INSENSITIVE**
    
    - Allows moving forward, backward, first, last, absolute, etc.
    - **Does NOT reflect** changes made in the database after ResultSet is created
    - Takes a snapshot of data
    
    Example:
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ```
    
    ---
    
    ## **4.3 TYPE_SCROLL_SENSITIVE**
    
    - Allows full navigation
    - **Reflects real-time changes** made to the database
    - Slower, less commonly used
    
    Example:
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_SENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ```
    
    ---
    
    # **5. Concurrency Types**
    
    ---
    
    ## **5.1 CONCUR_READ_ONLY (Default)**
    
    - Data cannot be updated through ResultSet
    - Safe, fast
    
    ```java
    ResultSet.CONCUR_READ_ONLY
    
    ```
    
    ---
    
    ## **5.2 CONCUR_UPDATABLE**
    
    - Allows updating, inserting, and deleting rows through ResultSet
    
    Example:
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_UPDATABLE
    );
    
    ```
    
    ### **Update Example**
    
    ```java
    rs.updateDouble("salary", 60000);
    rs.updateRow();
    
    ```
    
    ---
    
    # **6. Holdability Types**
    
    Defines what happens to ResultSet when the commit is executed.
    
    | --- | --- |
    
    ---
    
    # **7. Practical Example: Scrollable ResultSet**
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    rs.last();
    System.out.println("Total rows: " + rs.getRow());
    
    rs.beforeFirst();
    while(rs.next()) {
        System.out.println(rs.getString("name"));
    }
    
    ```
    
    ---
    
    # **8. JVM / Internal Behavior**
    
    - ResultSet cursor exists on client side but controlled through JDBC driver.
    - In scrollable ResultSets, drivers may load full data into memory.
    - In forward-only, data is fetched row by row to reduce memory usage.
    - Sensitive ResultSet checks database for underlying row modifications.
    
    ---
    
    # **9. Real-world Analogy**
    
    ResultSet behaves like a **bookmark in a book**:
    
    - You can move forward/backward (scrollable)
    - Or only forward (forward-only)
    - Read-only means you cannot modify text
    - Updatable means you can write in the book
    
    ---
    
    # **10. Summary**
    
    ResultSet is a cursor-based table returned from SQL queries.
    
    Its navigation features include moving to next, previous, first, last, or specific rows.
    
    ResultSets come in various types based on scrollability, concurrency, and holdability.
    
    Choosing the correct type is essential for performance, flexibility, and data handling requirements.
    
- ResultSetMetaData & DatabaseMetaData
    
    # **ResultSetMetaData & DatabaseMetaData**
    
    ---
    
    ## **1. ResultSetMetaData**
    
    ---
    
    ## **1.1 Definition and Explanation**
    
    `ResultSetMetaData` provides information **about the structure of a ResultSet**, such as:
    
    - Number of columns
    - Column names
    - Column data types
    - Column size
    - Whether a column is nullable or auto-increment
    
    It does **not** provide data stored in rows — only metadata about the columns.
    
    This helps in building **dynamic applications**, where the column structure is not known at compile time.
    
    ---
    
    ## **1.2 Use Cases**
    
    - Displaying table results dynamically (e.g., generic table viewer)
    - Writing generic export utilities (CSV, Excel, JSON)
    - Debugging unknown or dynamic queries
    - ORM and framework internals use it to map columns to fields
    
    ---
    
    ## **1.3 Syntax**
    
    ```java
    ResultSetMetaData rsmd = rs.getMetaData();
    
    ```
    
    ---
    
    ## **1.4 Common Methods**
    
    | --- | --- |
    
    ---
    
    ## **1.5 Example**
    
    ```java
    ResultSet rs = st.executeQuery("SELECT id, name, salary FROM employee");
    ResultSetMetaData rsmd = rs.getMetaData();
    
    int columnCount = rsmd.getColumnCount();
    
    for (int i = 1; i <= columnCount; i++) {
        System.out.println(
            rsmd.getColumnName(i) + " : " + rsmd.getColumnTypeName(i)
        );
    }
    
    ```
    
    ### **Output Example**
    
    ```
    id : INT
    name : VARCHAR
    salary : DOUBLE
    
    ```
    
    ---
    
    ## **1.6 Notes**
    
    - Column index starts at **1** (not 0).
    - Works only for ResultSet returned from SELECT queries.
    - Used by frameworks to auto-map database fields to Java objects.
    
    ---
    
    ---
    
    # **2. DatabaseMetaData**
    
    ---
    
    ## **2.1 Definition and Explanation**
    
    `DatabaseMetaData` provides metadata about the **database itself**, not the result of a query.
    
    It contains information like:
    
    - Database name and version
    - JDBC driver name and version
    - List of tables
    - List of columns
    - Primary keys and foreign keys
    - Supported SQL features
    
    It is extremely useful for:
    
    - Tooling
    - Schema inspection
    - ORM frameworks
    - Code generators
    - Database exploration utilities
    
    ---
    
    ## **2.2 Syntax**
    
    ```java
    DatabaseMetaData dbmd = con.getMetaData();
    
    ```
    
    ---
    
    ## **2.3 Common Methods**
    
    | --- | --- |
    
    ---
    
    ## **2.4 Example: Getting Database Info**
    
    ```java
    DatabaseMetaData dbmd = con.getMetaData();
    
    System.out.println("DB Name: " + dbmd.getDatabaseProductName());
    System.out.println("DB Version: " + dbmd.getDatabaseProductVersion());
    System.out.println("Driver: " + dbmd.getDriverName());
    
    ```
    
    ---
    
    ## **2.5 Example: Getting Tables**
    
    ```java
    ResultSet tables = dbmd.getTables(null, null, "%", new String[]{"TABLE"});
    
    while (tables.next()) {
        System.out.println(tables.getString("TABLE_NAME"));
    }
    
    ```
    
    ---
    
    ## **2.6 Example: Getting Columns of a Table**
    
    ```java
    ResultSet cols = dbmd.getColumns(null, null, "employee", null);
    
    while (cols.next()) {
        System.out.println(
            cols.getString("COLUMN_NAME") + " - " + cols.getString("TYPE_NAME")
        );
    }
    
    ```

### **13.2 CRUD Operations with JDBC**

- Insert, update, delete queries
    
    # **CRUD Operations with JDBC (Insert, Update, Delete)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    CRUD stands for **Create, Read, Update, Delete**, which represent the fundamental operations used in database-driven applications.
    
    In JDBC, these operations are performed using:
    
    - `Connection`
    - `PreparedStatement`
    - `Statement`
    - SQL commands (INSERT, UPDATE, DELETE, SELECT)
    
    This section focuses on **INSERT (Create)**, **UPDATE**, and **DELETE** operations using JDBC in a clean, structured manner.
    
    ---
    
    # **2. INSERT Operation (Create)**
    
    ---
    
    ## **2.1 Purpose**
    
    The INSERT operation adds a new record into a database table.
    
    PreparedStatement is preferred because it:
    
    - Prevents SQL injection
    - Supports dynamic parameters
    - Improves performance
    
    ---
    
    ## **2.2 Syntax**
    
    ```java
    String sql = "INSERT INTO employee(id, name, salary) VALUES (?, ?, ?)";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ```
    
    ---
    
    ## **2.3 Example**
    
    ```java
    Connection con = DriverManager.getConnection(url, user, pass);
    
    String sql = "INSERT INTO employee(id,name,salary) VALUES(?,?,?)";
    
    PreparedStatement ps = con.prepareStatement(sql);
    ps.setInt(1, 101);
    ps.setString(2, "John");
    ps.setDouble(3, 45000.0);
    
    int rows = ps.executeUpdate();
    System.out.println(rows + " row inserted.");
    
    ```
    
    ### **Output**
    
    ```
    1 row inserted.
    
    ```
    
    ---
    
    ## **2.4 Notes**
    
    - `executeUpdate()` returns number of affected rows.
    - Always parameterize values using `?`.
    - Close PreparedStatement and Connection after use.
    
    ---
    
    # **3. UPDATE Operation**
    
    ---
    
    ## **3.1 Purpose**
    
    The UPDATE operation modifies existing records in a table.
    
    ---
    
    ## **3.2 Syntax**
    
    ```java
    String sql = "UPDATE employee SET salary = ? WHERE id = ?";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ```
    
    ---
    
    ## **3.3 Example**
    
    ```java
    String sql = "UPDATE employee SET salary = ? WHERE id = ?";
    
    PreparedStatement ps = con.prepareStatement(sql);
    ps.setDouble(1, 55000.0);
    ps.setInt(2, 101);
    
    int rows = ps.executeUpdate();
    System.out.println(rows + " row updated.");
    
    ```
    
    ### **Output**
    
    ```
    1 row updated.
    
    ```
    
    ---
    
    ## **3.4 Notes**
    
    - Ensure WHERE clause is used; otherwise, all rows may update.
    - SQL injection-safe because values use placeholders.
    
    ---
    
    # **4. DELETE Operation**
    
    ---
    
    ## **4.1 Purpose**
    
    The DELETE operation removes existing records from a table.
    
    ---
    
    ## **4.2 Syntax**
    
    ```java
    String sql = "DELETE FROM employee WHERE id = ?";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ```
    
    ---
    
    ## **4.3 Example**
    
    ```java
    String sql = "DELETE FROM employee WHERE id = ?";
    
    PreparedStatement ps = con.prepareStatement(sql);
    ps.setInt(1, 101);
    
    int rows = ps.executeUpdate();
    System.out.println(rows + " row deleted.");
    
    ```
    
    ### **Output**
    
    ```
    1 row deleted.
    
    ```
    
    ---
    
    ## **4.4 Notes**
    
    - Use WHERE clause to avoid deleting full table.
    - `executeUpdate()` handles DELETE as well.
    
    ---
    
    # **5. Full CRUD Template (Insert, Update, Delete)**
    
    ```java
    Connection con = DriverManager.getConnection(url, user, pass);
    
    // INSERT
    PreparedStatement insert = con.prepareStatement(
        "INSERT INTO employee(id,name,salary) VALUES(?,?,?)"
    );
    insert.setInt(1, 101);
    insert.setString(2, "John");
    insert.setDouble(3, 45000);
    insert.executeUpdate();
    
    // UPDATE
    PreparedStatement update = con.prepareStatement(
        "UPDATE employee SET name=? WHERE id=?"
    );
    update.setString(1, "Jonathan");
    update.setInt(2, 101);
    update.executeUpdate();
    
    // DELETE
    PreparedStatement delete = con.prepareStatement(
        "DELETE FROM employee WHERE id=?"
    );
    delete.setInt(1, 101);
    delete.executeUpdate();
    
    con.close();
    
    ```
    
    ---
    
    # **6. Common JDBC Best Practices for CRUD**
    
    - Always use **PreparedStatement** instead of Statement.
    - Close ResultSet, Statement, and Connection inside **finally** block or use **try-with-resources**.
    - Enable transactions (`setAutoCommit(false)`) for multi-step operations.
    - Validate data before performing operations.
    - Log all SQL errors for debugging.
    
    ---
    
    # **7. Real-world Analogy**
    
    - **INSERT** → Adding a new entry to a logbook.
    - **UPDATE** → Editing an existing record.
    - **DELETE** → Removing an entry from the logbook.
    
    ---
    
    # **8. Summary**
    
    CRUD operations in JDBC involve using PreparedStatements to safely execute SQL commands.
    
    INSERT adds new rows, UPDATE modifies existing rows, and DELETE removes rows from tables.
    
    Following proper practices ensures secure, efficient, and maintainable database interactions in enterprise applications.
    
- Read operations with ResultSet
    
    # **Read Operations with ResultSet (SELECT Queries in JDBC)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Read operations retrieve data from a database using **SELECT** queries.
    
    In JDBC, read operations return a `ResultSet`, which acts like a cursor to iterate through the returned rows.
    
    `ResultSet` allows navigation row-by-row and provides methods to extract each column’s value using either index or column name.
    
    Reading data is one of the most widely used operations in JDBC and forms the foundation of reporting, validation checks, UI population, and business logic decisions.
    
    ---
    
    # **2. Basic Steps for Reading Data**
    
    1. Establish a `Connection`
    2. Create a `PreparedStatement` or `Statement`
    3. Execute SELECT query (`executeQuery()`)
    4. Receive a `ResultSet`
    5. Navigate through rows (`next()`)
    6. Extract column values (`getXxx()`)
    7. Close resources
    
    ---
    
    # **3. Basic SELECT Operation Example**
    
    ```java
    String sql = "SELECT id, name, salary FROM employee";
    PreparedStatement ps = con.prepareStatement(sql);
    ResultSet rs = ps.executeQuery();
    
    while (rs.next()) {
        int id = rs.getInt("id");
        String name = rs.getString("name");
        double salary = rs.getDouble("salary");
    
        System.out.println(id + " | " + name + " | " + salary);
    }
    
    ```
    
    ### **Output**
    
    ```
    101 | John | 45000.0
    102 | Sara | 52000.0
    
    ```
    
    ---
    
    # **4. Using Column Index vs Column Name**
    
    ### **4.1 Column Index (1-based)**
    
    ```java
    int id = rs.getInt(1);
    String name = rs.getString(2);
    
    ```
    
    ### **4.2 Column Name**
    
    ```java
    int id = rs.getInt("id");
    String name = rs.getString("name");
    
    ```
    
    ### **Notes**
    
    - Column name is more readable and less error-prone.
    - Column index is slightly faster but depends on exact column order.
    
    ---
    
    # **5. Reading Different Data Types**
    
    | --- | --- |
    
    Example:
    
    ```java
    int id = rs.getInt("id");
    String email = rs.getString("email");
    Date dob = rs.getDate("dob");
    boolean status = rs.getBoolean("active");
    
    ```
    
    ---
    
    # **6. Processing All Rows**
    
    The most common pattern:
    
    ```java
    while (rs.next()) {
        // read columns
    }
    
    ```
    
    `next()` returns **true** while there are more rows.
    
    When it returns **false**, the ResultSet has no more rows.
    
    ---
    
    # **7. Reading a Single Row**
    
    If expecting only one result:
    
    ```java
    ResultSet rs = ps.executeQuery();
    
    if (rs.next()) {
        // read row
    } else {
        System.out.println("Record not found");
    }
    
    ```
    
    ---
    
    # **8. Mapping ResultSet to an Entity Class**
    
    Typical service/DAO pattern:
    
    ```java
    Employee e = null;
    
    if (rs.next()) {
        e = new Employee(
            rs.getInt("id"),
            rs.getString("name"),
            rs.getDouble("salary")
        );
    }
    return e;
    
    ```
    
    This is how ORMs internally map table rows to objects.
    
    ---
    
    # **9. Handling No Results**
    
    ```java
    if (!rs.next()) {
        throw new EmployeeNotFoundException("No data found");
    }
    
    ```
    
    or
    
    ```java
    if (!rs.next()) return null;
    
    ```
    
    ---
    
    # **10. Advanced Read Using Scrollable ResultSet**
    
    Scrollable ResultSets allow moving in any direction:
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    rs.last();
    int count = rs.getRow();
    
    rs.beforeFirst();
    
    ```
    
    ---
    
    # **11. Reading Metadata Along with Data**
    
    Using `ResultSetMetaData`:
    
    ```java
    ResultSetMetaData md = rs.getMetaData();
    int count = md.getColumnCount();
    
    while (rs.next()) {
        for (int i = 1; i <= count; i++) {
            System.out.print(rs.getString(i) + " ");
        }
        System.out.println();
    }
    
    ```
    
    ---
    
    # **12. Real-world Analogy**
    
    Reading from a ResultSet is like:
    
    - Scanning a spreadsheet row by row.
    - Moving the cursor to the next row.
    - Picking values from each column by header or by column number.
    
    ---
    
    # **13. Best Practices**
    
    - Always close ResultSet, Statement, and Connection (use try-with-resources).
    - Prefer `PreparedStatement` over `Statement`.
    - Use column names for readability.
    - If retrieving large data, consider paging (`LIMIT`, `OFFSET`).
    - Avoid unnecessary scrolling unless required.
    - Avoid loading too many rows in memory at once.
    
    ---
    
    # **14. Summary**
    
    Read operations in JDBC rely on executing SELECT queries using a PreparedStatement and iterating through the returned ResultSet.
    
    By using `next()`, column getters, and mapping results to Java objects, applications can retrieve and process database data efficiently.
    
    ResultSet is central to JDBC read operations and provides flexible navigation and data retrieval capabilities.
    
- Preventing SQL injection (PreparedStatement)
    
    # **Preventing SQL Injection (Using PreparedStatement)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    SQL Injection is a security attack where malicious input is inserted into SQL queries to manipulate the database.
    
    Example: manipulating a login query to always return true, or deleting entire tables.
    
    In JDBC, **PreparedStatement** is the primary and recommended way to prevent SQL injection.
    
    PreparedStatements separate SQL structure from user input, ensuring input is treated as data, **not executable SQL**.
    
    ---
    
    # **2. What is SQL Injection?**
    
    SQL Injection happens when unsafe user inputs are concatenated into SQL queries.
    
    ### **Example: Vulnerable Code (Using Statement)**
    
    ```java
    String sql = "SELECT * FROM users WHERE username='" + user + "' AND password='" + pass + "'";
    Statement st = con.createStatement();
    ResultSet rs = st.executeQuery(sql);
    
    ```
    
    ### **Malicious Input**
    
    ```
    username: admin
    password: ' OR '1'='1
    
    ```
    
    Generated Query:
    
    ```sql
    SELECT * FROM users WHERE username='admin' AND password='' OR '1'='1'
    
    ```
    
    This will return all users and bypass login.
    
    ---
    
    # **3. Why PreparedStatement Prevents SQL Injection**
    
    PreparedStatement uses:
    
    - **Parameter placeholders (`?`)** instead of string concatenation.
    - **Precompiled SQL** where structure is fixed.
    - **Type-safe binding of parameters** using setter methods like `setString()`, `setInt()`, etc.
    
    This ensures that:
    
    - User inputs are treated only as **values**, not SQL code.
    - The database engine cannot interpret them as part of query logic.
    
    ---
    
    # **4. Correct and Safe Usage of PreparedStatement**
    
    ---
    
    ## **4.1 Example: Safe Login Query**
    
    ```java
    String sql = "SELECT * FROM users WHERE username = ? AND password = ?";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ps.setString(1, user);
    ps.setString(2, pass);
    
    ResultSet rs = ps.executeQuery();
    
    ```
    
    The database receives:
    
    - Fixed SQL structure
    - User inputs as **typed parameters**, not executable SQL
    
    ---
    
    ## **4.2 Example: Safe Insert**
    
    ```java
    String sql = "INSERT INTO employee(id, name, salary) VALUES (?, ?, ?)";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ps.setInt(1, id);
    ps.setString(2, name);
    ps.setDouble(3, salary);
    
    ps.executeUpdate();
    
    ```
    
    ---
    
    # **5. How PreparedStatement Works Internally**
    
    ---
    
    ### **A. SQL is precompiled by the database**
    
    The database prepares the SQL query before parameters are supplied.
    
    ### **B. Parameters are sent separately**
    
    Inputs are bound to the precompiled query as literal values.
    
    ### **C. Database engine treats input as data**
    
    Even if the input contains SQL keywords or symbols, it cannot break or alter the query.
    
    ---
    
    # **6. Example: Attempted SQL Injection with PreparedStatement**
    
    ```java
    String userInput = "' OR '1'='1";
    
    String sql = "SELECT * FROM users WHERE username = ?";
    PreparedStatement ps = con.prepareStatement(sql);
    
    ps.setString(1, userInput);
    
    ResultSet rs = ps.executeQuery();
    
    ```
    
    Actual query executed:
    
    ```
    username = '' OR '1'='1'
    
    ```
    
    This is treated as a single string literal, **NOT executable SQL**, so no injection is possible.
    
    ---
    
    # **7. Additional Protective Measures**
    
    PreparedStatement is the primary defense, but combine it with:
    
    ### **A. Input Validation**
    
    - Reject dangerous characters when not required
    - Validate length and format (e.g., email, phone)
    
    ### **B. Use `LIMIT` for read operations**
    
    Reduces impact even if something goes wrong.
    
    ### **C. Least Privilege DB Accounts**
    
    Never use an admin/root user in production.
    
    ### **D. Stored Procedures (optional)**
    
    Although not a full replacement, can help further restrict logic.
    
    ### **E. Escaping special characters**
    
    Useful in some rare cases (but PreparedStatement already handles this).
    
    ---
    
    # **8. Real-world Analogy**
    
    Using a PreparedStatement is like filling blanks in a **pre-printed government form**:
    
    - You cannot add extra fields or change structure.
    - You can only fill values in allowed spaces.
    - The structure (query) is fixed.
    
    Without PreparedStatement (concatenation) is like:
    
    - Writing your own form from scratch.
    - Someone could change rules or write unsafe statements.
    
    ---
    
    # **9. Important Points**
    
    - Always use PreparedStatement for any dynamic input.
    - Never concatenate strings directly into SQL queries.
    - All user inputs must be treated as unsafe until validated.
    - PreparedStatement removes *99%* of SQL injection risk.
    - Use parameters even for numeric fields.
    
    ---
    
    # **10. Summary**
    
    PreparedStatement is the safest and most reliable mechanism to prevent SQL injection in JDBC.
    
    It separates SQL logic from user input, precompiles SQL, and treats inputs strictly as values.
    
    Using PreparedStatement ensures that malicious inputs cannot alter or break SQL queries, keeping the application secure and stable.
    

### **13.3 Advanced JDBC**

- Batch operations
    
    # **Batch Operations in JDBC**
    
    ---
    
    ## **1. Definition and Explanation**
    
    Batch operations in JDBC allow multiple SQL statements to be sent to the database **in a single request** instead of executing them one by one.
    
    This significantly improves performance when inserting, updating, or deleting large volumes of data.
    
    Batching reduces:
    
    - Number of database round trips
    - Network overhead
    - Execution time
    
    Batch operations are typically used for bulk inserts, bulk updates, logs processing, and data migration tasks.
    
    ---
    
    # **2. Why Use Batch Operations?**
    
    ### **A. Performance Improvement**
    
    Sending 10,000 inserts one by one is slow.
    
    Batching sends them in groups → faster execution.
    
    ### **B. Reduced Network I/O**
    
    All statements are sent together.
    
    ### **C. Database Efficiency**
    
    Database processes the batch more efficiently.
    
    ### **D. Cleaner Code**
    
    Avoids repetitive execute calls.
    
    ---
    
    # **3. Batch with Statement**
    
    Useful when SQL is static, no parameters needed.
    
    ```java
    Statement st = con.createStatement();
    
    st.addBatch("INSERT INTO employee VALUES (101, 'John', 45000)");
    st.addBatch("INSERT INTO employee VALUES (102, 'Sara', 52000)");
    st.addBatch("UPDATE employee SET salary = 60000 WHERE id = 103");
    
    int[] result = st.executeBatch();
    
    ```
    
    ---
    
    # **4. Batch with PreparedStatement (Preferred)**
    
    PreparedStatement is more secure and faster when using parameters.
    
    ---
    
    ## **4.1 Example: Batch INSERT**
    
    ```java
    String sql = "INSERT INTO employee(id, name, salary) VALUES (?, ?, ?)";
    PreparedStatement ps = con.prepareStatement(sql);
    
    for (int i = 1; i <= 5; i++) {
        ps.setInt(1, 100 + i);
        ps.setString(2, "Emp" + i);
        ps.setDouble(3, 30000 + (i * 1000));
    
        ps.addBatch();
    }
    
    int[] result = ps.executeBatch();
    System.out.println("Rows affected: " + result.length);
    
    ```
    
    ---
    
    # **5. Batch with Explicit Transactions**
    
    Batching is even faster when auto-commit is disabled.
    
    ```java
    con.setAutoCommit(false);
    
    PreparedStatement ps = con.prepareStatement(
        "INSERT INTO product(id, name, price) VALUES (?, ?, ?)"
    );
    
    for (Product p : productList) {
        ps.setInt(1, p.getId());
        ps.setString(2, p.getName());
        ps.setDouble(3, p.getPrice());
        ps.addBatch();
    }
    
    ps.executeBatch();
    con.commit();
    
    ```
    
    ### **Benefits**
    
    - Reduces overhead of committing after each statement.
    - Ensures all operations succeed together.
    
    ---
    
    # **6. Partial Failure Handling with Batch**
    
    If one statement fails:
    
    - Some drivers stop execution and throw `BatchUpdateException`
    - `getUpdateCounts()` returns number of successful operations
    
    ### **Example**
    
    ```java
    catch (BatchUpdateException e) {
        int[] counts = e.getUpdateCounts();
        System.out.println("Successful updates: " + counts.length);
    }
    
    ```
    
    ---
    
    # **7. Batch Size and Memory Optimization**
    
    Sending too many statements in one batch may cause:
    
    - Memory overload
    - Driver buffer limits
    - Slowdowns
    
    ### **Recommended Practice**
    
    Use chunked batching:
    
    ```java
    int batchSize = 1000;
    int count = 0;
    
    for (Data d : list) {
        ps.setInt(1, d.getId());
        ps.setString(2, d.getName());
        ps.addBatch();
        count++;
    
        if (count % batchSize == 0) {
            ps.executeBatch();
            ps.clearBatch();
        }
    }
    
    ```
    
    ---
    
    # **8. Driver and Database Support**
    
    - Most modern databases support batching.
    - Some drivers require enabling batching via connection properties.
    
    MySQL example:
    
    ```
    ?rewriteBatchedStatements=true
    
    ```
    
    ---
    
    # **9. Real-world Analogy**
    
    Batch operations are like:
    
    - Sending **one big parcel** containing 100 documents
    - Sending **100 separate envelopes**
    
    Fewer couriers, less cost, faster delivery.
    
    ---
    
    # **10. Best Practices**
    
    - Always use PreparedStatement for batch operations.
    - Disable auto-commit and commit manually.
    - Use chunked batches for large datasets.
    - Clear the batch after executing.
    - Handle `BatchUpdateException`.
    - Ensure driver supports batching.
    
    ---
    
    # **11. Summary**
    
    Batch operations in JDBC allow multiple SQL commands to be executed together, improving performance and reducing overhead.
    
    PreparedStatement-based batching is the fastest and safest way to handle bulk inserts or updates.
    
    With proper transaction control and error handling, batch operations significantly enhance database performance in enterprise applications.
    
- Scrollable & updatable ResultSet
    
    # **Scrollable & Updatable ResultSet**
    
    ---
    
    ## **1. Definition and Explanation**
    
    A `ResultSet` normally supports **forward-only navigation** and is **read-only** by default.
    
    Scrollable and updatable ResultSets extend this default capability:
    
    - **Scrollable ResultSet**: allows movement in multiple directions (previous, first, last, absolute positioning).
    - **Updatable ResultSet**: allows modification of database rows through ResultSet itself (update, insert, delete).
    
    These enhanced ResultSets are configured when creating the Statement or PreparedStatement.
    
    ---
    
    ---
    
    # **2. Scrollable ResultSet**
    
    ---
    
    ## **2.1 Definition**
    
    A scrollable ResultSet allows navigation:
    
    - forward
    - backward
    - first row
    - last row
    - to a specific row (absolute)
    - relative movement (+/- rows)
    
    Default ResultSet does not provide this capability.
    
    ---
    
    ## **2.2 Types of Scrollable ResultSet**
    
    | --- | --- |
    
    ---
    
    ## **2.3 Creating Scrollable ResultSet**
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_READ_ONLY
    );
    ResultSet rs = st.executeQuery("SELECT * FROM employee");
    
    ```
    
    ---
    
    ## **2.4 Navigation Methods**
    
    | --- | --- |
    
    ---
    
    ## **2.5 Example**
    
    ```java
    rs.last();
    System.out.println("Total rows: " + rs.getRow());
    
    rs.beforeFirst();
    while(rs.next()) {
        System.out.println(rs.getString("name"));
    }
    
    ```
    
    ---
    
    ---
    
    # **3. Updatable ResultSet**
    
    ---
    
    ## **3.1 Definition**
    
    An updatable ResultSet allows modification of data directly from the ResultSet object.
    
    You can:
    
    - Update existing rows
    - Insert new rows
    - Delete rows
    
    Changes apply back to the database automatically.
    
    ---
    
    ## **3.2 Requirements**
    
    - Statement must be created with `CONCUR_UPDATABLE`
    - Query must reference key columns uniquely
    - Database and driver must support it
    
    ---
    
    ## **3.3 Creating Updatable ResultSet**
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_UPDATABLE
    );
    ResultSet rs = st.executeQuery("SELECT id, name, salary FROM employee");
    
    ```
    
    ---
    
    ## **3.4 Updating a Row**
    
    ```java
    rs.absolute(1);
    rs.updateDouble("salary", 60000.00);
    rs.updateRow();   // commit change to DB
    
    ```
    
    ---
    
    ## **3.5 Deleting a Row**
    
    ```java
    rs.absolute(2);
    rs.deleteRow();
    
    ```
    
    ---
    
    ## **3.6 Inserting a New Row**
    
    ```java
    rs.moveToInsertRow();
    rs.updateInt("id", 105);
    rs.updateString("name", "NewEmp");
    rs.updateDouble("salary", 40000);
    rs.insertRow();
    rs.moveToCurrentRow();
    
    ```
    
    ---
    
    ---
    
    # **4. Scrollable + Updatable ResultSet Together**
    
    ---
    
    ## **4.1 Combined Configuration**
    
    ```java
    Statement st = con.createStatement(
        ResultSet.TYPE_SCROLL_INSENSITIVE,
        ResultSet.CONCUR_UPDATABLE
    );
    
    ```

- CallableStatement (stored procedures)
    
    # **CallableStatement (Stored Procedures)**
    
    ---
    
    ## **1. Definition and Explanation**
    
    `CallableStatement` is a JDBC interface used to execute **stored procedures** in a relational database.
    
    Stored procedures contain precompiled SQL logic stored on the database server, often used for:
    
    - Business logic
    - Transactional operations
    - Performance optimization
    - Reusable routines
    - Security abstraction
    
    `CallableStatement` is created from a Connection object and allows passing **input parameters**, reading **output parameters**, and handling **result sets** returned by stored procedures.
    
    ---
    
    ---
    
    # **2. Why Stored Procedures?**
    
    ### **A. Performance**
    
    Precompiled SQL executes faster than sending raw SQL repeatedly.
    
    ### **B. Security**
    
    Database logic stays inside DB, hiding actual query details from the application.
    
    ### **C. Reusability**
    
    Multiple applications can use the same stored procedure.
    
    ### **D. Transaction Handling**
    
    Complex logic can be grouped and executed atomically.
    
    ### **E. Reduced Network Calls**
    
    Multiple SQL operations run within a procedure execution.
    
    ---
    
    ---
    
    # **3. CallableStatement Basics**
    
    ---
    
    ## **3.1 Syntax (Creating CallableStatement)**
    
    ```java
    CallableStatement cs = con.prepareCall("{ call procedure_name(?, ?) }");
    
    ```
    
    Braces `{}` indicate JDBC escape syntax for stored procedure calls.
    
    ---
    
    ## **3.2 Input Parameters**
    
    ```java
    cs.setInt(1, 101);
    cs.setString(2, "John");
    
    ```
    
    ---
    
    ## **3.3 Output Parameters**
    
    ```java
    cs.registerOutParameter(3, java.sql.Types.DOUBLE);
    
    ```
    
    Then read:
    
    ```java
    double sal = cs.getDouble(3);
    
    ```
    
    ---
    
    ---
    
    # **4. Stored Procedure Types & CallableStatement Usage**
    
    ---
    
    ## **4.1 Stored Procedure with No Parameters**
    
    ### **Database Procedure Example**
    
    ```sql
    CREATE PROCEDURE listEmployees()
    BEGIN
        SELECT * FROM employee;
    END;
    
    ```
    
    ### **JDBC Call**
    
    ```java
    CallableStatement cs = con.prepareCall("{ call listEmployees() }");
    ResultSet rs = cs.executeQuery();
    
    ```
    
    ---
    
    ---
    
    ## **4.2 Stored Procedure with Input Parameters**
    
    ### **Database Procedure Example**
    
    ```sql
    CREATE PROCEDURE getEmployee(IN empId INT)
    BEGIN
        SELECT * FROM employee WHERE id = empId;
    END;
    
    ```
    
    ### **JDBC Call**
    
    ```java
    CallableStatement cs = con.prepareCall("{ call getEmployee(?) }");
    cs.setInt(1, 101);
    ResultSet rs = cs.executeQuery();
    
    ```
    
    ---
    
    ---
    
    ## **4.3 Stored Procedure with Output Parameters**
    
    ### **Database Procedure Example**
    
    ```sql
    CREATE PROCEDURE getSalary(IN empId INT, OUT empSalary DOUBLE)
    BEGIN
        SELECT salary INTO empSalary FROM employee WHERE id = empId;
    END;
    
    ```
    
    ### **JDBC Call**
    
    ```java
    CallableStatement cs = con.prepareCall("{ call getSalary(?, ?) }");
    
    cs.setInt(1, 101);
    cs.registerOutParameter(2, java.sql.Types.DOUBLE);
    
    cs.execute();
    
    double salary = cs.getDouble(2);
    System.out.println("Salary = " + salary);
    
    ```
    
    ---
    
    ---
    
    ## **4.4 Stored Procedure Returning Multiple ResultSets**
    
    Some procedures return multiple SELECTs.
    
    `CallableStatement.execute()` handles this:
    
    ```java
    boolean hasResults = cs.execute();
    while (hasResults) {
        ResultSet rs = cs.getResultSet();
        // process rs
        hasResults = cs.getMoreResults();
    }
    
    ```
    
    ---
    
    ---
    
    # **5. JDBC Escape Syntax for Stored Procedures**
    
    | --- | --- |
    
    ---
    
    ### **Example Calling Function with Return Value**
    
    ```java
    CallableStatement cs = con.prepareCall("{ ? = call calcBonus(?) }");
    cs.registerOutParameter(1, Types.DOUBLE);
    cs.setInt(2, 101);
    
    cs.execute();
    System.out.println("Bonus = " + cs.getDouble(1));
    
    ```
    
    ---
    
    ---
    
    # **6. JVM / Driver Internal Behavior**
    
    - JDBC driver parses escape syntax.
    - Translates `{ call proc(?) }` to database-native syntax.
    - Sends invocation to DB server.
    - Extracts output values after execution.
    - Converts DB types to Java types.
    
    ---
    
    ---
    
    # **7. Real-world Analogy**
    
    Using `CallableStatement` is like:
    
    - Calling a **prewritten function** stored on a server.
    - You send input values, server executes the function, and you get output.
    - Instead of writing logic fresh each time, you call an already optimized routine.
    
    ---
    
    ---
    
    # **8. Best Practices**
    
    - Use CallableStatement for business logic inside database.
    - Catch `SQLException`.
    - Always close ResultSet and CallableStatement.
    - Prefer using **PreparedStatement over CallableStatement** for simple queries.
    - Use OUT parameters only when necessary.
    
    ---
    
    ---
    
    # **9. Summary**
    
    `CallableStatement` enables Java applications to run stored procedures efficiently.
    
    It supports passing input parameters, extracting output parameters, and handling multiple results.
    
    Stored procedures enhance performance, encapsulate database logic, and improve security — making CallableStatement an essential tool in enterprise JDBC applications.
